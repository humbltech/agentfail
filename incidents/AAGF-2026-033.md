# mcp-remote OS Command Injection via OAuth Metadata RCE (CVE-2025-6514)

## Executive Summary

A community-maintained npm package called `mcp-remote` — the standard bridge enabling Claude Desktop, Cursor, and Windsurf to connect to remote MCP servers with OAuth authentication — contained an OS command injection vulnerability in its OAuth handshake. When a developer connected their MCP client to a malicious remote server, the server could return a crafted `authorization_endpoint` URL that mcp-remote passed unsanitized to an OS shell invocation, achieving full arbitrary code execution on the developer's machine on Windows and executable launch on macOS/Linux. The vulnerability affected versions 0.0.5 through 0.1.15, covering approximately 437,000 download events (unique running installations were an unknown fraction of this figure — npm download counts include CI rebuilds, automated pulls, and duplicate installs). A fix was quietly released on June 17, 2025; the CVE was not publicly disclosed until July 9, 2025 — a 22-day window during which vulnerable installations had no advisory to prompt an upgrade.

## What Happened

### What mcp-remote Is — and Why It Exists

The Model Context Protocol was designed with two server transport modes: local stdio (a server process running on the same machine as the client) and remote HTTP/SSE (a server running elsewhere, accessed over the network). A structural gap emerged at MCP launch: major MCP clients — Claude Desktop, Cursor, and Windsurf — only supported local stdio servers natively. They had no built-in mechanism to connect to remote MCP servers with OAuth authentication.

`mcp-remote` was built to bridge this gap. Maintained by Glen Maddern (Head of MCP at Groq, not an Anthropic employee), it acts as a local stdio proxy: the MCP client starts mcp-remote as a local process, mcp-remote handles the OAuth authentication flow with the remote server, and then proxies the MCP protocol over the network connection. From the MCP client's perspective, it sees a local stdio server. From the remote server's perspective, it sees an authenticated HTTP client.

This made mcp-remote the de facto standard for connecting Claude Desktop, Cursor, and Windsurf to any remote MCP server requiring OAuth. Anthropic referenced it in MCP documentation and tutorials. It accumulated 437,000+ downloads. The package README labeled it "experimental" — a label that became meaningless as the package became load-bearing infrastructure for a significant fraction of the MCP ecosystem.

### The OAuth Handshake Attack

OAuth 2.0 authentication begins with a discovery phase: the client fetches metadata from the server to learn where to direct the user for authorization. This metadata — typically served at a well-known endpoint — includes an `authorization_endpoint` URL pointing to the OAuth authorization server. The client then opens this URL in the user's browser to begin the authorization flow.

In mcp-remote, the `discoverOAuthMetadata()` function fetches this metadata from the remote MCP server. The `authorization_endpoint` URL from that metadata is passed through the OAuth flow to `redirectToAuthorization()`, which calls the `open()` npm package to launch the URL in the user's browser.

The vulnerability: the `open()` package opens URLs by invoking an OS shell command — on Windows it calls `cmd.exe`; on macOS/Linux it calls `open` or `xdg-open`. It is not a pure browser API call. It is an OS shell invocation with a URL as an argument. Remote-server-supplied URL content that reaches this call without sanitization becomes OS shell input.

An attacker controlling a remote MCP server returns a crafted `authorization_endpoint` URL in their OAuth metadata response. mcp-remote fetches the metadata, extracts the URL, and passes it unsanitized to `open()`. The OS shell executes the embedded commands. The attack is triggered automatically during the OAuth handshake — before the user sees or interacts with any authorization page.

### The Vulnerable Code Path

The injection travels through the following call chain:

```
StreamableHTTPClientTransport.send()
  → auth.ts:auth()
    → discoverOAuthMetadata()        ← fetches attacker-controlled metadata
      → startAuthorization()
        → node-oauth-client-provider.ts:redirectToAuthorization()
          → open(authorization_endpoint_url)  ← OS shell invocation
```

The malicious `authorization_endpoint` is extracted from the server-provided metadata at `discoverOAuthMetadata()` and reaches `open()` without sanitization.

### Windows: Full Arbitrary RCE

On Windows, the `open()` package invokes `cmd.exe` or PowerShell. The attacker's crafted URL uses:

- A non-standard URI scheme (`a:`) that prevents URL encoding of special characters
- PowerShell's `$()` subexpression operator to embed arbitrary commands

Example payload:

```
a:$(Start-Process cmd.exe -ArgumentList '/c calc.exe')
```

When passed to the Windows shell via `open()`, PowerShell evaluates the `$()` subexpression and executes the embedded command. The `a:` scheme prevents the URL parser from encoding the `$`, `(`, `)`, and other special characters that would defuse the injection. The result: full arbitrary command execution under the user's account — the same privileges held by the running MCP client (Claude Desktop, Cursor, Windsurf).

### macOS/Linux: Constrained but Targeted Executable Launch

On macOS and Linux, the OS `open` and `xdg-open` commands do not pass arguments to shell in the same way as Windows. Full arbitrary command injection is not achievable through the same mechanism. macOS `open` and Linux `xdg-open` interpret `file://` URIs by opening the target path with the associated application — this is application launch, not shell command execution. An attacker who knows the target's installed software or default binary locations can launch specific executables, providing a constrained but lower-severity capability than the Windows vector. This is a meaningfully different severity class: Windows achieves arbitrary RCE; macOS/Linux achieves targeted application launch. The High severity rating is driven primarily by the Windows attack path.

### The 22-Day Quiet Patch Window

The fix (commit 607b226) was released as mcp-remote 0.1.16 on June 17, 2025. No security advisory, no CVE, no changelog entry flagging the security significance of the release. The only signal was the commit itself.

Developers and organizations with automated dependency management might have picked up the upgrade incidentally. Those pinned to specific versions, using lockfiles, or managing npm updates manually had no mechanism to know the update was security-critical. This 22-day window — from silent fix to public CVE — represents the period during which the vulnerability was patched in the package but exploitation against users of older versions was possible with no warning driving upgrades.

The disclosure timing reflects a genuine trade-off in responsible disclosure: publishing a full advisory immediately arms defenders but also gives attackers a weaponizable PoC roadmap. The 22-day quiet window may have been a deliberate strategy to allow ecosystem adoption of the fix before advertising the attack surface. However, for a critical RCE with a working PoC, some minimal signal — a "security release" note in the changelog or a npm security advisory without PoC details — would have preserved the trade-off without leaving users entirely uninformed. The absence of any signal, not the delay itself, is the failure.

## Technical Analysis

### Vulnerable Code Path: Why open() Is a Shell Invocation

The `open()` npm package is a browser launcher utility. Its implementation on each platform calls the OS mechanism for opening URLs: `start` on Windows, `/usr/bin/open` on macOS, `xdg-open` on Linux. These mechanisms interpret special characters according to the shell or platform launcher's rules. When they are handed a URL string that has not been sanitized against shell metacharacter injection, they are equivalent to shell command execution with attacker-controlled input.

The critical design assumption violated here: mcp-remote treats the `authorization_endpoint` URL from the OAuth metadata response as trusted input suitable for passing directly to a launcher. In any remotely authenticated protocol, server-provided data must be treated as untrusted. The OAuth metadata response originates from a server the client has not yet authenticated — it is, by definition, the data received before authentication completes. Trusting the content of pre-authentication server data as safe for OS-level operations is the vulnerability class in its simplest formulation.

### Windows Exploitation: PowerShell $() Subexpression Operator

The PowerShell `$()` subexpression operator evaluates an expression inside a string context. When Windows `open()` implementations pass the URL to `cmd.exe` or PowerShell for interpretation, a `$()` in the URL string is evaluated before the "URL" is passed to the browser. The `a:` non-standard URI scheme is the enabling condition: standard URL parsing applies percent-encoding to special characters like `$`, `(`, `)`, turning them into `%24`, `%28`, `%29`. The `a:` scheme bypasses this normalization in many URL parsing contexts, delivering the raw characters to the shell.

The attack requires:
1. Attacker-controlled MCP server (can be set up cheaply and trivially)
2. Victim connecting via mcp-remote (the default tool for this purpose)
3. OAuth metadata response with crafted `authorization_endpoint`
4. No user interaction beyond initiating the mcp-remote connection

### The "Experimental" Label and Ecosystem Adoption

The mcp-remote README carried an "experimental" label at the time of the vulnerability. This label is analytically irrelevant to the security consequences of 437,000 downloads of a package that was:

- Referenced in Anthropic's official MCP documentation and tutorials
- The only practical solution for the most common MCP deployment pattern (remote servers with OAuth, local stdio clients)
- Load-bearing infrastructure for every Claude Desktop, Cursor, and Windsurf user connecting to remote MCP servers

"Experimental" in software documentation signals to developers that APIs may change and stability guarantees are limited. It does not communicate "this package may execute arbitrary OS commands on your machine when you connect to a remote server." No developer reading that label would infer the applicable risk. The label was a documentation artifact that did not correspond to the package's actual role as critical AI infrastructure.

### Anthropic's Documentation Gap

Anthropic recommended mcp-remote in its documentation for the standard use case of connecting Claude Desktop to remote MCP servers — despite mcp-remote being a community package, not an Anthropic-maintained library. This creates an accountability gap characteristic of the AI ecosystem's growth phase: Anthropic provided the infrastructure expectations (OAuth-authenticated remote MCP servers), the community filled the gap (mcp-remote), Anthropic validated the solution through documentation, and the security posture of that solution was owned by neither party clearly.

Glen Maddern maintained mcp-remote in his capacity as an individual and in his role at Groq, not as an Anthropic employee. No evidence of Anthropic auditing or formally vetting mcp-remote's security before recommending it appeared during the disclosure period. No evidence of Anthropic issuing upgrade guidance to users after the patch was released emerged. The community package absorbed the adoption risk of official recommendation without the security review a first-party package would receive.

### Fix Scope: Narrow by Design

The fix in commit 607b226 applies `encodeURIComponent()` to the username and password components of the URL in a `sanitizeUrl()` function. This addresses the credential injection vector — an attacker embedding shell metacharacters in the username or password portion of the URL.

The narrow scope is notable: the underlying architectural issue — that a remote-server-provided URL reaches `open()` (an OS shell invocation) without comprehensive sanitization of all URL components capable of carrying injection payloads — may not be fully addressed. The `authorization_endpoint` URL itself is still attacker-controlled. The fix constrains one injection surface within the URL; whether other components of the URL can carry injection payloads on the same code path is not documented as having been analyzed or tested. Full remediation would validate the entire URL against an allowlist of safe schemes and characters before any OS launcher invocation, or replace `open()` with a browser API that does not route through the OS shell.

## Root Cause Analysis

**5 Whys:**

**Why 1:** Why could a malicious MCP server execute arbitrary OS commands on a developer's machine?

Because mcp-remote passed the `authorization_endpoint` URL from the server's OAuth metadata response directly to the `open()` npm package, which invokes an OS shell launcher on all platforms. Server-provided URL content containing shell metacharacters was evaluated by the OS shell, executing the embedded commands.

**Why 2:** Why was the server-provided URL passed unsanitized to an OS shell invocation?

Because the developer assumption was that `open()` is a browser launcher — a URL-handling utility — and that passing a URL to it is safe by definition. The abstraction of "launching a URL" conceals the implementation: `open()` achieves this by constructing an OS shell command. The abstraction was treated as a security boundary it does not actually provide. Pre-authentication server-provided data was treated as trusted input suitable for OS-level operations.

**Why 3:** Why did the implementation not validate or sanitize server-provided URLs before passing them to OS-level commands?

Because no secure-input-handling policy governed the treatment of data received from remote servers during the OAuth flow. The mcp-remote codebase was built to solve a functional problem — bridging MCP clients to remote OAuth servers — and was not designed with an explicit threat model for the case where the remote server is the attacker. The OAuth metadata response was treated as configuration data from a trusted party, not as attacker-controlled input from an unauthenticated source.

**Why 4:** Why was the package released and widely adopted without a threat model that included the malicious server scenario?

Because mcp-remote was built during the initial MCP ecosystem launch in an environment where adoption speed was the primary goal. The package filled a critical gap in the MCP ecosystem (no native OAuth support for remote servers in Claude Desktop, Cursor, or Windsurf), was informally validated by Anthropic's documentation references, and accumulated downloads rapidly. Security review — particularly for the threat model where the server being connected to is the attacker — was not part of the development or adoption process for a community package operating in an ecosystem with no established security review standards.

**Why 5 / Root cause:** Why was there no security review gate requiring threat modeling for the malicious-server scenario before a community package was recommended in official AI infrastructure documentation?

The AI agent ecosystem lacks the institutional maturity to enforce security standards on community packages that become load-bearing infrastructure through ecosystem adoption. No process existed requiring that packages recommended in Anthropic's documentation undergo security review. No standard required that OAuth client implementations treat server-provided metadata as untrusted input. The gap between "community package labeled experimental" and "critical infrastructure connecting 437,000+ developer machines to remote servers" was never reviewed as a security concern by anyone with authority to require remediation before the adoption curve ran. The root cause is an ecosystem-wide accountability gap: a package can become critical AI infrastructure through documentation references and adoption patterns without any security review gate activating.

**Root cause summary:** mcp-remote's OAuth implementation treated remote-server-provided metadata as trusted configuration rather than as attacker-controlled input, passing it to an OS shell invocation without sanitization — because no security review process existed requiring threat modeling for the malicious-server scenario before a community package achieved load-bearing adoption through official documentation references in the AI ecosystem.

## Impact Assessment

**Severity:** High

**Justification:** CVSS 9.6 is JFrog self-assigned, not independently reviewed by NVD. However, the underlying risk is genuinely high: full arbitrary RCE on Windows developer machines (EPSS: 6.169%, 91st percentile — high relative exploitation probability) with no user interaction beyond initiating a server connection. 437,000+ downloads, developer machines as targets, and a working PoC publicly available. High severity is defensible independent of the CVSS score controversy. The near-miss classification is appropriate: no confirmed exploitation was documented before or after disclosure, but "no confirmed exploitation" reflects absence of public reports, not forensic verification.

**Actual vs. potential:** Near-miss. The attack scenario requires an attacker to control a malicious MCP server and induce a victim to connect via mcp-remote — a plausible attack scenario (malicious MCP servers listed on third-party registries, targeted social engineering of developers, supply chain poisoning of legitimate MCP server packages). The 22-day quiet patch window created a period of maximum exposure with no advisory driving upgrades.

**Who was at risk:**
- All users of mcp-remote versions 0.0.5 through 0.1.15
- Claude Desktop, Cursor, and Windsurf users connecting to remote MCP servers with OAuth
- Organizations whose developers used these tools for AI-assisted development

**What was accessible post-exploitation:**
- Full OS-level command execution (Windows) or targeted executable launch (macOS/Linux)
- SSH private keys, cloud provider credentials (AWS, GCP, Azure), API tokens, OAuth tokens
- Source code on the developer's machine and any accessible network shares
- CI/CD configuration, deployment credentials, secrets management access
- Ability to inject malicious code into software the developer is building
- Browser session tokens and password managers accessible to the running user account

**Scale note:** "437,000+ downloads" measures download events, not unique installations. The actual population of running vulnerable installations is likely a fraction of this number. However, the developer machine attack surface is high-value: a single compromised developer machine with cloud credentials and source code access can cascade to organizational infrastructure at scale.

## How It Could Have Been Prevented

**1. Never pass remote-server-provided URLs to OS shell commands without comprehensive sanitization.**

Any URL received from an unauthenticated or potentially adversarial remote source — especially in pre-authentication flows like OAuth metadata discovery — must be treated as attacker-controlled input. Before passing any such URL to a launcher, browser API, or shell command, it must be validated against an explicit allowlist: permitted URI schemes (https:// only, for authorization endpoints), no shell metacharacters, no unusual characters in any component. The specific vulnerability here — `a:$(cmd)` scheme bypassing URL encoding — would be caught by any check requiring `https://` as the scheme before proceeding.

**2. Replace OS shell invocation with a browser API that does not route through the shell.**

The `open()` npm package achieves cross-platform browser launching by constructing OS shell commands. Any input to it is OS shell input. Alternative browser-launching approaches — using platform-specific APIs that do not construct shell strings — would eliminate the injection vector at the architectural level. At minimum, validating that the URL passed to `open()` uses only HTTPS and contains only URL-safe characters before calling `open()` would have prevented this specific attack.

**3. Apply the malicious-server threat model to all OAuth client implementations.**

OAuth flows are trust establishment protocols: the client is receiving information from a party whose trustworthiness has not yet been established. All data received before trust is established — including the OAuth metadata response, the `authorization_endpoint`, and any redirect URIs — must be treated as potentially adversarial. Any OAuth client implementation in the AI agent ecosystem should include explicit threat modeling for the scenario where the server being authenticated against is the attacker.

**4. Establish a security review requirement before recommending community packages in official AI documentation.**

A package recommended in official documentation — regardless of its "experimental" label — will be adopted as infrastructure. Anthropic should not recommend community packages for security-sensitive operations (OAuth, credential handling, shell operations) without completing a security review. The review need not be exhaustive: a basic threat model exercise asking "what happens if the remote server is malicious?" would have surfaced this vulnerability.

**5. Implement disclosure-coordinated upgrade communications.**

When a security vulnerability is patched in a widely-adopted community package, the 22-day gap between silent patch and public CVE should not exist. Coordinated disclosure should include a mechanism for communicating upgrade urgency to users — an npm security advisory, a changelog entry explicitly flagging the security nature of the release, or coordination with npm's security notification infrastructure — so users can upgrade before the CVE weaponizes the PoC.

## How It Was / Could Be Fixed

**Actual remediation taken:**

Commit 607b226 ("...") by Glen Maddern, released as mcp-remote 0.1.16 on June 17, 2025.

The fix applies `encodeURIComponent()` to the username and password components of the URL inside a `sanitizeUrl()` function in the OAuth client provider. This encodes shell metacharacters in those URL components (turning `$` into `%24`, `(` into `%28`, etc.), preventing the PowerShell subexpression injection via credential-bearing URLs.

**Fix scope limitation:**

The fix targets credential components of the URL. The full `authorization_endpoint` value from the server's metadata is still attacker-controlled. Whether other URL components (path, query parameters, fragment, or the scheme itself) can carry injection payloads on the same call path to `open()` is not documented as having been analyzed. A comprehensive fix would validate the entire URL structure before passing it to `open()`:

```javascript
function validateAuthorizationEndpoint(url: string): void {
  const parsed = new URL(url);
  if (parsed.protocol !== 'https:') {
    throw new Error('Authorization endpoint must use HTTPS');
  }
  // Additional checks for safe characters, allowlisted domains, etc.
}
```

**What was not done:**
- No public security advisory issued alongside the patch; 22-day gap before CVE disclosure
- No architectural review of whether `open()` is the appropriate mechanism for launching OAuth authorization URLs — a browser API that does not route through the OS shell would be safer
- No explicit threat model documentation for the malicious-server scenario added to the mcp-remote codebase or documentation
- No communication from Anthropic to users of mcp-remote via official channels about the upgrade requirement
- No update to MCP documentation removing or qualifying the mcp-remote recommendation to include security guidance

**Recommended additional measures:**
- Replace `open()` with a browser-launching approach that validates URL structure and does not construct OS shell strings
- Add an automated test asserting that `authorization_endpoint` URLs containing shell metacharacters are rejected or properly encoded at all points in the OAuth flow
- Publish explicit threat model documentation for mcp-remote covering the malicious-server scenario
- Add a `security.md` or security contact to the mcp-remote repository for future disclosures

## Solutions Analysis

### 1. URL Scheme Allowlisting Before OS Launcher Invocation

- **Type:** Input Validation / Secure Defaults
- **Plausibility:** 5/5 — Trivial to implement and fully prevents the injection vector. The `a:` URI scheme that enables Windows RCE is non-standard and has no legitimate use as an OAuth authorization endpoint. Any check requiring `https://` as the scheme before calling `open()` would have blocked the PoC payload entirely. This is a one-line guard with no false positives for legitimate OAuth flows.
- **Practicality:** 5/5 — Zero impact on legitimate use. All real OAuth authorization endpoints use HTTPS. Blocking non-HTTPS authorization endpoints is both a security improvement and a correct OAuth implementation posture. Can be added to any version of mcp-remote as a backward-compatible change.
- **How it applies:** `if (!url.startsWith('https://')) throw new Error(...)` before any call to `open()` in the OAuth flow would have prevented the disclosed Windows RCE. The scheme check should be supplemented with validation of all URL components against a safe character allowlist.
- **Limitations:** Addresses the scheme-based bypass specifically. A comprehensive fix requires validating all URL components, not just the scheme. Does not address the architectural issue of passing any remote-server-provided URL to an OS shell invocation — the correct long-term fix is replacing `open()` with a shell-free browser launcher.

### 2. Replace open() with Shell-Free Browser Launching

- **Type:** Architectural / Secure by Design
- **Plausibility:** 4/5 — Platform-specific browser launching APIs that do not construct OS shell strings exist on all three supported platforms. Node.js's `child_process.spawn()` with explicit binary paths and sanitized argument arrays (rather than shell-interpreted command strings) avoids shell interpretation. This addresses the root architectural issue rather than patching the symptom.
- **Practicality:** 3/5 — Requires replacing the `open()` dependency and testing across Windows, macOS, and Linux. The cross-platform complexity is why `open()` was chosen initially — it abstracts platform differences. Implementing a shell-free equivalent requires platform-specific logic and more maintenance. However, for a security-critical OAuth client, this investment is appropriate.
- **How it applies:** Would eliminate the entire class of OS shell injection vulnerabilities via the browser-launching code path, not just the specific injection vectors documented in CVE-2025-6514. Future changes to the OAuth flow could not reintroduce this class of vulnerability because the shell invocation would no longer exist.
- **Limitations:** Higher implementation complexity than URL sanitization. Requires thorough cross-platform testing. Does not prevent injection vectors in other parts of the codebase that might reach `open()` or similar shell-invoking functions. The dependency upgrade cycle for mcp-remote users still applies — users on affected versions still need to upgrade.

### 3. Coordinated Disclosure with Concurrent Public Advisory

- **Type:** Process / Vulnerability Management
- **Plausibility:** 5/5 — Standard responsible disclosure practice in the security industry includes releasing a public advisory simultaneously with or immediately following a patch, giving users the information they need to act. JFrog and Glen Maddern chose a 22-day quiet window before public disclosure. A coordinated disclosure policy requiring simultaneous advisory publication is straightforward to adopt.
- **Practicality:** 4/5 — Requires coordination between JFrog (researcher), Glen Maddern (maintainer), and Anthropic (documentation recommender). The coordination overhead is minimal compared to the risk the quiet window created. npm's security advisory infrastructure supports package-specific security notifications that users can subscribe to.
- **How it applies:** The 22-day gap was a risk amplifier, not the root vulnerability. But it extended the window during which users had no signal to upgrade. A policy of concurrent advisory publication — or at most a 7-day grace period for critical infrastructure packages — would have minimized this window.
- **Limitations:** Does not address the technical vulnerability. A public advisory without a patch gives attackers a head start; the 22-day gap was partly defensive against this. The right balance for a critical RCE in infrastructure used by 437,000 developers warrants discussion, but some public signal (a "security release" changelog entry) should have been present from day one.

### 4. Pre-Adoption Security Review for Documented Community Packages

- **Type:** Process / Organizational Control
- **Plausibility:** 4/5 — Anthropic referencing mcp-remote in documentation elevated its adoption from a community package to critical AI infrastructure. A security review requirement before documentation references — covering at minimum the threat model for the malicious-server scenario and a code audit of credential and URL handling — is a feasible organizational control.
- **Practicality:** 3/5 — Requires Anthropic to define a review process, assign ownership, and enforce the gate before documentation updates. For a fast-moving ecosystem with many community packages, this introduces friction. However, the review scope for a package like mcp-remote is narrow: OAuth flow security, URL handling, credential handling, and shell invocations. A focused review against this checklist could be completed in a day.
- **How it applies:** A pre-documentation review of mcp-remote asking "what happens if the server being connected to is malicious?" would have surfaced the `open()` invocation with server-provided URL content as a high-priority finding. The vulnerability is not subtle — it is visible in the call graph of the OAuth flow.
- **Limitations:** A one-time review at documentation inclusion does not catch vulnerabilities introduced in subsequent package versions. The review gate addresses the documentation adoption vector but requires ongoing monitoring of recommended packages for security regressions.

### 5. Ecosystem-Level OAuth Client Security Standard for MCP

- **Type:** Ecosystem / Standards
- **Plausibility:** 3/5 — An MCP ecosystem security standard specifying minimum requirements for OAuth client implementations — including requirements to treat server-provided metadata as untrusted, to validate authorization endpoint URLs before use, and to avoid OS shell invocations for URL handling — would provide a baseline that package authors implement against. The Anthropic MCP team has the authority and influence to define such a standard.
- **Practicality:** 2/5 — Standards take time to draft, socialize, and adopt. In a fast-moving ecosystem with many community contributors, enforcement is voluntary without tooling. The MCP specification would need to include security requirements, not just protocol mechanics, for this to reach package authors systematically.
- **How it applies:** Would establish the expectation that all MCP OAuth client implementations treat the authorization endpoint as untrusted input — a fundamental requirement that mcp-remote violated. Future packages built to this standard would not make the same architectural assumption.
- **Limitations:** Standards have the longest lead time of all solutions. They do not help the 437,000 existing downloads of affected mcp-remote versions. They require ongoing maintenance as new attack patterns emerge. The real value is in preventing the next iteration of this class of vulnerability across the ecosystem, not in remediating the current incident.

## Strategic Council Review

### Challenger Findings

1. **Download count conflated with installation count**: The report uses "437,000+ developer machines exposed" as though npm download events equal unique running installations. npm download counts include CI/CD pipeline pulls, automated rebuilds, mirror fetches, and duplicate installs on the same machine. The actual population of running vulnerable instances was almost certainly a fraction — possibly a small fraction — of 437,000. The draft acknowledges this in one sentence in the Impact Assessment but then allows the headline figure to carry severity framing throughout, including in the frontmatter `headline_stat`. This inflates the scale picture without evidence.

2. **CVSS 9.6 treated as legitimate despite explicit contradictory sourcing**: The frontmatter tags `cvss-96` and `epss-91st-percentile` without qualification, and the severity argument leans on this score even though the report itself correctly notes NVD marked it "Not scheduled for NVD enrichment prioritization" and that JFrog — a commercial security vendor with reputational interest in high-severity discoveries — self-assigned the score. Structurally inconsistent: acknowledging source bias in one sentence while using the score as evidence is not the same as resolving the source bias concern.

3. **macOS/Linux severity materially overstated**: The report describes macOS/Linux impact as "constrained but non-trivial arbitrary execution capability" via `file://` URIs. macOS `open` and Linux `xdg-open` do not execute shell commands when given a `file://` path — they open the file with the associated application. This is application launch, not OS-level RCE. The severity differential between Windows (arbitrary command execution, CVSS 9.6 defensible) and macOS/Linux (targeted application launch, substantially lower severity) is real and significant. Conflating them inflates the aggregate severity picture.

4. **22-day disclosure gap characterized as unambiguous failure without engaging the standard trade-off**: The report calls the quiet patch window a "risk management failure" without substantively engaging the opposing argument: simultaneous full advisory publication gives attackers a PoC roadmap before users have patched. The 22-day period may have been a deliberate, defensible strategy to allow ecosystem adoption before weaponizing. The solutions section briefly acknowledges this ("partly defensive") but the body language of the analysis is one-sided on a genuinely contested disclosure timing question.

5. **Root cause Why 5 is unfalsifiable and non-actionable**: "The AI agent ecosystem lacks institutional maturity to enforce security standards" is structurally unfalsifiable — it cannot be tested, cannot be fixed by a specific party, and applies equally to any software supply chain vulnerability in any emerging ecosystem. The specific and actionable root cause appears in Why 4: a community OAuth client treated pre-authentication server responses as trusted configuration, with no threat model for the malicious-server scenario. Why 5 provides contextual backdrop, not a root cause. Presenting it as the terminus of the 5 Whys chain dilutes the actionable finding.

6. **Fix scope critique asserts incomplete remediation without demonstrating a surviving exploit path**: The report argues the `encodeURIComponent()` fix "may not be fully addressed" because other URL components could carry injection payloads. This is theoretically plausible but not demonstrated. The specific disclosed PoC uses the `a:` scheme with `$()` subexpression in credential-bearing components — exactly what `encodeURIComponent()` targets. Whether the scheme, path, or query components provide surviving injection vectors post-fix would require code audit of 0.1.16 to establish. The architectural concern is valid; the implied surviving vulnerability is unsubstantiated.

### Steelman Defense

1. **High severity is independently defensible without the CVSS score**: The Windows mechanism is textbook Critical/High regardless of who scored it: remote attacker, no authentication, no user interaction, arbitrary command execution under the user's account. A standard CVSS v3.1 calculation against this mechanism would converge near 9.x regardless of who performs it. The source bias concern is valid as a sourcing caveat, not as a severity rebuttal. The report correctly hedges on the JFrog self-assignment while maintaining High, which is the right analytical posture.

2. **Fix scope critique is architecturally correct even if the PoC is fully resolved**: Whether or not the `encodeURIComponent()` fix closes the specific disclosed attack path, the underlying architecture — attacker-controlled server data reaching an OS shell invocation — is a structural vulnerability class, not a single instance. The next creative payload against the same code path would not need to use the same characters. Recommending a URL scheme allowlist and replacement of `open()` with a shell-free browser launcher is architecturally correct as a defense-in-depth posture regardless of whether a surviving exploit exists today.

3. **Anthropic documentation accountability gap is well-evidenced and analytically novel**: The facts are uncontested: mcp-remote appeared in Anthropic's MCP documentation; the maintainer is not an Anthropic employee; no public evidence of security review before documentation inclusion exists; no evidence of Anthropic issuing upgrade guidance after the patch appears. This finding generalizes to a pattern: "recommended in official documentation" and "security-reviewed" are not equivalent in the current AI ecosystem. This is the report's most significant systemic contribution and it holds up fully.

4. **Why 4 is specific, falsifiable, and repairable — Why 5 contextualizes it correctly**: The report's root cause summary correctly foregrounds Why 4 as the proximate root cause: "mcp-remote's OAuth implementation treated remote-server-provided metadata as trusted configuration rather than as attacker-controlled input... because no security review process existed requiring threat modeling for the malicious-server scenario." That is specific and remediable. Why 5 provides institutional context explaining why Why 4 was able to persist undetected — it is contextual framing, not a competing root cause. The 5 Whys structure is sound; the issue is presentational, not analytical.

5. **The bidirectional MCP attack surface finding is substantive and underrepresented in existing literature**: Most MCP security guidance addresses what the server can be made to do (prompt injection, tool misuse). This report documents the converse — a malicious server attacking the client during the OAuth handshake — with a specific, working mechanism. Paired with the related incidents in the mcp-protocol-security-crisis pattern group, this advances the understanding that MCP attack surfaces are bidirectional and that connecting to an untrusted server is not a passive act. This is a genuine analytical contribution.

### Synthesis

The report is analytically rigorous on its core claims. The Windows RCE mechanism is accurately described, the sourcing is transparent about its limitations, and the Anthropic accountability gap finding is novel and well-supported. Three targeted corrections are required: (1) the headline and Executive Summary have been updated to distinguish download events from unique installations; (2) the macOS/Linux severity description has been corrected to clarify that `file://` URI handling is application launch, not shell command injection; and (3) the 22-day disclosure gap section has been amended to fairly represent the disclosure timing trade-off before characterizing the absence of any signal as the failure.

The CVSS source bias concern (Challenge 2) is addressed by the existing hedge language and does not require further update — the severity rating is independently defensible. The fix scope concern (Challenge 6) is valid architecturally and the report's architectural recommendation stands, but asserting a specific surviving exploit path would require code audit evidence not currently available. The Why 5 root cause concern (Challenge 5) is resolved by recognizing that Why 4 carries the actionable finding; Why 5 is contextual backdrop. The disclosure timing concern (Challenge 4) warranted the targeted addition to the body text; the core finding — no signal of any kind during the window — stands.

The near-miss classification is appropriate. "No confirmed exploitation" reflects absence of public reports, not forensic verification of non-exploitation. The 22-day window, the working PoC, and the EPSS 91st percentile together make exploitation plausible but undocumented. High severity is the correct aggregate rating driven by the Windows attack path.

**Confidence level:** Medium-High — the Windows RCE mechanism and sourcing are well-evidenced; the macOS/Linux severity is a genuine uncertainty now appropriately caveated; the "no confirmed exploitation" near-miss status means the damage ceiling is estimated, not observed; the download count versus installation count gap introduces uncertainty about actual exposure scale.

**Unresolved uncertainties:**
- Whether any surviving injection path exists in the full `authorization_endpoint` URL after the `encodeURIComponent()` fix — would require code audit of mcp-remote 0.1.16 across all URL components (scheme, path, query, fragment) to resolve
- Actual number of running vulnerable installations at time of disclosure — 437K download events is the only available proxy; lockfile analysis across affected repositories and npm download decomposition (unique IPs, CI vs. human) would be needed to narrow the real exposure count
- Whether the macOS/Linux `file://` application launch vector enables meaningful exploitation against real developer environments — would require platform-specific testing with representative developer tool configurations (IDE plugins, shell launchers, script associations) to assess practical severity

## Key Takeaways

**1. MCP attack surface is bidirectional: malicious servers can attack clients.**
Most MCP security guidance focuses on preventing malicious tool execution by agents — what happens when a server sends bad instructions. CVE-2025-6514 demonstrates the inverse: a malicious server attacking the client that connects to it. Security teams deploying AI agents must model both directions. Connecting to an untrusted MCP server is not a passive act — it initiates an authentication handshake that can be weaponized against the connecting machine.

**2. "Experimental" labels on load-bearing infrastructure are meaningless security signals.**
A package with 437,000+ downloads, referenced in official documentation, serving as the de facto standard for a critical capability (remote MCP OAuth) is not experimental in any meaningful sense. "Experimental" communicates API instability, not security unverifiability. Organizations deploying AI development tools should evaluate packages by their actual role and adoption — not by labels that no longer match their deployment reality.

**3. Pre-authentication server data must be treated as attacker-controlled.**
The OAuth metadata discovery phase receives data from a server before authentication is complete. That data is, by definition, untrusted. Any URL, credential, or parameter received from a remote server before authentication should be validated against an explicit allowlist before use in security-sensitive operations. The `authorization_endpoint` URL is the most obvious example: it must be HTTPS, must contain only safe characters, and must not be passed to an OS shell invocation in any form.

**4. Quiet security patches without advisories leave 437,000 users flying blind.**
The 22-day gap between silent patch (June 17) and public CVE (July 9) created a period where the fix existed but no advisory signaled urgency to upgrade. For security-critical packages, especially those involved in OAuth flows and shell operations, a "security release" changelog entry — even without full CVE details — provides the signal users need to prioritize upgrades. Silent security patches in critical infrastructure are a risk management failure, not a cautious disclosure strategy.

**5. Recommendations carry implicit security endorsements — they must be backed by security review.**
When Anthropic's documentation recommended mcp-remote as the standard bridge for connecting Claude Desktop to remote servers, it implicitly endorsed mcp-remote's security posture. That endorsement was not backed by a security audit. Any organization similarly recommending or adopting community packages for security-sensitive AI infrastructure operations should require a threat model review — especially for the malicious-counterparty scenario — before the recommendation is published.
