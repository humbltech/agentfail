# Claude Code Guide Agent Recommends `claude -p` Cron Pattern to Max Subscriber: $1,818 in API Charges Over Two Days

## Executive Summary

John Shelburne, a Claude Max 20x subscriber paying $200/month for unlimited interactive use, was charged $1,818 in API fees over two days (March 20–21, 2026) after following advice from Claude Code's own built-in `claude-code-guide` subagent, which recommended using `claude -p` with cron jobs to "maximize" a March 2026 2x usage promotion. When Shelburne explicitly asked the guide agent whether this pattern would work for Max subscribers, it confirmed it would — technically accurate in the sense that the commands would execute, but entirely silent on the fact that `claude -p` unconditionally routes to `ANTHROPIC_API_KEY` billing in non-interactive mode, bypassing the Max subscription entirely. The billing-path switch was undocumented at the point of interaction, Anthropic's GitHub issue tracking the problem was marked stale with zero staff responses, and a near-identical incident (vinsidious, $1,700 overnight) occurred in the same period.

## What Happened

John Shelburne ran CatFix Technology and subscribed to Claude Max 20x at $200/month — a plan designed to provide high-volume, unrestricted interactive use of Claude without per-token API billing. In March 2026, Anthropic launched a 2x usage promotion for Max subscribers, encouraging heavier use of the platform.

Shelburne wanted to automate Claude Code workflows using cron jobs to take advantage of the promotion. Before setting this up, he did what a responsible developer should do: he asked the built-in `claude-code-guide` subagent — Claude Code's own documentation and onboarding assistant — whether the `claude -p` flag (which enables non-interactive "print" mode, allowing Claude Code to be piped or scripted) would work for Max subscribers in cron jobs. The guide agent confirmed that it would.

The confirmation was not wrong in the narrow operational sense. `claude -p` does execute in non-interactive mode. Claude Code does run in cron jobs. The commands do complete. But the guide agent's answer was fatally incomplete: it did not mention — and Shelburne had no reason to independently discover — that when `ANTHROPIC_API_KEY` is present in the environment, `claude -p` routes all API calls through the API key unconditionally, bypassing the OAuth-authenticated Max subscription entirely. The Max subscription covers only interactive Claude Code sessions. The moment Shelburne's cron jobs ran `claude -p`, every request was billed at full API rates.

Shelburne's setup compounded this into significant financial exposure. He was using Claude Opus 4.6 — the highest-cost model tier — with approximately 47,000 input tokens per request. Running dozens of requests per hour over two full days at Opus 4.6 pricing produced $1,818 in charges, all billed to his API key rather than covered by his $200/month Max plan.

The billing dashboard provided no real-time warning. The `claude -p` command displayed no notice that it had switched billing modes. The guide agent had already told him the approach would work. By the time Shelburne discovered what had happened, two days of automated API usage had accumulated.

He filed GitHub issue #37686 on March 23, 2026. The issue was labeled and remained open. No Anthropic staff member responded to it. A separate but near-identical incident from user "vinsidious" — who hit $1,700 in overnight charges from what they described as a "ralph loop" on their 20x Max plan — appeared in issue comments around the same period, suggesting the pattern was not isolated.

## Technical Analysis

### The `claude -p` Authentication Precedence Mechanism

Claude Code supports two authentication modes:

1. **OAuth / Max subscription authentication**: Used for interactive Claude Code sessions. Charges are covered by the Max plan subscription.
2. **`ANTHROPIC_API_KEY` authentication**: Used when the environment variable `ANTHROPIC_API_KEY` is set. Charges are billed per-token to the associated API account.

The credential priority rule, documented as intentional behavior, is:

```
ANTHROPIC_API_KEY (env var) > OAuth subscription token
```

When running in non-interactive mode (`claude -p`), Claude Code checks for `ANTHROPIC_API_KEY` in the environment. If present, it uses it — unconditionally, without prompting, without displaying a warning that billing mode has switched from subscription to per-token. The OAuth subscription is silently bypassed.

This behavior is by design. It serves legitimate use cases: developers building pipelines against the API who have Claude Code installed should be able to use their API key without OAuth interfering. The problem is not that the behavior exists — it is that the switch is invisible, that the guide agent was not equipped with billing-context awareness, and that no warning appears at the point of mode transition.

### Token Math and Charge Calculation

With Opus 4.6 pricing (the highest-cost model at the time):

- Input: ~$15 per million tokens
- Output: ~$75 per million tokens
- Per-request input cost at 47,000 tokens: approximately $0.71 in input tokens alone
- At 30 requests/hour (a conservative estimate for cron jobs running frequently): approximately $21/hour in input tokens
- Over 48 hours: approximately $1,008 in input tokens, plus output tokens

The $1,818 figure is consistent with this math at moderate request frequency. Higher request densities (dozens per hour) align with the reported figure without requiring extraordinary volumes — this is what high-frequency Opus usage looks like when the cost control (Max subscription) is silently not active.

### How the Guide Agent's Recommendation Was Misleading

The `claude-code-guide` subagent is a built-in documentation and assistance resource embedded in Claude Code. Its role is to help users understand how to use Claude Code effectively.

When Shelburne asked whether `claude -p` with cron jobs would work for Max subscribers, the guide agent faced a question with two distinct correct answers:

- **Operationally correct**: Yes, `claude -p` executes in non-interactive mode and works in cron jobs.
- **Billing-accurate complete answer**: No — not in the way you intend. `claude -p` with an API key in the environment routes to API billing, not your Max subscription. To use Claude Code in headless/automated mode while remaining on your Max plan, use `claude setup-token` to configure subscription-based auth for non-interactive use.

The guide agent gave the first answer and omitted the second. Whether this was because:
- The guide agent's training or knowledge base did not include the billing precedence rule in the context needed to answer this class of question
- The guide agent's prompt context did not include the user's subscription status or environment configuration
- The guide agent's optimization function (answer the literal question asked) did not include a "billing accuracy" obligation

...is not determinable from the available evidence. What is clear is that a guide agent embedded in a product has an implied warranty to give billing-accurate answers, not just operationally-accurate answers, particularly when the user's question is explicitly about a paid subscription.

### Why Cron Jobs Are a Particularly High-Risk Exposure Pattern

The `claude -p` billing-path issue affects any non-interactive use. But cron jobs amplify the financial exposure in a specific way: they run unattended, on a schedule, without a human in the loop to notice that billing has silently switched modes. A user who runs `claude -p` interactively will see their terminal output and likely notice anomalies. A user whose cron job runs 40 times/hour overnight has no opportunity to observe the problem until the charges have accumulated.

The combination of:
1. Silent billing-path switch (no warning at mode transition)
2. High-cost model (Opus 4.6)
3. High token volume per request (~47,000 input tokens)
4. Automated/unattended execution (cron)
5. No spending cap
6. No real-time billing visibility

...is structurally identical to AAGF-2026-001's five-mechanism compounding pattern, with the `claude -p` billing-path switch substituting for the cache TTL issue as the primary failure-enabling mechanism.

## Root Cause Analysis

**Proximate cause:** The user's cron jobs used `claude -p` in an environment where `ANTHROPIC_API_KEY` was set, causing all API calls to be billed at per-token API rates rather than covered by the Max subscription, resulting in $1,818 in charges over two days.

**Why 1:** The user configured this pattern because the built-in `claude-code-guide` agent recommended it and explicitly confirmed it would work for Max subscribers, providing no information about the billing-path consequence.

**Why 2:** The `claude-code-guide` agent gave an incomplete answer because — whether by training, prompt configuration, or capability boundary — the agent was optimized to answer operational questions ("does this work?") without billing-accuracy obligations ("does this work the way you financially expect it to?"). The agent had no mechanism to detect the billing-path risk embedded in the user's question.

**Why 3:** The guide agent had no billing-accuracy mechanism because Anthropic did not design the agent with subscription-context awareness or billing-safety obligations. The guide agent could not access the user's subscription status, did not know whether `ANTHROPIC_API_KEY` was set in their environment, and was not prompted to surface billing caveats when recommending automation patterns.

**Why 4:** Anthropic did not design the guide agent with these constraints because the billing-path precedence behavior (`ANTHROPIC_API_KEY` overrides OAuth) was documented as intended technical behavior, not as a safety-critical UX concern. The failure mode — a guide agent recommending a pattern that silently bypasses the user's subscription — was not anticipated as an agent safety problem, only as a documentation gap.

**Why 5 / Root cause:** Anthropic's Claude Code architecture treats billing-path switching as a low-level credential-precedence rule rather than a user-facing safety boundary, and does not require built-in agents (including its own onboarding guide) to be aware of or warn about billing-path consequences when recommending automation patterns. When a platform embeds an agent that gives advice, that agent inherits an implied obligation to give billing-accurate advice — an obligation the platform design does not enforce.

**Root cause summary:** Anthropic embedded a guide agent in Claude Code that recommends automation patterns without billing-accuracy awareness, in a product where credential precedence silently switches billing modes without user notification, creating conditions where following the platform's own advice produces unexpected API charges for subscription users.

## How It Could Have Been Prevented

1. **Guide agent billing-accuracy constraint.** The `claude-code-guide` subagent should have been designed with an explicit obligation to surface billing caveats when recommending automation patterns. Concretely: if a user asks whether a pattern "works" for Max subscribers, the agent's answer should include "operationally yes, but note that this pattern routes to API billing if `ANTHROPIC_API_KEY` is set — use `claude setup-token` for subscription-safe headless use." This is a prompt-engineering and agent-design responsibility, not a user responsibility.

2. **Warning at billing-mode transition in `claude -p`.** Every time `claude -p` is invoked and the active billing mode is `ANTHROPIC_API_KEY` rather than Max subscription OAuth, a single-line warning should be printed to stderr before execution: `Warning: Non-interactive mode is using API key billing, not your Max subscription. Set spending limits at console.anthropic.com.` This costs zero engineering complexity — it is a conditional print statement. It would have stopped this incident.

3. **Mandatory spend cap prompt at API key creation / first automation pattern use.** If Anthropic had implemented a mandatory spending cap enrollment step when users first configure `ANTHROPIC_API_KEY`, or when Claude Code first detects a non-interactive invocation, the blast radius would have been bounded. A default daily cap of $50–$100 would have terminated the cron job pattern well before $1,818 accumulated.

4. **Pre-execution billing-mode disclosure for non-interactive invocations.** A startup check in `claude -p` could display — once, with a "don't show again" opt-out — which billing mode is active and what the approximate cost implications are at the current model and context size. This aligns with AAGF-2026-001's Solution 5 (pre-execution cost warning) and addresses the same root transparency gap.

5. **`claude setup-token` as the documented first path for headless use.** The `claude setup-token` command (added post-incident) configures subscription-aware authentication for non-interactive use. Had this been the prominently documented path for automation in the Max subscription context — rather than `claude -p` with an API key — users would have had a safe default to follow rather than discovering the hard way that API keys take precedence.

## Solutions Analysis

### Solution 1: Billing-Mode Warning at `claude -p` Invocation

- **Type:** Guardrails / output filters
- **Plausibility:** 5/5 — Trivially implementable. A conditional print statement in the CLI startup path checking whether API key billing is active in non-interactive mode. Requires no new infrastructure, no model changes, no significant engineering effort.
- **Practicality:** 5/5 — Can be shipped in a single PR. Has no downside for API-only users (the warning is only surfaced when a Max subscriber's ANTHROPIC_API_KEY might be active). Can include a `--no-billing-warning` flag for users who explicitly want to suppress it.
- **How it applies:** Directly addresses the silent billing-mode switch that made this incident possible. Shelburne would have seen "Warning: API key billing active — this is not covered by your Max subscription" before the first cron job ran and would have stopped immediately.
- **Limitations:** Only effective if the user sees the warning before the automated run begins. For cron jobs where stderr is not monitored, the warning would be logged but not acted upon in real time. Requires the user to know what to do with the warning (which is why Solution 5's documentation path matters as a complement).

### Solution 2: Guide Agent Billing-Accuracy Constraint

- **Type:** Policy and governance controls
- **Plausibility:** 4/5 — Implementable via prompt engineering, system prompt updates, or fine-tuning the guide agent's knowledge base to include billing-context awareness. The technical path is clear; the challenge is defining and maintaining the billing-accuracy rules as the product evolves.
- **Practicality:** 3/5 — Requires sustained investment: the guide agent's billing knowledge must be kept current with every billing model change (new models, new subscription tiers, new precedence rules). Prompt-engineering solutions can be brittle; edge cases will always exist. However, the high-value cases (Max subscribers asking about automation patterns) are a tractable and documentable subset.
- **How it applies:** If the guide agent had answered "Yes, this works operationally, but note that `claude -p` with an API key in your environment routes to API billing rather than your Max subscription — for subscription-safe headless use, see `claude setup-token`," the incident would not have occurred. The user followed the agent's advice; the agent's advice was incomplete.
- **Limitations:** LLM-based agents cannot guarantee billing-accuracy in all cases. Novel configurations, new subscription tiers, or edge cases in credential precedence may produce incorrect answers even with improved prompting. This solution reduces probability of recurrence but cannot eliminate it. Requires ongoing maintenance as billing models change.

### Solution 3: Mandatory Spending Cap Enrollment

- **Type:** Permission scoping / least privilege
- **Plausibility:** 5/5 — Technically trivial. Other API platforms enforce spend limits during onboarding. Anthropic's Console already supports workspace spend limits; the missing element is making enrollment mandatory (or at minimum, explicitly prompted) rather than optional.
- **Practicality:** 4/5 — Requires a one-time change to the API key creation flow and the Max subscription onboarding flow, plus backend enforcement for the new default cap. Enterprise/invoiced customers may require exemptions or higher default caps, adding a small amount of tiering complexity.
- **How it applies:** A default daily API spend cap of $50 would have terminated the cron job pattern after approximately 2–3 hours rather than allowing two full days of accumulation. The maximum exposure would have been $50–$150 rather than $1,818.
- **Limitations:** Does not prevent billing-mode switching — only limits blast radius. A user who knowingly raises or removes the cap to enable legitimate high-volume workloads is again unprotected. Does not address the guide agent's incomplete advice. Effective as a last-resort financial guardrail, not as a root-cause fix.

### Solution 4: Human-in-the-Loop Confirmation for Automated Billing-Mode Changes

- **Type:** Human-in-the-loop (HITL)
- **Plausibility:** 3/5 — Technically feasible. Claude Code could require explicit confirmation ("You are about to run Claude Code in non-interactive mode using API key billing. Estimated cost at current model/context: $X per run. Confirm? [y/N]") when a non-interactive invocation is detected in an environment where Max subscription credentials are also present.
- **Practicality:** 2/5 — Interactive confirmation is incompatible with cron jobs and CI pipelines by definition. A `--yes` flag to bypass the prompt would be necessary, defeating the HITL protection for exactly the users most at risk (those running automated, unattended workflows). The warning variant (Solution 1) achieves the informational goal without the operability cost.
- **How it applies:** For the initial cron job setup (where Shelburne would have been in a terminal), an interactive confirmation prompt would have surfaced the billing-mode information before any automated runs began. It would not have protected against subsequent unattended cron executions if bypassed.
- **Limitations:** Fundamentally incompatible with automation use cases unless bypass flags are available, which then eliminate the protection for the highest-risk scenarios. Better implemented as Solution 1 (warning) rather than blocking confirmation.

### Solution 5: Real-Time Billing Mode Status and Monitoring

- **Type:** Monitoring and detection
- **Plausibility:** 4/5 — Feasible. A `claude billing-status` or similar command could display the active billing mode, current session costs, and daily API spend against any configured cap. The underlying data is available to the Claude Code runtime.
- **Practicality:** 3/5 — Requires a dedicated CLI subcommand and integration with Anthropic's billing API for near-real-time data. The multi-day dashboard lag (documented in AAGF-2026-001) would need to be addressed for this to be useful. Medium engineering investment.
- **How it applies:** If Shelburne had run `claude billing-status` after setting up his cron jobs (or if the cron job output included billing status), he would have immediately seen that API key billing was active at Opus 4.6 rates and could have corrected the configuration before the first overnight run accumulated significant charges.
- **Limitations:** Requires proactive use by the developer — the at-risk scenario is precisely when developers are not actively monitoring. Most effective when paired with push alerting (email/webhook when daily API spend exceeds a threshold) rather than as a pull-only status command.

## Strategic Council Review

### Challenger Findings

**1. Is $1,818 really "High" severity, or is this a Medium incident being pattern-inflated?**

The taxonomy defines High as $10K–$100K financial loss. The documented charge is $1,818 — well below the High threshold and technically within Medium range. The report justifies High based on pattern-level framing (multiple affected users, aggregate exposure). This is a defensible analytical choice for an incident database, but it should be stated explicitly rather than implied. If every $1,000–$2,000 incident is rated High because it belongs to a pattern group with higher-severity members, the severity scale loses discriminatory value. A stronger framing would rate this incident Medium at the individual level and note the pattern-group High rating separately, or define a "pattern-multiplied severity" field.

**2. Is this a platform bug or documented behavior the user could have discovered?**

The `ANTHROPIC_API_KEY` precedence over OAuth is documented as by-design behavior. Anthropic's authentication documentation (post-incident update) explicitly warns about this precedence. The pre-incident documentation is not directly available in the source materials, but the incident description characterizes the behavior as "documented." If the behavior was documented — even inadequately — before March 2026, the "could have been discovered" bar is partially met. The counter-argument is that documentation that a developer using a guide agent would never encounter (versus documentation surfaced at the point of interaction) does not meaningfully satisfy the duty to warn. The distinction matters for assessing platform liability versus user responsibility.

**3. Was the guide agent's advice actually wrong, or just incomplete?**

The report frames the guide agent as having given misleading advice. But "misleading" implies a duty to provide complete billing context in response to an operational question. The guide agent said `claude -p` would "work" for Max subscribers — and it does work, operationally. If a user asks a doctor "can I take ibuprofen with this condition?" and the doctor says "yes," but the patient is also on a blood thinner the patient didn't mention, is the doctor giving wrong advice? The analogy is imperfect (the guide agent arguably had context clues about the subscription status from the user's question), but the "misleading" framing requires defending the implicit obligation that the guide agent should have volunteered billing caveats without being explicitly asked about billing.

**4. The vinsidious and nine-GitHub-issues citation is a pattern claim with variable source quality.**

The nine related GitHub issues are cited by number but not individually analyzed. Some may be closely analogous; others may share only superficial characteristics. The vinsidious "$1,700 overnight" incident is cited from comments in the primary issue — a secondary, unverified reference. Citing these as evidence of a "pattern spanning months" without individual verification risks overstating the breadth of the problem. The core claim (that `claude -p` API key precedence is a recurring billing harm) is well-supported by even the directly verifiable issues (#37686, #43333, #39903); the nine-issue citation should be qualified.

**5. Does the "guide agent confirming it would work" framing fairly represent what happened?**

The exact transcript of the guide agent interaction is not available in the source materials. The report characterizes the interaction based on Shelburne's account in the GitHub issue. Anthropic has not disputed this characterization, and the issue has received zero staff responses — but absence of dispute is not confirmation. It is possible the guide agent's actual response was more qualified, and Shelburne's account (written after he had already lost $1,818) frames it as more definitive than it was. This does not exonerate the platform — the billing-path switch was still undisclosed — but it affects the specific framing of the guide agent's culpability.

**1. The guide-agent-as-culpable-advisor framing is the most important insight in this report.**

The distinction is critical for platform liability and for solution design — fixing documentation or adding external warnings is insufficient when the problem is that the built-in advisory system actively guided the user toward the harmful pattern. Platform changes made in response to an incident are the strongest available evidence that the platform was at fault.

**4. The solutions analysis is actionable and proportionate.**

All five proposed solutions are technically implementable without novel research. Solutions 1 (billing-mode warning) and 3 (mandatory spend cap) have near-zero engineering cost and could have been shipped before this incident occurred. The guide agent billing-accuracy constraint (Solution 2) is more complex but directly addresses the most distinctive feature of this incident. The solutions are correctly ordered by implementation simplicity and impact, providing a clear prioritization for platform engineers.

## Key Takeaways

1. **Never trust a platform's own guide agent as the sole source of billing advice.** The `claude-code-guide` agent was optimized to answer operational questions ("does this work?") without billing-accuracy obligations. Any time you ask a built-in assistant whether a workflow will "work" for your subscription tier, independently verify the billing implications in the platform's authentication and pricing documentation. A guide agent that says "yes" is not certifying that the costs will be what you expect.

2. **`claude -p` is always API-key-billed when `ANTHROPIC_API_KEY` is in the environment — full stop.** Non-interactive Claude Code invocations (`claude -p`, piped commands, cron jobs, CI scripts) do not share billing with your Max subscription if `ANTHROPIC_API_KEY` is set in the environment. Use `claude setup-token` to configure subscription-aware authentication for headless use, or explicitly unset `ANTHROPIC_API_KEY` in any environment where Max subscription billing is expected. Verify your billing mode before running any automated workload.

3. **Cron jobs and unattended automation require a spending cap before the first run, not after.** The financial exposure in this incident accumulated over 48 unattended hours. A daily API spend cap configured before enabling the cron jobs would have bounded the damage to a fraction of the final charge. Treat an absence of a spending cap in an automated API workflow the same way you treat a missing timeout in a database query: a known footgun that will eventually fire.

4. **Platform silence on a filed issue is not the same as "working as intended."** GitHub issue #37686 was marked stale with zero Anthropic staff responses. Issue #43333 (same class) received a fix. The inconsistency is not evidence that the issue is resolved — it is evidence that Anthropic's triage process does not reliably surface billing harm reports. If you file a bug report about unexpected billing, follow up through support channels and do not assume "stale" means "fixed" or "not a real problem."

5. **The safest automation pattern uses the lowest-cost model that meets your quality bar.** Shelburne's cron jobs used Opus 4.6 with ~47,000 input tokens per request — the most expensive possible configuration. For automated tasks running at high frequency, the model choice is a financial safety decision, not just a quality decision. Establish a written policy for your automation stack: Sonnet or Haiku for high-frequency automation; Opus only for one-shot, high-value tasks where quality difference is verified. Then enforce it in your cron job configurations.
