# Antigravity: Prompt Injection to Sandbox Escape via Native Tool Flag Injection

## Executive Summary

In January 2026, security researcher Dan Lisichkin of Pillar Security discovered that Google's Antigravity IDE -- an agent-first development platform powered by Gemini -- could be exploited through a three-step attack chain: indirect prompt injection via malicious code comments, followed by agent-directed creation of a shell script, and finally, flag injection into the `find_by_name` tool's Pattern parameter that caused the underlying `fd` utility to execute the script as arbitrary code. The attack achieved full remote code execution on the developer's machine. Most significantly, Antigravity's Secure Mode -- the product's strictest security configuration, with VM isolation, network throttling, and sandboxed execution -- provided no protection, because native tool invocations execute before the security boundary evaluates the call. Google accepted the report through its Vulnerability Reward Program, patched the issue by February 28, 2026, and awarded a bug bounty. No in-the-wild exploitation was confirmed before the patch.

## What Happened

### The Discovery

Pillar Security, an Israeli cybersecurity company specializing in AI agent security, investigated the attack surface of Google's Antigravity IDE. Antigravity is Google's "agent-first" development platform, designed to delegate complex engineering tasks to autonomous AI agents powered by Gemini. The IDE provides native tools for file operations, code generation, and development assistance -- including a `find_by_name` tool that wraps the `fd` filesystem search utility.

Researcher Dan Lisichkin identified that the `find_by_name` tool passed user-supplied Pattern parameter values directly to `fd` without input validation or argument termination. Because `fd` interprets strings beginning with `-` as command-line flags, an attacker who could control the Pattern parameter could inject arbitrary `fd` flags -- including the `-X` (exec-batch) flag, which executes a specified binary against matched files.

### The Three-Step Attack Chain

The exploit combines two individually permitted operations -- file creation and file search -- to achieve an unpermitted outcome: arbitrary code execution.

**Step 1 -- Indirect Prompt Injection.** The attacker delivers adversarial instructions to the Antigravity agent through malicious comments embedded in source code files from public repositories. A developer performing normal workflow -- cloning a repository, opening a project, or reviewing code -- exposes the agent to the injected instructions. As Dan Lisichkin stated: a developer simply pulls a "benign-looking source file from an untrusted origin, such as a public repository, containing attacker-controlled comments that instruct the agent to stage and trigger the exploit." No interaction beyond normal development activity is required.

**Step 2 -- Malicious File Staging.** The injected prompt instructs the agent to create a shell script in the workspace. File creation is an unrestricted, permitted operation in Antigravity -- it is core IDE functionality. In the proof of concept, the agent creates a file `a.orange` containing `open -a Calculator`. In a real attack, the payload would be data exfiltration, backdoor installation, or source code modification.

**Step 3 -- Flag Injection via find_by_name.** The injected prompt instructs the agent to call `find_by_name` with Pattern: `-Xsh`. Because the Pattern parameter is passed to `fd` without sanitization and without a preceding `--` argument terminator, `fd` interprets `-X` as the exec-batch flag and `sh` as the binary to execute. `fd` then passes all matched files (including the staged `a.orange`) to `sh`, which executes them as shell scripts. Full arbitrary code execution is achieved.

A critical implementation detail: spaces within the Pattern string are treated as literal characters rather than argument separators by Antigravity's tool invocation layer. This is why the attacker uses `-Xsh` (no space) rather than `-X sh`. The ability to pre-stage files with controlled extensions in Step 2 circumvents this limitation.

### The Secure Mode Bypass

The most significant finding is not the flag injection itself -- it is that Antigravity's Secure Mode, the product's most restrictive security configuration, provided zero protection against this attack.

Secure Mode is designed to constrain agent operations through:
- Virtual machine isolation
- Network throttling
- File write restrictions to the working directory
- Sandboxed command execution

None of these controls prevented exploitation. The architectural reason: `find_by_name` is classified as a **native system tool**. When the agent invokes a native tool, the execution happens *before* Secure Mode's security boundary evaluates the call. The agent processes the invocation as an internal tool call, not as a shell command that would trigger sandbox evaluation.

Dan Lisichkin described the failure: "The security boundary that Secure Mode enforces simply never sees this call." The Cloud Security Alliance's analysis characterized this as "pre-evaluation execution authority" -- built-in tools were architecturally positioned outside the security perimeter they were supposed to enforce.

This means Antigravity users who enabled Secure Mode -- believing they were protected against exactly this class of attack -- received no security benefit against the actual threat.

### Related Antigravity Vulnerabilities

This incident exists within a broader pattern of Antigravity security findings:

- **"Forced Descent" (Mindgard):** A separate vulnerability enabling persistent MCP configuration backdoors via `.agent` rules that write malicious `mcp_config.json` to `~/.gemini/antigravity/`. The backdoor persists through complete application reinstall because global configuration loads at startup prior to any project trust verification.

- **Browser extension file write (Hacktron AI):** A path traversal vulnerability in Antigravity's `SaveScreenRecording` browser extension functionality. Google awarded a $10,000 bug bounty.

- **Fake Antigravity installer campaign (SecurityWeek):** Malware distributors created fake Antigravity installer packages targeting developers searching for the tool. This is not a vulnerability in Antigravity itself but indicates that threat actors are actively targeting the product's user base.

The CSA analysis noted that multiple independent security teams finding distinct vulnerability classes in the same product suggests "security architectures appear to have adapted existing approaches rather than developing models accounting for autonomous agent properties."

## Technical Analysis

### Architecture and Attack Surface

Google Antigravity provides AI agents with a set of native tools for filesystem operations, code generation, terminal access, and development workflows. The `find_by_name` tool is a wrapper around the `fd` utility (a modern alternative to `find`), designed to search for files by name pattern.

The tool accepts a Pattern parameter and passes it to `fd` as a command-line argument. The critical architecture decision: native tools execute within the agent runtime, outside the security boundary that Secure Mode enforces for shell commands and other externally-facing operations.

### The Flag Injection Mechanism

The `fd` utility, like most Unix command-line tools, interprets arguments beginning with `-` as flags. The standard defense is the `--` argument terminator, which signals that all subsequent arguments are operands (search patterns), not flags:

```bash
# Vulnerable: user input interpreted as flags
fd -Xsh

# Secure: -- prevents flag interpretation
fd -- "-Xsh"
```

Antigravity's `find_by_name` tool omitted both:
1. **Input validation:** No check that the Pattern parameter contains only legitimate search characters. Any string beginning with `-` is passed through to `fd` as-is.
2. **Argument termination:** No `--` separator before the user-supplied Pattern value.

The specific flag exploited, `-X` (exec-batch), is one of `fd`'s most powerful features -- it collects all matched files and executes a specified command against them. Combined with `sh` as the target binary, any matched file becomes a shell script to execute.

### Composition of Permitted Operations

The attack's elegance lies in combining two operations that are individually benign and permitted:

| Operation | Individually Permitted? | Individually Dangerous? |
|-----------|------------------------|------------------------|
| Create a file in workspace | Yes -- core IDE function | No -- files are inert data |
| Search for files by name | Yes -- core IDE function | No -- search is read-only |
| **Combined: create file + search with exec flag** | **N/A** | **Yes -- arbitrary code execution** |

This is a classic example of emergent capability risk: the security model evaluated each tool in isolation, but the dangerous behavior emerges from their composition. Neither tool's individual risk assessment would flag the combined attack.

### Why the PoC Used `.orange` Files

The proof of concept created files with the `.orange` extension to ensure that `fd`'s file matching would locate the staged payload. The extension is arbitrary -- any uncommon extension serves the same purpose by allowing the attacker to match only their staged files without interference from legitimate workspace files.

### Secure Mode Architecture Failure

The Secure Mode bypass is architecturally distinct from typical sandbox escapes. In most sandbox escape scenarios, the attacker finds a way to break out of the sandbox (buffer overflow, symlink traversal, etc.). Here, the attacker never enters the sandbox. The native tool execution path exists entirely outside the security boundary:

```
Normal command execution:
  Agent → Shell command → Secure Mode evaluation → Sandbox → Execution

Native tool execution (vulnerable path):
  Agent → Native tool invocation → Execution (Secure Mode never evaluated)
```

The security boundary was designed to intercept shell commands. Native tools were architecturally classified as trusted internal components that did not require security evaluation. The vulnerability demonstrates that this trust classification was incorrect -- any tool that accepts external-influenced parameters and can affect system state must be within the security boundary, regardless of whether it is "native" or "external."

## Root Cause Analysis

**Proximate cause:** The `find_by_name` tool passed unsanitized user-influenced input to `fd`, which interpreted it as command-line flags enabling arbitrary code execution.

**Why 1:** Why was the input unsanitized? Because the `find_by_name` tool performed no validation on the Pattern parameter -- it accepted arbitrary strings including those beginning with `-` and passed them directly to `fd` without a `--` argument terminator.

**Why 2:** Why was there no input validation on a tool that interfaces with a system utility? Because native tools were developed as convenience wrappers around existing utilities, with the implicit assumption that the agent would only supply well-formed search patterns. The threat model did not account for an agent whose instructions had been hijacked via prompt injection.

**Why 3:** Why did the threat model not account for prompt injection influencing tool parameters? Because the security architecture treated the agent as a trusted intermediary -- if the agent invokes a native tool, the invocation was assumed to be legitimate. The possibility that an agent's behavior could be manipulated by untrusted content in its context was not incorporated into the native tool security model.

**Why 4:** Why were native tools excluded from the security boundary? Because Antigravity's security architecture drew the trust boundary at shell command execution. Native tools were classified as internal platform components -- trusted by definition -- and placed outside Secure Mode's evaluation scope. The architecture prioritized agent capability and performance over defense-in-depth for built-in tools.

**Why 5 / Root cause:** Antigravity's security architecture applied a pre-AI trust model to an agentic system. Traditional IDEs trust their own internal tools because tool invocations originate from the user or from deterministic code. In an agentic IDE, tool invocations originate from an LLM whose behavior can be influenced by untrusted external content. The fundamental error was granting native tools pre-evaluation execution authority without recognizing that, in an agentic context, "native" does not mean "trusted" -- the agent's tool calls are only as trustworthy as the content that shaped them.

**Root cause summary:** Antigravity's security model treated native tools as inherently trusted, placing them outside Secure Mode's security boundary. In an agentic system where tool invocations are influenced by potentially hostile external content, every tool parameter that reaches a system command is an injection point -- regardless of whether the tool is "native" or "external." The two missing safeguards (input validation and `--` argument termination) are symptoms of this architectural trust model failure.

## How It Could Have Been Prevented

1. **Argument termination before user-supplied values.** Prepending `--` before the Pattern parameter in the `find_by_name` tool's invocation of `fd` would have prevented flag injection entirely. This is a single-line fix and the most direct prevention measure. Every system utility invocation that includes user-influenced parameters must use `--` to separate flags from operands.

2. **Input validation on tool parameters.** The Pattern parameter should reject strings beginning with `-` or containing characters that are not valid in file search patterns. A strict allowlist of permitted characters (alphanumeric, glob wildcards, path separators) would prevent injection of arbitrary flags.

3. **Native tools within the security boundary.** Secure Mode's evaluation scope should include all tool invocations, not just shell commands. In an agentic system, the agent's tool calls originate from LLM output influenced by potentially untrusted content -- they cannot be assumed trustworthy based on their origin.

4. **Content sanitization for prompt injection.** Files ingested from untrusted sources (public repositories, external content) should be sanitized or classified before being included in the agent's context. While prompt injection defense is an unsolved problem, basic mitigations (stripping instruction-like patterns from code comments, separation of code content from agent instructions) would raise the attack bar.

5. **Principle of least privilege for native tools.** The `find_by_name` tool should not have access to `fd` flags that enable command execution (`-X`, `-x`, `--exec`, `--exec-batch`). If the tool's purpose is file searching, its interface with `fd` should restrict the available functionality to search operations only.

**Actual remediation taken:**
- Google accepted the VRP report on January 24, 2026
- Google deployed a fix by February 28, 2026 (53 days from report, 35 days from acceptance)
- Google awarded a bug bounty on March 26, 2026 (amount undisclosed)
- Specific technical remediation details were not publicly disclosed
- No CVE was assigned

**Recommended additional fixes:**
1. **Audit all native tool parameters** across the Antigravity platform for similar injection vulnerabilities -- the `find_by_name` tool is unlikely to be the only native tool that passes parameters to system utilities
2. **Extend Secure Mode's security boundary** to evaluate native tool invocations, not just shell commands -- ensure the sandbox sees all operations that can affect system state
3. **Implement parameter sandboxing** for all tool-to-system-utility interfaces, with strict allowlists of permitted flags and argument patterns
4. **Add prompt injection detection** at the agent's input layer, specifically targeting instruction patterns embedded in source code files and repository content
5. **Publish security advisory with affected versions** -- the absence of version information and a CVE leaves operators unable to verify they are running patched versions

## Solutions Analysis

### 1. Input Validation and Argument Termination
- **Type:** Input Validation and Sanitization
- **Plausibility:** 5/5 -- Directly prevents the specific attack. Adding `--` before user-supplied values and validating that the Pattern parameter does not begin with `-` are well-understood, deterministic mitigations with no false-positive risk.
- **Practicality:** 5/5 -- This is a minimal code change in the `find_by_name` tool's invocation of `fd`. No architectural changes required. Can be applied to all native tools as a systematic audit. This is almost certainly what Google's fix included.
- **How it applies:** Prevents Step 3 (flag injection) of the attack chain entirely. The agent can still be prompt-injected and can still create files, but cannot weaponize `find_by_name` to execute them.
- **Limitations:** Only fixes this specific tool and this specific injection vector. Other native tools with similar patterns remain vulnerable until individually audited and patched. Does not address the broader architectural issue of native tools executing outside the security boundary.

### 2. Security Boundary Extension (Native Tool Inclusion)
- **Type:** Sandboxing and Isolation
- **Plausibility:** 5/5 -- Including native tool invocations within Secure Mode's evaluation scope would ensure that all agent-initiated operations are subject to security policy, regardless of whether they are classified as "native" or "external." This addresses the architectural root cause.
- **Practicality:** 3/5 -- Requires re-engineering the agent runtime to route native tool calls through the Secure Mode evaluation layer. May introduce latency for tool invocations. Requires careful policy definition to avoid breaking legitimate agent workflows. Significant but bounded engineering effort.
- **How it applies:** Even if the input validation fix is bypassed (e.g., a novel injection vector), Secure Mode would evaluate the `find_by_name` call with its injected flags and could block the execution based on policy (e.g., "file search tools should not execute binaries").
- **Limitations:** Security policy must be comprehensive enough to detect malicious tool usage patterns. Over-restrictive policies may degrade agent capability. Does not prevent prompt injection itself -- only limits the damage from successful injection.

### 3. Prompt Injection Mitigation at the Input Layer
- **Type:** Input Validation and Sanitization
- **Plausibility:** 2/5 -- Preventing prompt injection in source code comments and repository content is an unsolved problem. The boundary between legitimate code comments and adversarial instructions is semantic. Current detection methods have high false-positive/negative rates.
- **Practicality:** 2/5 -- Requires deploying prompt injection classifiers on all content the agent ingests. Must process content at scale (entire repositories). Developers may object to content modification that alters code comments. No production-grade solution exists as of May 2026.
- **How it applies:** Would prevent Step 1 (prompt injection delivery) of the attack chain. If the agent does not accept the malicious instructions, it never creates the shell script or calls `find_by_name` with injected flags.
- **Limitations:** Fundamental unsolved problem in AI security. Any classifier will face adversarial evasion. Cannot be relied upon as a primary defense -- must be combined with downstream controls.

### 4. Least-Privilege Tool Interfaces
- **Type:** Permission Scoping / Least Privilege
- **Plausibility:** 4/5 -- Restricting the `find_by_name` tool's interface with `fd` to search-only functionality (no `-X`, `-x`, `--exec` flags) would eliminate the execution capability entirely. The tool only needs to find files, not execute commands against them.
- **Practicality:** 4/5 -- Requires defining a restricted `fd` invocation that strips or blocks execution-related flags. Straightforward for `find_by_name` specifically; more complex when applied systematically to all native tools. Some native tools may legitimately require broad system utility access.
- **How it applies:** Even if flag injection succeeds, the restricted interface would not support the `-X` flag, so `fd` would return an error instead of executing `sh` against matched files.
- **Limitations:** Only prevents exploitation via the `fd` execution flags. An attacker might find other flag combinations with dangerous side effects. Requires per-tool security analysis to define appropriate privilege restrictions.

### 5. Monitoring and Detection (Agent Behavioral Anomaly Detection)
- **Type:** Monitoring and Detection
- **Plausibility:** 3/5 -- Monitoring agent behavior for anomalous patterns -- such as creating shell scripts followed by file search operations with unusual parameters -- could detect this attack chain in progress.
- **Practicality:** 3/5 -- Requires instrumenting the agent runtime to log tool invocations with parameters, and defining baseline "normal" patterns for file search operations. The three-step attack chain (create file, search with exec flag, execute) has a distinctive behavioral signature that could be detected. However, response time determines whether the RCE is prevented or merely logged post-exploitation.
- **How it applies:** Would detect the combination of file creation + `find_by_name` with `-X` flag as anomalous. Alert or block in real-time could prevent Step 3 from completing.
- **Limitations:** Detection is reactive, not preventive. Sophisticated attackers may distribute the three steps across time to evade temporal correlation. Does not address root cause.

The CSA comparative analysis documents six distinct agentic IDE vulnerabilities across Antigravity, Cursor, and Copilot, all sharing the structural characteristic that "agentic tools have been brought to market with capability profiles that substantially exceed prior IDE plugins, while security architectures appear to have adapted existing approaches."

## Strategic Council Review

### Phase 1 -- Challenger

1. **Severity rating of "Medium" may be inconsistent with business criticality and damage potential ratings.** The report assigns `business_criticality: "high"` and `potential_damage: "Critical"`, yet rates severity as "Medium." The OECD independently rated it Critical. The justification -- "no confirmed wild exploitation" -- assumes absence of evidence equals evidence of absence. A flag injection attack on developer workstations would leave minimal forensic traces unless the victim had endpoint detection specifically monitoring for anomalous `fd` invocations. The Medium rating may reflect an evidence gap rather than actual safety.

2. **The "53 days minimum" vulnerability window is significantly understated.** The report says the window was "at least Jan 6 - Feb 28, 2026." But the vulnerability was introduced when the `find_by_name` tool was originally implemented -- potentially at Antigravity's launch. The true exposure window could be months or years longer. The report does not attempt to estimate when the tool was introduced or when Antigravity became generally available.

3. **The `damage_speed: "instantaneous"` rating is theoretical, not observed.** The YAML presents this as fact while the inline comment acknowledges it is theoretical. The PoC demonstrates the chain in a controlled environment, but real-world execution might differ -- prompt injection success rates and agent compliance with multi-step injected instructions are unexamined.

4. **The root cause analysis may be too generous in its framing.** The "Why 5" conclusion -- that Antigravity applied a "pre-AI trust model" -- frames the failure as an honest conceptual gap. An alternative root cause: Google shipped Secure Mode without testing it against the attack class it was marketed to prevent. This could be characterized as an inadequate adversarial testing program rather than a novel architectural challenge.

5. **All technical details derive from a single primary source.** The Pillar Security blog is the sole source for the PoC, the Secure Mode bypass mechanism, and the flag injection details. All secondary sources are derivative. Google did not comment publicly. There is no independent reproduction of the attack or independent verification of the Secure Mode bypass mechanism.

6. **The supply chain risk framing assumes unrealistic prompt injection reliability.** The report emphasizes supply chain scenarios (malicious comments triggering exploitation for all developers who clone a repo) but does not examine the real-world reliability of indirect prompt injection achieving a precise three-step instruction sequence against an agent that also has competing legitimate instructions from the user. The supply chain framing implicitly assumes a near-100% success rate.

### Phase 2 -- Steelman

1. **The Medium severity rating is methodologically sound and explicitly justified.** The report states its framework clearly: "severity is assessed on actual impact." It acknowledges the OECD Critical rating and explains the divergence. The `actual_vs_potential: "near-miss"` field, null financial figures, and "0 confirmed" affected users all support Medium. The report does not downplay risk -- it provides a separate `potential_damage: "Critical"` field, maintaining the distinction without conflation. Inflating severity based on potential would be inconsistent across the incident database.

2. **The root cause analysis is independently corroborated and structurally rigorous.** The five-why chain progresses logically from symptom through architecture to fundamental cause. The CSA analysis independently reached the same conclusion ("security architectures appear to have adapted existing approaches rather than developing models accounting for autonomous agent properties"), providing external validation. The architectural framing has greater explanatory power than the "insufficient testing" alternative because it explains *why* the testing gap existed -- native tools were not within the test scope precisely because of the trust model.

3. **The solutions analysis demonstrates unusual analytical calibration.** Each solution is rated on both plausibility and practicality with justified scores. Solution 3 (prompt injection mitigation) is honestly rated 2/5, acknowledging the unsolved nature of the problem. This calibration -- refusing to propose silver bullets -- demonstrates intellectual integrity uncommon in incident reports.

4. **The single-source concern is mitigated by corroborating evidence.** Google's acceptance of the VRP report and bounty payment constitutes implicit validation. The CSA's independent analysis corroborates the architectural findings. The Mindgard "Forced Descent" discovery independently confirms Antigravity has native tool execution gaps. Eight distinct sources are cited, and the derivative nature of the secondary sources does not invalidate the primary source's findings -- it indicates the security community found them credible.

5. **The composition risk insight is the report's strongest analytical contribution.** The observation that two permitted operations compose into an unpermitted outcome identifies a general vulnerability class applicable to all tool-using agents. The table format makes this immediately actionable for agent platform developers beyond Google.

### Phase 3 -- Synthesis

The challenger raised six points; three are substantive and require acknowledgment, while three are adequately addressed by the steelman or by the report itself.

The strongest challenger points are: (1) the true vulnerability window is almost certainly much longer than the stated 53-day minimum, and the report should acknowledge this more prominently; (2) the single-source nature of the technical details is a legitimate epistemic limitation that deserves explicit disclosure; and (3) the supply chain scenario's assumed prompt injection reliability is unexamined. However, the steelman successfully defends the severity rating methodology (actual vs. potential is consistently applied and clearly justified), the root cause analysis depth (independently corroborated by CSA), and the solutions analysis calibration. The challenger's reframing of the root cause as "insufficient adversarial testing" rather than "architectural trust model failure" is a valid alternative interpretation but not necessarily superior -- both explanations can coexist, and the report's architectural framing has greater explanatory power for the Secure Mode bypass specifically, because it explains why the testing gap existed in the first place.

The report's overall analytical quality is strong. It maintains clear separation between confirmed facts and theoretical scenarios, provides unusually rigorous solution ratings, and identifies a generalizable vulnerability pattern (composition of permitted operations) that has value beyond this specific incident. The main weaknesses are the uncritical reliance on a single technical source and the understated vulnerability exposure window.

**Confidence Level: Medium-High.** The technical analysis is well-structured and the conclusions are reasonable, but the single-source technical details and the unexamined prompt injection reliability in real-world conditions introduce uncertainty that prevents a High confidence rating.

**Unresolved Uncertainties:**
- **True vulnerability window duration:** When was `find_by_name` introduced? Resolvable by examining Antigravity release history or changelog.
- **Whether exploitation occurred before discovery:** Resolvable only by Google's internal telemetry, which is not publicly available.
- **Real-world prompt injection success rate for the three-step chain:** Resolvable by adversarial testing against current/patched agentic IDEs or similar systems.
- **Specific technical details of Google's fix:** Resolvable by Google publishing a security advisory or by reverse-engineering the patched tool invocation.
- **Google's internal severity and scope assessment:** Resolvable only if Google chooses to comment publicly.

## Key Takeaways

1. **Two permitted operations can compose into an unpermitted outcome.** The attack combines file creation (permitted) and file search (permitted) to achieve code execution (unpermitted). Neither operation is individually dangerous. Security models that evaluate tools in isolation will miss composition risks. In agentic systems, the security boundary must account for sequences of tool invocations, not just individual calls.

2. **Security features that do not cover the actual attack surface provide false assurance.** Antigravity's Secure Mode -- with VM isolation, network throttling, and sandboxed execution -- protected against shell commands but not native tool invocations. Users who enabled Secure Mode believing it protected against RCE received no benefit against this attack. Security features must cover the full execution surface, or they must clearly communicate their scope limitations to users.

3. **Every native tool parameter that reaches a system command is an injection point.** Dan Lisichkin's observation applies to all agentic systems: when an AI agent can influence parameters passed to system utilities, every parameter is a potential injection vector. The standard defense -- `--` argument termination before user-supplied values -- must be applied systematically to all tool-to-utility interfaces.

4. **Agentic IDEs are a new privileged software class requiring new security models.** Traditional IDEs trust their internal tools because tool invocations originate from deterministic code or direct user action. Agentic IDEs must not extend this trust to LLM-originated invocations because the LLM's behavior is influenced by external content. As the CSA analysis concludes: security architectures for agentic tools must account for autonomous agent properties, not adapt existing models designed for human-driven workflows.

5. **Repository content is an indirect prompt injection delivery vector for developer tools.** Malicious comments in source code files from public repositories can trigger exploitation without any user interaction beyond normal development workflow. This creates a supply chain attack surface: a single adversarial contribution to a popular open-source project could compromise every developer who processes it with a vulnerable agentic IDE.
