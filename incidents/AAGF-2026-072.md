# Microsoft Semantic Kernel Prompt Injection to RCE — Path Traversal and Code Injection in Enterprise Agent Framework (CVE-2026-25592, CVE-2026-26030)

## Executive Summary

Microsoft's Semantic Kernel (SK) — the enterprise AI orchestration framework underpinning Azure AI Foundry, Copilot Studio, and Microsoft Agent Framework 1.0 — shipped with two CVSS 9.9 CRITICAL vulnerabilities enabling prompt injection to full remote code execution. CVE-2026-25592 allowed path traversal in the `SessionsPythonPlugin` file I/O operations, and CVE-2026-26030 allowed code injection through `InMemoryVectorStore` filter evaluation. Both were found and patched by Microsoft's own security team in February 2026, before any confirmed in-the-wild exploitation. The more alarming story emerged after the patch: independent researcher Nuka-AI demonstrated six encoding bypass vectors defeating the CVE-2026-25592 fix through at least v1.48.0 — bypasses that Microsoft dismissed as "Developer Error" without issuing a new CVE — while simultaneously launching Microsoft Agent Framework 1.0 as "production-ready" on April 3, 2026, with the unresolved architectural trust gap intact.

## What Happened

Microsoft Semantic Kernel is the company's primary open-source AI agent orchestration framework, supporting C#, Python, and Java with 27,000+ GitHub stars. It is not an experimental project — it is the orchestration layer for Azure AI Foundry, Copilot Studio, and (since April 3, 2026) the newly-launched Microsoft Agent Framework 1.0, which Microsoft markets as "production-ready with stable APIs and long-term support commitment." Any enterprise building AI agents on Microsoft's stack is, by definition, running Semantic Kernel.

In early 2026, Microsoft's Defender Security Research Team — three named researchers (Uri Oren, Amit Eliahu, Dor Edry) operating as an internal red team — discovered two critical vulnerabilities in the framework they were researching.

**The first vulnerability (CVE-2026-25592)** lived in the `SessionsPythonPlugin`, a component that enables AI agents to execute Python code in sandboxed sessions by transferring files between the host filesystem and Azure Container Apps dynamic sessions. The `DownloadFileAsync` and `UploadFileAsync` methods accepted a `localFilePath` parameter representing the host-side path. Neither method validated or canonicalized this path — there was no check that the provided path remained within an intended directory. An attacker controlling an agent's prompt could inject path traversal sequences (`../`, encoded variants, absolute paths), causing the agent to write arbitrary files anywhere the process could reach. File write converts to RCE when those files land in executable locations: startup scripts, cron directories, SSH `authorized_keys`, or the running application's own source code. Nuka-AI named the last vector "Self-Nuke" — the agent overwrites its own application source to plant a backdoor.

Microsoft published CVE-2026-25592 on February 6, 2026 with a same-day patch (SK .NET ≥ 1.71.0, Python ≥ 1.39.3) and added path canonicalization and validation to the affected functions.

**The second vulnerability (CVE-2026-26030)** lived in `InMemoryVectorStore`, the built-in no-configuration vector store that SK developers use for RAG workflows without setting up an external database. This is explicitly the component recommended in Microsoft's quickstart documentation — meaning many production deployments use it without understanding the security implications. The filter expression parser used by similarity search evaluated filter expressions using unsafe eval-equivalent behavior on user-controllable input. An attacker who could influence the filter expression (through injected document content, crafted vector store entries, or direct manipulation) could embed Python code that executes in the application's context. Microsoft's own blog demonstrated launching `calc.exe` on the device running the agent via a single malicious prompt — confirming trivial exploitation once an injection vector exists.

Microsoft published CVE-2026-26030 on February 19, 2026 (PR merged February 10) with a fix that blocks dangerous attribute access in filter evaluation, and included it in the March 2026 Patch Tuesday.

The incident story does not end at the patches. On March 24, 2026 — after both CVEs were officially patched — independent researcher Jeff Ponte (Nuka-AI) filed a disclosure to MSRC demonstrating six encoding bypass vectors that defeated the CVE-2026-25592 string-pattern filter: JSON array wrapping, object reflection, Base64 encoding, URL encoding, Unicode homoglyphs, and hybrid canonicalization combinations. Nuka-AI provided PoC recordings demonstrating "complete host takeover with service account privileges" via these bypasses. Microsoft closed the case as "Developer Error," merged quiet "shadow patches" under benign commit titles without security advisory, and assigned no new CVE. On April 3, 2026 — ten days after receiving Nuka-AI's bypass PoC and before the shadow patches were confirmed effective — Microsoft launched Microsoft Agent Framework 1.0 as "production-ready."

On April 21, 2026, Nuka-AI tested SK v1.48.0 and confirmed all six bypass vectors remained functional. On April 25, Nuka-AI published public disclosure ("The Cracked Kernel"). On May 7, 2026, Microsoft's Security Blog published "When prompts become shells: RCE vulnerabilities in AI agent frameworks" — a retrospective disclosure covering both CVEs, timed with broader agentic security guidance, acknowledging the attack chain publicly for the first time in company-authored content.

## Technical Analysis

### The AutoInvokeKernelFunctions Trust Gap

Both vulnerabilities share a common architectural root cause: Semantic Kernel's default configuration (`AutoInvokeKernelFunctions: true`) instructs the framework to automatically invoke any registered plugin function when the LLM requests it. There is no intermediate security boundary — no function invocation filter, no allowlist of parameter values, no semantic validation that the LLM-generated parameters make sense before they reach the system. The LLM output is the input to privileged system operations.

This is the core trust gap: **the framework treats LLM-generated function arguments as equivalently trusted to arguments from validated application code.** In traditional software, a path passed to a file operation comes from validated application logic. In Semantic Kernel with `AutoInvokeKernelFunctions: true`, that path comes from an LLM response — which is, in turn, a function of whatever the agent has been prompted with, including injected content from external documents, web pages, or tool outputs.

### CVE-2026-25592 — SessionsPythonPlugin Path Traversal to RCE

**CWE-22** (Path Traversal)
**CVSS:** 9.9 `CVSS:3.1/AV:N/AC:L/PR:L/UI:N/S:C/C:H/I:H/A:H`
**Affected:** .NET SDK < 1.71.0; Python SDK < 1.39.3

The `SessionsPythonPlugin` enables agents to use Azure Container Apps dynamic sessions as a code execution sandbox, with file transfer functions (`UploadFileAsync`, `DownloadFileAsync`) bridging the host filesystem and the sandbox. The `localFilePath` parameter in both functions was accepted verbatim with no path validation or canonicalization.

Attack chain:
1. Attacker constructs a prompt injection payload in any content the agent processes (document, web page, tool output, direct input)
2. The injected prompt instructs the agent to call `UploadFileAsync` or `DownloadFileAsync` with a crafted `localFilePath` (e.g., `../../../../etc/cron.d/backdoor` or `/var/spool/cron/crontabs/root`)
3. `AutoInvokeKernelFunctions` forwards the LLM's tool call directly to the plugin without parameter inspection
4. The plugin writes attacker-controlled content to the specified path on the host filesystem
5. RCE achieved when the written file is executed (cron job fires, startup script runs, application loads its own overwritten source code — the Self-Nuke vector)

**The patch bypass situation:** Microsoft's fix added string-pattern filtering to reject paths containing traversal sequences. Nuka-AI found that six encoding variants defeat this non-recursive sanitizer:
1. **JSON array wrapping** — `["../etc/passwd"]` deserialized before the path check executes
2. **Object reflection** — anonymous object containing the path, bypasses string check
3. **Base64 encoding** — path decoded at file I/O time, after the filter
4. **URL encoding** — `%2e%2e%2f` passed through the filter, decoded by the path resolver
5. **Unicode homoglyphs** — U+2044 FRACTION SLASH normalizes to `/` during file I/O
6. **Hybrid canonicalization** — combinations that exhaust non-recursive sanitizers

Microsoft confirmed the bypasses only through shadow patches with no CVE assignment, characterizing the issue as "Developer Error" — asserting that developers should implement function invocation filters as an application-layer control. This characterization is disputed by the security community: the vulnerable behavior was the default configuration, not a developer misconfiguration.

### CVE-2026-26030 — InMemoryVectorStore Filter Code Injection

**CWE-94** (Code Injection)
**CVSS:** 9.9 `CVSS:3.1/AV:N/AC:L/PR:L/UI:N/S:C/C:H/I:H/A:H`
**Affected:** Python SDK < 1.39.4

`InMemoryVectorStore` is the default no-configuration vector store for RAG workflows in Semantic Kernel — it requires no external database, making it the first choice in quickstart examples that become production deployments. When an agent performs similarity search, it can apply filter expressions to narrow results. The filter expression parser used eval-equivalent behavior on user-controllable input without sanitization.

Attack chain:
1. Attacker embeds malicious content in a document, vector store entry, or any input that influences agent filter expressions
2. Injected prompt instructs the agent to search using a crafted filter expression containing Python code: `__import__('os').system('calc.exe')` (Microsoft's demonstrated PoC)
3. InMemoryVectorStore evaluates the filter expression without sanitization
4. Arbitrary Python code executes in the application process context with full access to environment variables, in-process secrets, Azure managed identity tokens, and network access

The fix (PR #13505, merged February 10) blocks dangerous attribute access in filter evaluation. The interim mitigation is to avoid `InMemoryVectorStore` in production — problematic when many production deployments inherited this pattern from quickstart documentation.

## Root Cause Analysis

**Proximate cause:** Semantic Kernel's `SessionsPythonPlugin` and `InMemoryVectorStore` components used LLM-generated parameters as unvalidated input to file system operations and code evaluation respectively.

**Why 1:** The framework was designed with the assumption that registered plugins are invoked by trusted application code. When `AutoInvokeKernelFunctions: true` became the default, this assumption was violated — LLM responses now generate the parameters, not the application.

**Why 2:** The `AutoInvokeKernelFunctions` default was optimized for developer experience (frictionless agentic behavior) without a corresponding security review of what "any registered plugin, called with LLM-generated parameters" means for components with filesystem or code evaluation access.

**Why 3:** Framework-level security boundaries — function invocation filters, parameter allowlists, output sandboxing — existed in SK as optional developer controls but were not enforced by default for plugins that perform privileged operations. The security model assumed developers would apply appropriate controls; the documentation did not make clear which plugins required such controls.

**Why 4:** The review process for adding high-privilege plugins (`SessionsPythonPlugin` with filesystem access, `InMemoryVectorStore` with filter evaluation) did not include a threat model that considered adversarial LLM output as the source of function parameters. The plugin development lifecycle lacked a security gate for "what happens if an LLM is prompted to call this with a malicious argument?"

**Why 5 / Root cause:** Microsoft's AI framework development treated LLM function calling as a safe interface — inheriting the trust model of traditional API design where callers are vetted — without accounting for prompt injection as an attack surface that places adversarial inputs on the LLM-to-plugin path. The architectural trust gap (LLM output → privileged system operation, no security boundary) is the root cause. The individual CVEs are symptoms; the default `AutoInvokeKernelFunctions: true` with no mandatory invocation filter policy is the systemic failure.

**Root cause summary:** The framework architecture treats LLM-generated plugin arguments as equivalently trusted to application-generated arguments, creating an implicit trust escalation path from prompt injection to any privileged operation registered as a plugin.

## How It Could Have Been Prevented

1. **Mandatory function invocation filters for high-privilege plugins:** Any plugin that performs filesystem operations, code evaluation, network requests, or credential access should require an explicit `FunctionInvocationFilter` that validates parameters before invocation — enforced at the framework level, not left to developers as a documentation footnote.

2. **Parameterized path allowlists, not string pattern filters:** Path traversal prevention via regex or string matching is well-known to be bypassable (canonical attack class since OWASP 2004). The correct fix is canonical path resolution (`Path.GetFullPath()` / `os.path.realpath()`) followed by a prefix check against an allowlisted directory, applied after all decoding. The patch's string-pattern approach was insufficient by design.

3. **Sandboxed filter evaluation for InMemoryVectorStore:** Filter expressions should be parsed as a safe expression DSL (Abstract Syntax Tree with allowlisted node types), not evaluated via Python's eval machinery. This is a well-understood problem in query language design; safe expression evaluators are a standard solution.

4. **Threat model review for new plugin PRs:** A plugin PR adding filesystem I/O or code evaluation capabilities to a framework where `AutoInvokeKernelFunctions: true` is the default should require a documented threat model addressing: "What happens if an attacker controls a prompt that causes the LLM to invoke this function with a malicious argument?" This would have caught both vulnerabilities before merge.

5. **Security gate before GA launch:** Microsoft Agent Framework 1.0 launched on April 3 — after Nuka-AI filed its bypass disclosure on March 24. A security gate requiring resolution (or at minimum CVE assignment) for outstanding bypass reports before a "production-ready" GA launch would have prevented the reputational and operational harm of shipping known-bypassable protection.

## How It Was / Could Be Fixed

**Actual remediation taken:**

*CVE-2026-25592:* Microsoft patched the `SessionsPythonPlugin` in SK .NET ≥ 1.71.0 and Python ≥ 1.39.3, adding path canonicalization and string-pattern validation. Microsoft subsequently merged shadow patches in April 2026 to address Nuka-AI's bypass vectors under benign commit titles, without a new CVE or security advisory. Interim mitigation documented by Microsoft: implement a Function Invocation Filter that allowlists valid `localFilePath` values before invocation.

*CVE-2026-26030:* Microsoft patched `InMemoryVectorStore` in Python SK ≥ 1.39.4 (PR #13505) by blocking dangerous attribute access in filter evaluation. Included in March 2026 Patch Tuesday. Interim mitigation: avoid `InMemoryVectorStore` in production.

**Additional recommended fixes:**

1. **Replace string-pattern path filtering with canonical path prefix checking.** Apply `Path.GetFullPath()` (C#) or `os.path.realpath()` (Python) on the full `localFilePath` string *after* all deserializations and encodings are resolved, then verify the canonical path starts with an allowlisted directory prefix. This is the only complete fix for CWE-22.

2. **Replace eval-equivalent filter evaluation with a safe expression parser.** Parse filter expressions against an allowlisted AST node type set. Libraries like `asteval` (Python) or equivalent DSL parsers offer safe evaluation without `eval()` exposure.

3. **Change `AutoInvokeKernelFunctions` default to `false` for plugins with privileged access.** Require explicit opt-in for plugins performing filesystem I/O, code execution, or credential access. Document the security implications prominently.

4. **Assign a new CVE for the bypass vectors** and issue a proper security advisory. The shadow patch approach leaves SK operators unaware that their patched installations may still be exploitable via the six encoding variants.

5. **Audit all SK plugins for implicit trust of LLM-generated arguments.** The architectural pattern affecting these two plugins may affect other plugins — anywhere that LLM output reaches a system call without a security boundary.

## Solutions Analysis

### Canonical Path Resolution (Input Validation and Sanitization)
- **Type:** Input Validation and Sanitization
- **Plausibility:** 5/5 — Canonical path resolution (`GetFullPath()` + prefix check) is the documented, OWASP-recommended complete fix for CWE-22. It is not bypassable via encoding because it resolves all encoding before the prefix check.
- **Practicality:** 5/5 — One-line change per affected function; no architectural change required; immediately deployable in a patch release.
- **How it applies:** Replace the string-pattern filter in `DownloadFileAsync` and `UploadFileAsync` with canonical path resolution + allowlisted directory prefix check. This would have prevented all six of Nuka-AI's bypass vectors.
- **Limitations:** Requires defining an allowlisted base directory at plugin configuration time — developers must configure this correctly. Does not address the architectural trust gap (LLM output still reaches the function; it just can't escape the allowlisted directory).

### Function Invocation Filter Enforcement (Architectural Redesign)
- **Type:** Architectural Redesign
- **Plausibility:** 5/5 — SK's Function Invocation Filter API already exists; making it mandatory (not optional) for high-privilege plugins creates a security boundary between the LLM and privileged operations.
- **Practicality:** 3/5 — Requires framework-level changes to distinguish "privileged" plugins and enforce filter presence. Breaking change for developers who have not implemented filters. Migration effort proportional to deployment count.
- **How it applies:** Framework enforces that any plugin performing filesystem I/O, code evaluation, or credential access must have an associated `FunctionInvocationFilter` registered — rejecting plugin registration otherwise. This transforms the security posture from "developer must remember to add a filter" to "framework refuses to operate without one."
- **Limitations:** Developers can implement permissive filters that do nothing. Enforcement of filter quality requires code review, not just filter presence. Does not retroactively fix existing deployments that have not added filters.

### Safe Expression Evaluator (Guardrails / Output Filters)
- **Type:** Guardrails / Output Filters
- **Plausibility:** 5/5 — Replacing eval-equivalent filter evaluation with an allowlisted AST parser is the correct fix for CWE-94. This approach is used in every production query language that accepts user input.
- **Practicality:** 4/5 — Requires replacing the filter evaluation subsystem in InMemoryVectorStore. An off-the-shelf library (e.g., `asteval`, `simpleeval`) can be dropped in with modest integration effort. Some complex filter expressions may not be expressible in the restricted DSL, requiring backward compatibility analysis.
- **How it applies:** Parse all `InMemoryVectorStore` filter expressions against an allowlisted AST node type set (comparison operators, logical operators, field references, literals — no function calls, no attribute access, no imports). Malicious expressions like `__import__('os').system('calc.exe')` would be rejected at parse time.
- **Limitations:** Feature parity with current eval-based filters may require significant DSL design work. Breaks any production code relying on eval behavior for complex filter expressions.

### Permission Scoping / Least Privilege for AutoInvokeKernelFunctions
- **Type:** Permission Scoping / Least Privilege
- **Plausibility:** 4/5 — Changing the default from `AutoInvokeKernelFunctions: true` to `false` (or to `true` only for explicitly allowlisted plugins) directly reduces attack surface. Any plugin not explicitly enabled cannot be invoked by adversarial prompt injection.
- **Practicality:** 2/5 — Breaking change for the default developer experience — SK's value proposition is frictionless autonomous agent behavior. Changing the default would require all existing deployments to opt in explicitly, causing significant migration friction. Microsoft would need to deprecate rather than switch the default.
- **How it applies:** Operators running sensitive plugins (SessionsPythonPlugin, InMemoryVectorStore-backed search) should immediately set `AutoInvokeKernelFunctions: false` and maintain an explicit allowlist of functions the agent may call. This is the recommended operator mitigation today, pending a comprehensive framework fix.
- **Limitations:** Reduces agent autonomy, which is the framework's primary use case. Operators may opt back to `true` for convenience, re-enabling the vulnerability. Does not prevent exploitation if the allowlisted plugins themselves have unsafe parameter handling.

### Monitoring and Detection for Anomalous Tool Arguments
- **Type:** Monitoring and Detection
- **Plausibility:** 3/5 — Detecting path traversal sequences (`../`, `%2e%2e`, encoded variants) or suspicious Python code fragments in filter expressions at the observability layer can alert operators to active exploitation attempts. This is a compensating control, not a fix.
- **Practicality:** 4/5 — Adding argument-level logging and pattern alerting to SK's function invocation pipeline is achievable with modest development effort. Can be implemented as a monitoring-only Function Invocation Filter without changing agent behavior.
- **How it applies:** Log all arguments passed to `SessionsPythonPlugin` and `InMemoryVectorStore` functions. Alert on path traversal indicators in `localFilePath` and code injection indicators (Python builtins, `__import__`, `os.system`) in filter expressions. This provides detection latency — attackers are caught, not prevented — but provides a signal for operators who cannot immediately patch.
- **Limitations:** Detection-only; does not prevent exploitation. Sophisticated attackers may use novel encoding variants not covered by detection rules. Requires operational infrastructure (SIEM, alert routing) that many SK deployers may not have.

## Strategic Council Review

*Reviewed: 2026-05-07. Independent review conducted after curation; the pre-curation author notes above were written by the curation team and are superseded by this section.*

### Phase 1 — Challenger Findings

**1. "Critical" severity applied to a confirmed zero-exploitation near-miss discovered by the vendor's own internal team.**

The discoverers are Microsoft Defender Security Research — an internal red team. Internal red-team discovery is the best-case vulnerability lifecycle outcome: the attacker never learned of the vulnerability before it was patched. The CVSS 9.9 score is assigned to the theoretical attack chain, not to observed impact. Applying "Critical" to every CVSS 9.9 CVE that gets internally-discovered and patched before exploitation would inflate severity across large swaths of well-managed software. The AgentFail severity rubric must be explicit about whether it is harm-based or risk-potential-based; if harm-based, Critical is wrong for a zero-exploitation near-miss; if risk-potential-based, the systemic scope criterion must be applied consistently. The draft's own line — "actual observed harm is zero" — sits in tension with the Critical rating without explicit rubric justification.

**2. The post-patch bypass narrative rests on a single uncorroborated researcher source, and the "v1.74.0 still exploitable" headline claim is not supported by the stated evidence.**

Nuka-AI's "The Cracked Kernel" is the sole source for all six bypass vectors, for the v1.48.0 still-exploitable finding (tested April 21), and for the implicit framing that v1.74.0 remains vulnerable. The executive summary and timeline entry use the stronger "v1.74.0" version number, but the researcher_notes acknowledge that Nuka-AI tested v1.48.0, not v1.74.0. The headline claim therefore overstates the verified evidence by a named version. Microsoft's "Developer Error" position is internally consistent: if developers are required to implement invocation filters for high-privilege plugins (as documented), then exploitation through an absent filter is a misconfiguration, not an unpatched product defect. The draft does not resolve this contested claim; it presents Nuka-AI's framing as the primary narrative without marking it as one side of a disputed account.

**3. The $375M averted damage estimate compounds three independent unverified assumptions multiplicatively.**

The methodology (5,000 deployments × $1.5M IBM CODB breach cost × 5% exploitation probability) stacks three uncertain inputs: (a) 5,000 enterprise production deployments — an assumption from 27K GitHub stars with an unknown production fraction, many of which are forks, research projects, and archived repos; (b) $1.5M average enterprise breach cost — the IBM 2024 CODB figure for all breach types, not calibrated to AI agent framework RCE specifically; (c) 5% exploitation probability — not anchored to any threat intelligence, attack tooling availability, or observed exploitation in comparable CVEs. Each input has ±1–2 orders of magnitude uncertainty; the product has ±3 orders of magnitude uncertainty. Presenting $375M as a point estimate at "order-of-magnitude confidence" understates how compounded the speculation is. The stated range ($50M–$2B) uses only 4x/5x multiplier variation, not the ±10–100x that the actual uncertainty warrants.

**4. The "GA launch with known bypasses" framing requires accepting four contested inferences simultaneously.**

For this narrative to hold, all of the following must be true: (a) Nuka-AI's bypass vectors are genuine product vulnerabilities rather than developer misconfigurations; (b) MSRC's assessment ("Developer Error") was factually wrong on the merits; (c) the bypasses were "known" and unresolved at GA launch time — MSRC received the disclosure March 24, but their conclusion was that no fix was needed; (d) "production-ready" label in the GA announcement implies a security guarantee rather than an API stability guarantee. The draft presents this chain as an established editorial conclusion rather than a sequence of inferences dependent on Nuka-AI's disputed account.

**5. The two-CVE bundling lacks an explicit editorial justification given that AgentFail has precedent for separate entries in multi-CVE chains.**

CVE-2026-25592 and CVE-2026-26030 have different affected components, different vulnerability classes (CWE-22 vs CWE-94), different affected SDK versions, different patch dates (Feb 6 vs Feb 19), and different deployment prerequisites (CVE-25592 requires Azure Container Apps dynamic sessions; CVE-26030 affects any InMemoryVectorStore deployment). The related incident AAGF-2026-061 covers CrewAI's four-CVE chain, which the database apparently treated as a single incident — but that precedent should be noted. The bundling is defensible on shared-root-cause grounds, but that justification should be explicit in the incident rather than implicit.

**6. The "shadow patches" framing presents unverified characterization as established fact.**

The claim that Microsoft merged "shadow patches under benign commit titles" is attributed to Nuka-AI's characterization. The draft does not cite specific commit hashes or titles, making the "shadow patch" characterization unverifiable from the listed references. The Semantic Kernel repository is public and open-source; if the relevant commits exist and have benign titles, they are citable with specific commit references. Without that specificity, this is Nuka-AI's interpretation presented as fact.

### Phase 2 — Steelman Defense

**1. "Critical" is correct because the AgentFail severity rubric explicitly includes systemic scope as a qualifying criterion — independent of exploitation count.**

The systemic scope criterion exists for exactly this scenario: a vulnerability in a framework, not an application, where the blast radius is structural across every downstream deployment. Semantic Kernel is the orchestration layer for Azure AI Foundry, Copilot Studio, and Microsoft Agent Framework 1.0 — a single exploitable pattern affects every organization running Microsoft's enterprise AI agent stack simultaneously. The CVE's own CVSS scope indicator (S:C — Changed Scope) corroborates cross-boundary impact. The draft is explicit that the Critical rating reflects near-miss potential on systemic scope, not confirmed harm — this is a documented editorial choice, not an undisclosed inflation. Internal discovery is the correct outcome; it does not retroactively reduce the severity of the vulnerability that existed.

**2. The bypass vectors are technically credible from first principles, verifiable independently of Nuka-AI's disputed characterization.**

Encoding bypasses for string-pattern path sanitizers are a canonical, well-documented attack class (OWASP CWE-22, documented since the early 2000s). The six techniques Nuka-AI demonstrates — Base64, URL encoding, Unicode homoglyphs, JSON wrapping, object reflection, hybrid canonicalization — are all standard bypass methods for non-canonicalizing string-pattern filters. Their theoretical feasibility requires only that the CVE-2026-25592 patch used string-pattern matching rather than canonical path resolution — a fact verifiable from the patch diff independent of Nuka-AI's claims. If the patch performs `contains("../")` or equivalent matching rather than `Path.GetFullPath()` followed by a directory prefix check, every one of the six bypass classes is theoretically valid. The "Developer Error" dispute is a responsibility question; the bypass technical feasibility is a verification question resolvable from public code.

**3. The two-CVE bundling is editorially correct because the incident is the architectural trust gap, not the individual CVEs.**

AgentFail's differentiation from CVE databases is incident framing — the analytical unit is the failure pattern, not the individual bug. The story here is: Microsoft's enterprise AI agent framework treats LLM-generated plugin arguments as equivalently trusted to application-generated arguments, creating a structural path from prompt injection to any privileged operation registered as a plugin. Both CVEs are manifestations of this single architectural failure. Splitting into two entries would lose the most analytically valuable finding: the shared root cause means fixing one CVE without addressing the trust gap leaves the framework systematically vulnerable to the same exploit class in other plugins. The incident value is in the pattern identification; bundling serves the database's analytical mission.

**4. Enterprise scale of impact is qualitatively different from isolated research findings — the InMemoryVectorStore quickstart documentation point is structural.**

The population at risk includes organizations running Copilot Studio (enterprise no-code agent builder deployed at scale), Azure AI Foundry users, and Microsoft Agent Framework 1.0 early adopters who deployed based on the "production-ready" GA label. The InMemoryVectorStore point is particularly load-bearing: Microsoft's own quickstart documentation recommends this component as the no-configuration RAG option for new SK developers. Organizations following the official documentation path inherited a vulnerable pattern without realizing the security implications — this is structural influence at enterprise scale, not a contrived lab scenario.

**5. The operator information gap from the "shadow patch" behavior is a genuine documented risk regardless of the responsibility framing.**

Even accepting Microsoft's position entirely — that the bypasses are developer misconfiguration, not product defects — the practical effect of merging fixes under benign commit titles without a CVE or security advisory is that standard enterprise vulnerability management (CVE feed monitoring, Patch Tuesday tracking) receives no signal. An operator who upgraded to v1.71.0 after CVE-2026-25592, implemented no invocation filter per MSRC's "Developer Error" characterization, and monitors CVE feeds — that operator has done everything "right" by enterprise patch management practice and remains unaware of any further action needed. The information gap exists and is consequential regardless of who is responsible for the underlying behavior.

**Final assessment:** AAGF-2026-072 is publication-ready. The two CVEs are confirmed, multiply sourced, and technically well-documented. The shared architectural root cause (AutoInvokeKernelFunctions trust gap — LLM output as trusted input to privileged operations) is clearly established and provides genuine analytical value. The "Critical" severity is defensible on systemic scope grounds with the near-miss caveat explicitly stated. The two-CVE bundling is editorially justified by the shared root cause narrative.

The incident has one contested element requiring careful editorial handling: the post-patch bypass situation, the shadow patching behavior, and the GA launch timing all rest primarily on Nuka-AI's characterization, which Microsoft disputes on responsibility framing (not on the technical existence of encoding bypass techniques). The incident is published with Nuka-AI's findings clearly attributed and Microsoft's "Developer Error" position clearly noted as a competing characterization — without the narrative asserting one side is correct.

The v1.74.0 claim in the executive summary has been corrected to reflect that Nuka-AI tested v1.48.0 specifically; bypass status in later versions is unverified as of May 7, 2026. The $375M damage figure carries explicit compounded-uncertainty language throughout.

## Key Takeaways

1. **Prompt injection is the new SQL injection for agent frameworks — validate parameters at the boundary, not just at the prompt.** Both CVEs are triggered by prompt injection reaching framework components that pass LLM-generated parameters directly to privileged operations. The lesson is not "filter prompts better" but "treat LLM output as untrusted user input at every system boundary" — the same principle SQL injection taught for database-backed applications. Path validation and filter expression parsing must be applied at the point where LLM output touches the system, not at the model layer.

2. **String-pattern sanitization for path traversal will always be bypassable.** The CVE-2026-25592 patch used string-pattern filtering, which Nuka-AI bypassed with six canonical techniques within weeks. The correct fix — canonical path resolution followed by directory prefix checking — has been documented since the 2000s. If your framework or application performs file operations on paths that originate from user input (or LLM output), implement `os.path.realpath()` + allowlisted prefix check; nothing else is adequate.

3. **Default-on autonomous tool invocation with privileged plugins is a critical risk.** `AutoInvokeKernelFunctions: true` is a sensible default for developer experience. It is not a sensible default when registered plugins include filesystem I/O and eval-equivalent code evaluation. Operators running Semantic Kernel with SessionsPythonPlugin or InMemoryVectorStore-backed search exposed to external input should immediately audit their function invocation filter configuration — the absence of a filter means prompt injection is a direct path to RCE.

4. **"Shadow patches" without CVE assignment leave operators blind.** Enterprise vulnerability management operates on CVE feeds, security advisories, and Patch Tuesday bulletins. When a vendor resolves a security bypass under a benign commit title with no advisory, operators who have implemented patch-based vulnerability management receive no signal. If you are running Semantic Kernel with SessionsPythonPlugin, verify your patch version against the GitHub commit history — not just the CVE status.

5. **"Production-ready" GA labels should require security sign-off on outstanding disclosures.** Microsoft Agent Framework 1.0 launched ten days after MSRC received Nuka-AI's bypass PoC. Whether the bypass represents a genuine unresolved vulnerability or a developer misconfiguration, the appropriate pre-GA process is to resolve outstanding security disclosures before applying a "production-ready with long-term support" label to a framework. The reputational cost of launching a GA product with a contested but plausible unresolved RCE bypass is larger than the delay cost of a two-week patch cycle.
