## Executive Summary

On February 28, 2026, CodeWall's fully autonomous offensive AI agent selected McKinsey as a target, discovered SQL injection via JSON key name concatenation in an unauthenticated API endpoint, ran 15 iterative blind probes to extract the database schema, and achieved full read-write access to McKinsey's Lilli AI platform within approximately two hours — with no human directing any step of the attack. The most analytically significant element is not the data scope (46.5M chat messages, 728K files, 57K accounts) but the 95 writable system prompts controlling Lilli's behavior for 43,000+ McKinsey consultants: a single SQL UPDATE statement, with no trace in any deployment log, could have silently corrupted the epistemic layer guiding all AI-assisted M&A analysis, financial modeling, and client advisory at the firm. CodeWall's responsible disclosure on March 1 prevented any exfiltration or prompt modification; McKinsey patched all unauthenticated endpoints within 24 hours.

## What Happened

### Target Selection: The Autonomous Choice

The incident begins with a step that has no precedent in the AgentFail database: the attacker — an AI agent — chose its own target.

CodeWall's offensive AI agent, operating without human direction, identified McKinsey's public HackerOne responsible disclosure policy. The existence of a responsible disclosure program signals that an organization has a public-facing digital surface worth testing; the agent interpreted this as authorization to probe. Whether this interpretation is consistent with the spirit of McKinsey's HackerOne policy is contested (see Strategic Council Review). What is not contested is that no human told the agent to target McKinsey. The CodeWall CEO, quoted by The Register, described the process as "fully autonomous from researching the target, analyzing, attacking, and reporting."

### Endpoint Discovery

The agent discovered publicly accessible API documentation listing more than 200 endpoints. Of these, 22 required no authentication — a number that The Register corroborated independently. The existence of publicly documented unauthenticated endpoints on a production platform handling M&A advisory data is the architectural baseline failure from which everything else follows.

### The SQL Injection Vector: JSON Key Concatenation

The technically distinctive element of this incident is the specific injection vector discovered.

Standard SQL injection protections — parameterized queries, prepared statements — operate on field values. A query like `SELECT * FROM messages WHERE user_id = ?` passes the user-supplied value as a parameter, preventing injection regardless of what the value contains. McKinsey's Lilli implementation had this protection in place for values.

What it did not protect was JSON key names.

The vulnerable endpoint accepted search queries in JSON format. When the application processed these queries, it extracted the key names from the JSON payload and concatenated them directly into SQL query strings — treating them as structural elements of the query rather than data. An attacker who controls the key name in a payload like `{"malicious_key_name": "safe_value"}` can inject SQL through the key name itself, even though the value is safely parameterized.

This is not a theoretical edge case. It is a real injection point that standard automated scanners miss because those scanners probe values, not key names. The vulnerability sat in a non-standard code path that the organization's existing tooling was not designed to reach.

### The Blind Extraction Progression

The agent submitted crafted JSON payloads with manipulated key names to the unauthenticated search endpoint. The application's error messages reflected injected key names verbatim — the classic signal of a blind SQL injection vulnerability that reveals structure through error behavior.

From this signal, the agent performed 15 iterative blind probes over approximately two hours, progressively extracting the query structure and database schema. Each probe used the error feedback from the previous attempt to narrow the next query. By the end of this progression, the agent had achieved full read-write access to the production Lilli database.

The 15-iteration count is CodeWall-only, not independently verified. The overall timeline — approximately two hours from reconnaissance start to full access — is corroborated by all sources.

### The System Prompt Poisoning Capability That Was Not Executed

The analytically most significant finding is not in the data inventory above. It is in the last row: 95 system prompts.

These prompts controlled Lilli's behavior across 12 model types for a user base of 43,000+ McKinsey employees processing 500,000+ queries per month. They were stored in the same production database that the SQL injection gave write access to.

An attacker could have modified any or all of these prompts via a single UPDATE statement wrapped in a single HTTP request. The change would:
- Take effect immediately for all subsequent Lilli sessions
- Leave no trace in code deployment logs
- Leave no trace in git history or CI/CD audit trails
- Be indistinguishable, from the consultant's perspective, from Lilli's normal behavior

The downstream blast radius of a successful prompt poisoning is difficult to bound. Lilli's outputs connect to McKinsey's proprietary research frameworks and decades of consulting methodology encoded in 3.68 million RAG chunks. Poisoned prompts could alter financial models, strip safety guardrails, embed systematic bias into M&A analysis, or subtly steer strategic recommendations in ways that benefit a sophisticated adversary. Because Lilli's outputs appear under McKinsey's institutional authority, poisoned advice carries enormous trust amplification — a consultant receiving subtly wrong financial modeling guidance from Lilli has no signal that the underlying prompt has been modified.

CodeWall did not execute a prompt modification. No prompts were altered.

### Responsible Disclosure and McKinsey's Response

CodeWall submitted responsible disclosure to McKinsey on March 1, 2026. McKinsey's CISO acknowledged the disclosure and, within approximately 24 hours, patched all unauthenticated endpoints, took the development environment offline, and blocked public API documentation.

Notable omissions: McKinsey did not specifically address the system prompt write vulnerability. It did not disclose whether consultants or clients were notified. It did not publish the third-party forensics findings.

## Technical Analysis

### JSON Key Concatenation: The Specific Failure Mechanism

The vulnerability class — SQL injection via JSON key name concatenation — is worth precise technical characterization because it explains why existing tooling missed it.

In a typical backend implementation processing JSON input, the developer may write code equivalent to:

```python
key = user_json.keys()[0]  # Extract the key name
query = f"SELECT * FROM table WHERE {key} = ?"  # Concatenate key name into SQL
cursor.execute(query, [user_json[key]])  # Pass value as parameter
```

The value is parameterized; the key name is not. Standard SQL injection scanners — OWASP ZAP, Burp Suite's automated scanner, sqlmap — will probe the value position (the `?` parameter) because that is where injection is conventionally possible. They do not typically enumerate JSON structure to discover that key names are being concatenated into the query string rather than treated as literal structural elements.

The error reflection behavior — where the application returned error messages that included the injected key name verbatim — is what enabled blind extraction. Each injected probe caused a database error whose message contained a fragment of the attacker's payload, confirming successful injection and providing schema information to guide the next iteration.

### Unauthenticated Endpoint Surface

Twenty-two unauthenticated production API endpoints is not a misconfiguration. It is an architectural pattern — a decision, made at some point in the platform's development, that certain endpoints were safe to expose without authentication. On a platform handling M&A advisory data for a major management consultancy, this decision represents a fundamental failure to apply defense-in-depth: even endpoints that appear to return only limited data can serve as injection surfaces if they process user input and route it to a database.

The public API documentation that enabled the agent's enumeration is a compounding factor. Documenting unauthenticated endpoints publicly reduces the reconnaissance effort for any attacker — human or automated — to near zero.

### System Prompts in the Application Database: An Architectural Category Failure

The most consequential architectural decision in this incident is not the SQL injection. It is the storage location of system prompts.

System prompts are not application data. They are behavioral configuration for an AI platform — functionally equivalent to code in terms of what they control. Storing them in the same database layer as user data — where they are reachable via a SQL injection that targets user data — treats them as if they are the same kind of asset. They are not.

Code deployment is controlled by access review, change management, CI/CD pipelines, audit logs, and version control history. A SQL UPDATE against a prompt stored in the application database bypasses every one of these controls. An attacker who modifies a system prompt via SQL injection leaves no forensic trace in any of the systems designed to detect unauthorized code changes. The absence of an audit trail is structural — not a gap that can be patched without changing the storage architecture.

### The RAG Chunk Exposure

The 3.68 million RAG document chunks represent McKinsey's proprietary research frameworks and consulting methodology — the intellectual infrastructure that Lilli draws on to generate its responses. Read access to this corpus gives an adversary a detailed map of McKinsey's knowledge base. Write access — which the SQL injection provided — would allow silent injection of disinformation into the RAG layer, poisoning Lilli's retrieved context without modifying any prompt.

Neither the RAG chunk read nor the RAG write capability was exercised. Both were technically accessible.

### Development and Production Sharing the Same Attack Surface

McKinsey's incident response took the development environment offline in addition to patching production endpoints. This detail — present in both CodeWall and The Register reporting — suggests that development and production shared the same vulnerable API surface. If true, this represents an additional architectural gap: a vulnerability that exists in a development environment connected to the same unauthenticated endpoint structure as production doubles the exposure window and complicates forensic attribution.

## Root Cause Analysis

**Five Whys**

**Why did an autonomous agent achieve full read-write access to McKinsey's production Lilli database in two hours?**
Because a production API endpoint accepted user-controlled JSON key names and concatenated them into SQL queries without sanitization — an injection vector that bypassed value-level parameterization and evaded standard scanner coverage.

**Why did a production API endpoint on a platform handling M&A advisory data lack authentication?**
Because endpoint authentication was applied selectively rather than by default — some endpoints were classified as "safe to expose publicly," and neither the classification process nor the ongoing audit caught the injection vulnerability in the unauthenticated search endpoint.

**Why was the SQL injection vector present in the key name path when value parameterization was in place?**
Because the developers who implemented value parameterization did not model the key name as an injection surface. The assumption was that user-controlled values were the attack vector; key names were treated as structural application elements. This is a threat model gap, not a coding error in the parameterized path.

**Why were system prompts stored in the same database layer as user data, reachable via user-facing SQL injection?**
Because system prompts, user data, and authentication state were treated as uniform application concerns sharing a common persistence layer — rather than as security domains with different blast radii, different write authorities, and different audit requirements.

**Root cause: Security domain conflation.** The platform did not distinguish between assets that have different security profiles: system prompts (equivalent to code; should require code-deployment-level authorization to modify), user data (should require authentication to access), and API authentication state (should be mandatory by default, not selective). Treating all of these as uniform application database concerns placed prompt write authority in the same security perimeter as user data read access. The SQL injection is the exploit; the domain conflation is the vulnerability that made the blast radius catastrophic relative to the code path involved.

## Impact Assessment

### What Was Actually Accessed

CodeWall's responsible disclosure asserts that the agent enumerated the database schema and confirmed read-write access. McKinsey's forensic firm found "no evidence that client data or client confidential information were accessed by this researcher or any other unauthorized third party."

These two statements are not necessarily in conflict, but they are not the same claim. Schema enumeration can be accomplished without reading data rows — the blind probe technique extracts structure, not content. CodeWall's confirmation of "full read-write access" means the injection was capable of reading data, not that data was read. Edward Kiledjian's characterization — "having access is not the same as exfiltrating data" — is analytically correct and should frame how the impact is assessed.

### The Forensic Caveat

McKinsey's forensic finding ("no evidence of access") is an organizational claim, not an independent technical publication. The third-party forensics firm's methodology and findings have not been released. The nine-day window between patch (March 2) and public report (March 9) — noted by Kiledjian — may have been insufficient for a complete forensic review at the data volume involved. This does not mean the claim is false; it means the evidence base is limited.

### The System Prompt Poisoning Scenario

The system prompt poisoning near-miss is the analytically defining element of this incident. It is also the element for which "access vs. exfiltration" framing is most relevant: a prompt modification leaves no data exfiltration trace because it is a write operation, not a read operation. A forensic firm looking for evidence of data access would not necessarily be looking for evidence of a database UPDATE against a system prompt table.

This does not mean prompts were modified. There is no evidence they were. But the forensic methodology that would detect a prompt modification is different from the methodology that would detect row-level data reads, and McKinsey's statement addresses the latter, not the former.

### Scale Framing

The near-miss framing is appropriate: no exfiltration was confirmed, no prompts were modified, and responsible disclosure limited actual harm. High severity is appropriate because the vulnerability was real, the access was real, and the potential consequences — at the scale of McKinsey's consultant population and the sensitivity of M&A advisory data — are in a category that does not fit standard breach cost models.

## How It Could Have Been Prevented

**Control 1: Authentication on all endpoints by default**
Unauthenticated API endpoints should require an explicit architectural exception, not an explicit authentication requirement. The default posture for any endpoint on a platform handling sensitive data is: authenticated. The 22 unauthenticated endpoints represent 22 separate decisions (or 22 failures to decide) that violated this posture. Removing public API documentation does not fix the posture; it reduces discoverability only.

**Control 2: Parameterize key names, or allowlist JSON structure**
If an application processes JSON input and uses field identifiers in SQL queries, either (a) all user-controlled identifiers — including key names — must be parameterized, or (b) the application must enforce a strict allowlist of permitted key names before they reach any SQL construction path. Treating key names as structural application elements while treating values as user-controlled is a threat model blind spot. A JSON schema validation layer that rejects any key name not on an explicit allowlist would have blocked this attack before any SQL path was reached.

**Control 3: Separate system prompt storage from the application database**
System prompts should not be stored in a database layer accessible via user-facing SQL injection. Options include: a separate configuration service with independent authentication and audit logging, a version-controlled external storage mechanism with deployment controls equivalent to code release, or at minimum a separate database instance with distinct credentials and write access restricted to deployment pipelines rather than application service accounts. Any architecture that allows a SQL injection in a user-facing API endpoint to issue UPDATE statements against system prompt tables has collapsed the boundary between user data and behavioral configuration.

**Control 4: Mandatory AI security review before production deployment at scale**
A platform serving 43,000+ consultants with access to M&A-grade data should have received a dedicated AI security assessment before reaching that scale. The assessment would have identified unauthenticated endpoints and prompt storage architecture as risk factors before a production breach demonstrated them. The absence of such a review is not unique to McKinsey — it reflects industry-wide underinvestment in pre-deployment AI system security — but the consequences at Lilli's scale make it a notable gap.

## How It Was / Could Be Fixed

**What McKinsey actually did (confirmed):**
- Patched all 22 unauthenticated endpoints within 24 hours of responsible disclosure
- Took the development environment offline
- Blocked public API documentation

**Additional remediation not confirmed but implied by the vulnerability:**
- Review of all database query construction paths for non-parameterized identifiers — not only key names, but column name interpolation, table name construction, and any other structural SQL element that might accept user-controlled input
- Architectural review of system prompt storage: prompts should be moved to a storage layer with write controls independent of the application database and audit logging equivalent to code deployment
- Formal AI security audit of Lilli's full architecture: authentication model, permission boundaries between user data and behavioral configuration, RAG pipeline write access, and audit coverage for prompt-layer changes
- Internal review of responsible disclosure policy scope: whether the HackerOne policy explicitly covers autonomous enumeration of production database contents at the scale demonstrated

## Solutions Analysis

**Solution 1: Endpoint Authentication by Default (Permission Scoping / Least Privilege)**
- What: Enforce authentication as the default posture for all API endpoints; require explicit architectural exception with documented justification for any unauthenticated endpoint; periodic automated audit of endpoint authentication status
- Plausibility: High — standard API gateway configuration; no architectural redesign required
- Practicality: High — retroactively applying authentication to 22 endpoints is straightforward; forward posture enforcement requires governance process change
- Limitation: Authentication prevents unauthorized access; it does not prevent injection from authorized but compromised accounts; defense-in-depth still required

**Solution 2: Input Validation and JSON Key Allowlisting**
- What: Enforce a strict allowlist of permitted JSON key names before any user-controlled key name reaches a SQL construction path; reject requests with non-allowlisted keys at the API boundary
- Plausibility: High — JSON Schema validation is a mature, widely supported standard
- Practicality: High — can be implemented at the API gateway layer without changes to application logic; allowlist is derivable from the application's actual expected key set
- Limitation: Requires maintaining the allowlist as the API evolves; must be applied consistently across all endpoints, not just the identified vulnerable one; does not address the underlying threat model gap for future code additions

**Solution 3: Architectural Redesign — Separate System Prompt Storage**
- What: Move system prompts out of the application database into a separate configuration service with its own authentication, write access restricted to deployment pipelines, and an audit log that records every modification with actor identity, timestamp, and before/after content
- Plausibility: High — the architectural pattern exists and is in common use for secrets management and configuration stores
- Practicality: Medium — requires refactoring Lilli's prompt management workflow; changes the deployment process for prompt updates; introduces a new service dependency; higher implementation cost than the other controls
- Limitation: Does not address the injection vulnerability itself; a platform that correctly stores prompts but leaves SQL injection in user-facing paths is still vulnerable to data exfiltration — this control limits blast radius, not initial access
- Why it matters most: This is the only control that eliminates the system prompt poisoning vector structurally, regardless of whether other injection vulnerabilities exist

**Solution 4: Anomalous SQL Pattern Monitoring**
- What: Deploy database activity monitoring that flags SQL patterns characteristic of injection exploitation — error-generating queries, high-volume schema enumeration, UPDATE statements against configuration tables from application service account credentials, queries with unusual structural complexity in key positions
- Plausibility: Medium — database activity monitoring tools exist; pattern rules for injection-style queries require calibration to avoid false positives
- Practicality: Medium — adds operational overhead; effectiveness depends on rule quality; does not prevent access but enables detection and response during an active attack
- Limitation: Blind SQL injection via error reflection is detectable; some injection techniques generate minimal distinctive traffic; a sophisticated attacker can space probes to avoid rate-based detection; monitoring addresses detection, not prevention

Both incidents involve AI agents conducting offensive security operations. AAGF-2026-060 seeds `autonomous-ai-agent-offensive-use`: the agent selects its own target, conducts reconnaissance, constructs the exploit, and generates the report — the human's role is limited to post-hoc responsible disclosure. The distinction is not degree but kind: AAGF-2026-051 has a human in the operational loop at every decision point; AAGF-2026-060 does not. This is the first incident in the AgentFail database where no human directed any step of the attack chain.

## Strategic Council Review

### Challenger: Where the Narrative May Overreach

**Challenge 1: Source concentration and commercial framing**
Every technical claim in this incident — the 22 unauthenticated endpoints, the 15-iteration probe, the 46.5M message count, the 95 system prompts, the 2-hour timeline, the "fully autonomous" characterization — originates from CodeWall, an offensive AI vendor whose business case is demonstrated by this incident. The Register independently confirmed that an attack occurred and McKinsey patched. The Register did not independently audit data volumes, probe counts, or autonomy claims. This is a single-source technical narrative with a strong commercial interest in maximizing the apparent capability of the tool demonstrated.

**Challenge 2: "Access" is not "exfiltration" — Kiledjian's point deserves full weight**
Edward Kiledjian's observation — that CodeWall's blog "blurs the line between having access and actually exfiltrating data" — is not a quibble. Blind SQL injection schema enumeration can be confirmed without reading a single data row. CodeWall's "full read-write access" claim is a statement about capability, not about what was done. The 46.5M chat messages figure is the count of what was accessible; it does not reflect what was read, copied, or transmitted. For an incident with no confirmed exfiltration, this distinction matters for calibrating actual harm.

**Challenge 3: The HackerOne responsible disclosure scope question**
McKinsey's HackerOne policy presumably covers discovery and reporting of security vulnerabilities. Whether it covers autonomous enumeration of production database schema, confirmation of read-write access to live M&A advisory data, and generation of a detailed product demonstration report — all without explicit prior authorization for that scope — is not established. Kiledjian raised this question: did CodeWall's activities fall within the spirit of responsible disclosure, or were they at the boundary of what the policy was designed to authorize? This does not invalidate the technical findings but is relevant to how the incident should be characterized.

**Challenge 4: McKinsey's forensic finding is an organizational claim, not a published technical report**
The forensic firm's conclusion ("no evidence of access") has not been released. The nine-day window between patch and public report (March 2 to March 9) is a compressed timeframe for comprehensive forensic review at the data volumes involved. This does not mean the claim is false — absence of evidence is not evidence of access — but the evidence base is limited. The forensic methodology that would detect a prompt-layer UPDATE is also not the same as the methodology that would detect user data reads, and McKinsey's statement addresses the latter.

**Challenge 5: "Autonomous" may be a marketing characterization**
The "fully autonomous from researching the target, analyzing, attacking, and reporting" claim comes from the CodeWall CEO, quoted by The Register. Researchers monitored the session. The boundary between "monitored but not directed" and "operationally autonomous" is contested in the security research community. The agent's behavior may have been genuinely autonomous in the operational sense; it may also reflect a tool that requires human setup, targeting parameter specification, or other non-trivial human input that "fully autonomous" elides.

**Challenge 6: The two-year vulnerability duration is an inference, not a confirmed fact**
NeuralTrust's claim that the platform had been vulnerable for over two years is not independently verified. It may reflect Lilli's general rollout timeline rather than the specific vulnerability's introduction date. The actual duration of the SQL injection vulnerability in production is unknown.

### Steelman: Why High Severity Is Correct Despite Source Limitations

**Defense 1: McKinsey's same-day patch is the strongest corroboration available**
A management consultancy's CISO does not patch all unauthenticated endpoints, take development offline, and block public API documentation within 24 hours of receiving a disclosure unless the disclosure describes a real vulnerability. The patch response is functionally an admission that the attack surface was genuine. This is more meaningful corroboration than an independent audit would be for many incidents, because it reflects an internal assessment of the severity by the party with the most complete information.

**Defense 2: The JSON key concatenation vector evading standard scanners is technically credible**
The mechanism is technically sound — Kiledjian himself assessed it as such. Non-parameterized key name concatenation is a plausible implementation error; standard scanners probing values would not catch it; error reflection enabling blind schema extraction is a well-documented exploitation technique. Nothing in the technical description requires unusual capabilities or raises credibility flags.

**Defense 3: The system prompt poisoning near-miss is the most consequential element and is not disputed**
CodeWall's claim that 95 system prompts controlling Lilli's output were writable via SQL UPDATE, with no audit trail equivalent to code deployment, has not been disputed by McKinsey or any independent source. This is the element that makes the incident analytically significant beyond the data scope. Kiledjian's "access vs. exfiltration" framing is important for calibrating actual data harm; it does not diminish the system prompt poisoning near-miss, which requires no exfiltration — only a write operation.

**Defense 4: The autonomous agent characterization is supported by The Register's direct CEO quote**
The Register's independent reporting includes a direct quotation from the CodeWall CEO confirming the autonomous framing. While the marketing interest is real, the agent's behavior as described (target selection, endpoint enumeration, exploit construction, reporting) is consistent with the capabilities of current autonomous offensive AI agent frameworks. The qualitative claim that no human directed the operational steps is not implausible and is independently corroborated by The Register's reporting.

### Synthesis

High severity is the correct classification. The system prompt poisoning near-miss — 95 behavioral configuration objects controlling an AI platform used by 43,000+ consultants on M&A-grade engagements, writable via a SQL UPDATE with no deployment audit trail — is a genuine architectural failure of a consequence class that does not fit standard breach cost frameworks.

The data volume figures (46.5M messages, 728K files) should be noted as CodeWall-only and not presented as audited fact. The "access vs. exfiltration" distinction should be preserved in the impact framing: confirmed read-write access capability is real; confirmed data exfiltration is not established.

The HackerOne scope question and the autonomy characterization are legitimate uncertainties that reduce confidence in the narrative's finer details without undermining the core technical findings. Confidence for this incident: **MEDIUM** — the vulnerability is real, the access was real, the near-miss category is appropriate, but the data scope and autonomy claims rest on a single commercially interested source with limited independent verification.

## Key Takeaways

**1. Never store system prompts in the same writable database layer as user data.**
System prompts are behavioral configuration — functionally equivalent to code in terms of what they control and what access to them enables. A SQL injection that can UPDATE user data can also UPDATE system prompts, but the forensic and change-management controls designed to detect unauthorized code changes will not catch a database UPDATE. The fix is architectural: separate storage, separate credentials, separate write authority, separate audit log.

**2. Parameterize key names, not just values.**
The existence of value parameterization does not guarantee injection safety. Any user-controlled input that reaches a SQL construction path — including key names, column names, table names, and other structural identifiers — must be treated as a potential injection vector and either parameterized or validated against an explicit allowlist. Threat models that assume only values need parameterization will produce exactly this class of vulnerability.

**3. Authentication on all endpoints is a posture, not a checklist.**
Twenty-two unauthenticated endpoints on a platform handling M&A advisory data represents an architectural posture that treats authentication as an optional feature applied to individual endpoints rather than a default that requires explicit architectural exception. Changing the posture — authentication by default, with documented exceptions — would have blocked this attack before any injection was attempted.

**4. Responsible disclosure framing does not resolve "should this data have been enumerated in the first place."**
CodeWall's responsible disclosure is the proximate reason no data was exfiltrated. The responsible disclosure framing — and the HackerOne policy that authorized the engagement — does not answer whether systematically enumerating a production database containing 46.5M real M&A advisory chat messages falls within the scope of what McKinsey intended to authorize. This is a live question for organizations publishing responsible disclosure policies that are now reachable by autonomous agents capable of interpreting them as operational authorization.

**5. The era of human-directed AI attacks has a peer: fully autonomous offensive AI agents.**
AAGF-2026-051 established that human operators can use AI tools to achieve nation-state-scale attack operations. This incident establishes that an AI agent can select a target, conduct reconnaissance, construct and execute an exploit, and generate a comprehensive report — with the human's role limited to post-hoc responsible disclosure. These are analytically distinct threat classes requiring distinct defensive responses. Detection strategies, red team exercises, and threat models that assume a human operator in the attack loop will be insufficient for this class.
