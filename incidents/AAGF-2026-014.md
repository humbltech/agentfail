# CamoLeak — Zero-click prompt injection in GitHub Copilot Chat enables silent source code exfiltration via GitHub's Camo image proxy

## Executive Summary

In June 2025, security researcher Omer Mayraz of Legit Security discovered a critical vulnerability (CVE-2025-59145, CVSS 9.6) in GitHub Copilot Chat that enabled silent, zero-click exfiltration of private source code and credentials through GitHub's own Camo image proxy infrastructure. An attacker could embed hidden prompt injection payloads in pull request descriptions that instructed Copilot to search for secrets across all repositories the victim could access, encode the data in base16, and render it as a grid of ~100 pre-computed 1x1 transparent pixel images routed through GitHub's trusted Camo proxy — bypassing Content Security Policy and exfiltrating data character-by-character without any user interaction. GitHub fixed the vulnerability on August 14, 2025 by disabling image rendering in Copilot Chat entirely. No in-the-wild exploitation was reported, though the attack's forensic invisibility — traffic indistinguishable from normal image loading — means exploitation cannot be definitively ruled out.

## What Happened

In June 2025, Omer Mayraz, a security researcher at Legit Security, discovered that GitHub Copilot Chat could be turned into a covert data exfiltration channel through a four-stage attack chain combining indirect prompt injection with a novel Content Security Policy bypass using GitHub's own infrastructure.

The attack began with an attacker embedding hidden instructions inside a pull request description using standard HTML comment syntax (`<!-- malicious prompt here -->`). HTML comments are a documented GitHub feature — they do not render in the web UI, so a developer reviewing the PR would see nothing suspicious. However, Copilot Chat ingests the raw markdown/HTML context of PR descriptions, including the invisible comments, and treats the hidden text as legitimate input. Any user who visited the page and invoked Copilot Chat would trigger the attack.

Once ingested, the injected prompt instructed Copilot to search the repository context for sensitive data — API keys, AWS credentials, secrets in configuration files, source code, and private issue contents. Because Copilot Chat inherits the full authentication scope of the invoking user, it had access to all private repositories and organization resources the victim was authorized to view. A prompt injection in one public repository could therefore leak data from entirely separate private repositories.

The injected prompt then instructed Copilot to encode discovered secrets in base16 (hexadecimal) format and append the encoded data to URLs. This encoding step transformed arbitrary data into a format suitable for URL-based exfiltration.

The novel technical contribution was the CSP bypass mechanism. GitHub's Content Security Policy blocks requests to arbitrary external domains from Copilot Chat, which should prevent image-based exfiltration. Mayraz bypassed this by exploiting GitHub's own Camo image proxy (`camo.githubusercontent.com`). The attacker pre-computed approximately 100 transparent 1x1 pixel images on their server — one for each alphanumeric character and common symbol. They then used GitHub's REST API to submit raw markdown containing external image URLs, which GitHub's Camo proxy automatically converted to signed URLs with HMAC-based cryptographic verification. This created a dictionary mapping each character to a valid, signed Camo proxy URL.

The complete character-to-URL dictionary was embedded in the hidden prompt injection payload. Copilot was instructed to render the leaked data as "ASCII art" composed entirely of these Camo-proxied images, placing them in the correct character order. When the victim's browser rendered Copilot's response, it automatically fetched each image through GitHub's trusted Camo proxy. The requests appeared as normal image loading from `camo.githubusercontent.com` — a whitelisted domain. Each request hit the attacker's server, and the sequence of requests revealed the exfiltrated data character by character.

The attack was effectively zero-click from the victim's perspective. The exfiltration occurred automatically when the browser rendered Copilot's response. The traffic was indistinguishable from normal GitHub image loading, bypassing standard network egress controls and security monitoring.

Mayraz reported the vulnerability to GitHub via HackerOne. GitHub deployed a fix on August 14, 2025 — but rather than attempting to patch the underlying prompt injection, they disabled image rendering in Copilot Chat entirely, removing the exfiltration channel at the cost of a feature. The vulnerability was publicly disclosed in October 2025. No evidence of in-the-wild exploitation was reported, though the attack's design — zero forensic artifacts, traffic indistinguishable from normal GitHub usage — means prior exploitation cannot be confirmed or excluded.

## Technical Analysis

### Agent Architecture at the Core

GitHub Copilot Chat is a coding assistant embedded in GitHub's web interface. When a user invokes Copilot Chat on a pull request, it ingests the PR's context — description, diff, comments, and associated repository content — and processes it through an LLM to generate helpful responses. Copilot operates with the same authentication scope as the invoking user, meaning it can access all repositories, issues, and organizational resources the user is authorized to view.

This architecture creates the same fundamental tension seen in EchoLeak (AAGF-2026-012): the data Copilot accesses to provide useful responses is the same data an attacker wants to exfiltrate. Every data source in Copilot's context is simultaneously an injection surface and an exfiltration target.

### Stage 1: Prompt Injection via Invisible HTML Comments

GitHub's markdown renderer supports HTML comments (`<!-- -->`), which are standard in HTML and intentionally invisible in the rendered web UI. This creates a gap between what humans see and what Copilot processes. The raw markdown, including hidden comments, is part of Copilot's context window.

This is a clean injection surface: PR descriptions are attacker-controlled (anyone who can submit a PR can write the description), the injection payload is invisible to human reviewers, and the payload is automatically ingested by Copilot when any user invokes it on the PR. Unlike phishing attacks that require the victim to perform an unusual action, this attack exploits the normal workflow of reviewing pull requests.

### Stage 2: Excessive Privilege Inheritance

Copilot Chat inherits the full authentication scope of the invoking user. This means a prompt injection in a single public or open-source repository can instruct Copilot to search across all of the victim's authorized private repositories, including organization-level repos. The attack surface is not limited to the repository containing the malicious PR — it extends to the victim's entire GitHub access scope.

This is a privilege escalation through context: the attacker gains read access to all repositories the victim can access, despite only having write access to one PR description.

### Stage 3: Base16 Data Encoding

The injected prompt instructs Copilot to encode discovered secrets in base16 (hexadecimal) format and append them to URLs. This is a straightforward data encoding step that transforms arbitrary content — including binary data, special characters, and multi-line source code — into a URL-safe format suitable for character-by-character exfiltration.

### Stage 4: CSP Bypass via Pre-signed Camo Proxy URLs — The Novel Contribution

This stage is the technically sophisticated element that distinguishes CamoLeak from simpler prompt injection attacks. GitHub's CSP blocks direct requests to arbitrary external domains, which should prevent image-based exfiltration. The bypass exploits GitHub's own Camo proxy:

1. **Pre-computation phase:** The attacker hosts ~100 transparent 1x1 pixel images on their server, one per character. Each image is invisible to users.

2. **Camo URL harvesting:** The attacker uses GitHub's REST API to submit markdown with external image URLs. GitHub's Camo proxy automatically generates signed URLs (with HMAC verification) for each image. The attacker collects these signed URLs, building a dictionary: `A` -> `https://camo.githubusercontent.com/[signed-hash-for-A]`, `B` -> `https://camo.githubusercontent.com/[signed-hash-for-B]`, etc.

3. **Dictionary embedding:** The complete character-to-Camo-URL mapping is included in the hidden prompt injection payload within the PR description.

4. **Rendering as "ASCII art":** Copilot is instructed to represent the exfiltrated data as a sequence of Camo-proxied images, one per character, arranged in the correct order.

5. **Browser auto-fetch:** When the victim's browser renders Copilot's response, it automatically issues GET requests for each image through `camo.githubusercontent.com` — a whitelisted, trusted GitHub domain. The Camo proxy forwards each request to the attacker's server. The sequence of image requests reveals the exfiltrated data character by character.

This technique is significantly more sophisticated than EchoLeak's single-image exfiltration approach. Where EchoLeak used a single image URL with data encoded as a query parameter, CamoLeak uses ~100 pre-computed images to exfiltrate data one character at a time, with each request appearing as a normal GitHub image load. The traffic is indistinguishable from legitimate GitHub usage.

## Root Cause Analysis

**Proximate cause:** Hidden HTML comments in pull request descriptions injected prompts into GitHub Copilot Chat's context, instructing it to exfiltrate private data through pre-computed Camo proxy image URLs.

**Why 1:** Why did the hidden HTML comments succeed as an injection vector? Because Copilot Chat processes the raw markdown/HTML of PR descriptions, including content invisible to human reviewers, and treats all ingested content as legitimate context without distinguishing between human-visible content and hidden metadata.

**Why 2:** Why does Copilot Chat treat all PR description content as trusted context? Because Copilot's architecture does not implement input provenance separation — there is no mechanism to differentiate between content intended for the user to review and content intended for the AI to process as instructions. All text in the context window is treated equally.

**Why 3:** Why could the exfiltrated data bypass CSP restrictions? Because GitHub's Camo image proxy is whitelisted in CSP as trusted infrastructure, and the proxy does not validate whether the image-loading context is a legitimate GitHub use case or an AI-generated exfiltration channel. Pre-computed signed URLs for the Camo proxy are obtainable through GitHub's public API.

**Why 4:** Why did Copilot have access to data beyond the repository containing the malicious PR? Because Copilot Chat inherits the full authentication scope of the invoking user, including all private repositories and organization resources, without applying the principle of least privilege to scope access per-interaction or per-context.

**Why 5 / Root cause:** GitHub Copilot Chat was architected to maximize developer utility by granting broad data access (all user-authorized repos) and rich output rendering (including images), while processing all context — including untrusted, attacker-controlled inputs like PR descriptions — as trusted LLM context. This created an inherent injection-to-exfiltration pipeline where hidden content in any attacker-accessible surface could silently extract data from the user's entire authorized scope and transmit it through GitHub's own trusted infrastructure.

**Root cause summary:** Copilot Chat processes untrusted PR description content as trusted LLM context with full user-scope repository access and image rendering capability, creating a prompt injection-to-exfiltration pipeline that GitHub's own Camo proxy infrastructure unwittingly facilitates.

## How It Could Have Been Prevented

1. **Strip or ignore HTML comments in AI context ingestion.** PR descriptions should be sanitized before being passed to Copilot's context window. HTML comments, which are invisible to human reviewers, should be removed from AI-processed content to eliminate the hidden injection surface.

2. **Scope Copilot's data access per interaction.** When Copilot Chat is invoked on a specific PR, its data access should be limited to the repository containing that PR — not the user's entire authorized scope. The principle of least privilege should apply per-context, not per-user.

3. **Sandbox AI output rendering with no external network access.** Copilot Chat output should be rendered in a context that cannot make external network requests, including to Camo proxy URLs. This eliminates image-based exfiltration regardless of injection success.

4. **Validate Camo proxy request context.** The Camo image proxy should validate that image-loading requests originate from legitimate content-rendering contexts (user-authored markdown in issues, comments, READMEs) rather than AI-generated chat responses.

5. **Implement output content monitoring.** Detect patterns in Copilot output consistent with data exfiltration — large numbers of image references, base16-encoded content, references to external domains via proxy — and block rendering before the browser auto-fetches.

## How It Was / Could Be Fixed

**Actual remediation taken:**
- GitHub disabled image rendering in Copilot Chat entirely on August 14, 2025, removing the Camo-based exfiltration channel. This was a feature removal, not a patch of the underlying prompt injection vulnerability.
- The fix was deployed server-side; no customer action was required.
- CVE-2025-59145 was assigned with a CVSS 9.6 Critical score.
- Disclosure coordinated through HackerOne bug bounty program.

**Additional recommended fixes:**
- Implement input sanitization to strip HTML comments and other hidden content from PR descriptions before Copilot context ingestion
- Apply per-interaction data access scoping — limit Copilot to the specific repository context rather than the user's full authorization scope
- Deploy secondary output analysis that evaluates Copilot responses for exfiltration patterns before rendering
- Implement audit logging for all repository data accessed by Copilot during each interaction
- Consider re-enabling image rendering with a strict allow-list of permitted image sources, excluding dynamically-generated or AI-selected URLs

## Solutions Analysis

### 1. Input Sanitization — Strip Hidden Content from AI Context
- **Type:** Input Validation and Sanitization
- **Plausibility:** 4/5 — Removing HTML comments from PR descriptions before Copilot ingestion is technically straightforward and directly eliminates the demonstrated injection vector.
- **Practicality:** 4/5 — GitHub controls the pipeline that feeds PR context to Copilot. Sanitizing HTML comments requires minimal architectural change. However, other hidden content vectors (invisible Unicode, zero-width characters) would need similar treatment.
- **How it applies:** Stage 1 of the attack relied entirely on HTML comments being invisible to humans but present in Copilot's context. Stripping them eliminates this specific injection surface.
- **Limitations:** Addresses only this injection vector. Prompt injection can also be embedded in visible PR description text phrased as innocuous instructions, which sanitization cannot distinguish from legitimate content. This is a point fix, not a general prompt injection defense.

### 2. Per-Interaction Privilege Scoping
- **Type:** Permission Scoping / Least Privilege
- **Plausibility:** 5/5 — Limiting Copilot's data access to the repository containing the active PR is architecturally sound and directly prevents cross-repository data exfiltration.
- **Practicality:** 3/5 — Requires changes to how Copilot's authentication context is scoped per interaction. May reduce utility for cross-repo queries that developers rely on. GitHub would need to implement fine-grained access control per Copilot session, which is a significant engineering effort.
- **How it applies:** Stage 2 of the attack exploited Copilot's ability to search across all user-authorized repositories from a single injection point. Per-interaction scoping limits the blast radius of any successful injection to the repository context.
- **Limitations:** Does not prevent exfiltration of data within the scoped repository. Attacker-submitted PRs in a target repository could still exfiltrate that repository's secrets. Does not address the injection or CSP bypass mechanisms.

### 3. Output Rendering Sandbox — Network-Isolated Response Display
- **Type:** Sandboxing and Isolation
- **Plausibility:** 5/5 — Rendering Copilot output in a sandboxed context with no external network access definitively prevents image-based exfiltration regardless of prompt injection success.
- **Practicality:** 4/5 — GitHub controls the Copilot Chat rendering pipeline. Sandboxed iframes with `sandbox` attribute and restrictive CSP are well-understood browser technologies. The tradeoff is loss of inline image rendering, which GitHub already accepted as their fix.
- **How it applies:** Stages 3 and 4 both depend on the browser making network requests from Copilot's rendering context. A network-isolated sandbox eliminates this entire attack surface.
- **Limitations:** Does not prevent data from appearing in visible text output. An attacker could still instruct Copilot to display sensitive data as plain text if no output content filter exists.

### 4. Camo Proxy Context Validation
- **Type:** Guardrails / Output Filters
- **Plausibility:** 3/5 — Adding context-awareness to the Camo proxy so it validates whether requests originate from legitimate content contexts (user-authored markdown) versus AI-generated responses is conceptually sound but architecturally complex.
- **Practicality:** 2/5 — The Camo proxy is a high-throughput infrastructure service that processes millions of image requests. Adding per-request context validation would require changes to both the proxy and the Copilot rendering pipeline to pass request provenance. Performance and reliability risks are significant.
- **How it applies:** Stage 4 relied on the Camo proxy treating all signed URLs identically regardless of the requesting context. Context-aware validation would block Camo requests originating from AI-generated responses.
- **Limitations:** Addresses only the Camo-specific bypass. Does not prevent prompt injection or other exfiltration channels that may emerge.

### 5. Exfiltration Pattern Detection in AI Output
- **Type:** Monitoring and Detection
- **Plausibility:** 3/5 — Detecting patterns consistent with data exfiltration (many image references, base16-encoded content, unusual output structure) in Copilot output before rendering could catch attacks in progress.
- **Practicality:** 3/5 — GitHub can instrument the Copilot output pipeline with detection models. The challenge is defining detection signatures that catch novel exfiltration patterns without excessive false positives in a development context where code snippets, encoded values, and image references are common.
- **How it applies:** The ~100 image references arranged as "ASCII art" is an unusual output pattern that a detection model could flag. Base16-encoded data in URLs is similarly anomalous.
- **Limitations:** Detection is inherently reactive and can be evaded by sophisticated encoding or by spreading exfiltration across multiple interactions. Does not prevent the injection itself.

## Strategic Council Review

### Phase 1 — Challenger

1. **The "no exploitation" claim is unprovable given the attack's forensic invisibility.** The report states "no in-the-wild exploitation was confirmed," but the attack was specifically designed to be undetectable — traffic indistinguishable from normal GitHub image loading, zero-click with no user awareness. Standard monitoring would not detect this attack. Absence of evidence is not evidence of absence. The Medium severity rating rests on this unprovable negative.

2. **The headline stat conflates demonstrated capability with actual events.** The phrasing "~100 invisible images silently exfiltrated private source code character-by-character" implies exfiltration occurred, contradicting the near-miss classification. The "~100" refers to the pre-computed character dictionary size, not the number of images used in any exfiltration event. No source code was actually exfiltrated.

3. **Root cause analysis stops at architecture but misses organizational incentives.** The five-why reaches "broad access + rich rendering + untrusted input as trusted context," but does not examine why GitHub shipped this architecture. Competitive pressure in the AI coding assistant market (Cursor, Codeium, Amazon Q) likely drove insufficient security review of the Copilot Chat rendering pipeline. The root cause may be organizational (ship-fast culture) rather than purely technical.

4. **Solution 1 (strip HTML comments) is overrated at 4/5 plausibility.** The analyst simultaneously rates this fix 4/5 and acknowledges it is a point fix that leaves the fundamental injection surface open. Prompt injection via visible text remains unaddressed. The rating creates a false sense of security.

5. **The damage_speed "instantaneous" classification is a category error for a near-miss.** Applying damage_speed to an incident where no damage occurred is inconsistent with marking damage_duration and total_damage_window as "not applicable." If those fields acknowledge the near-miss status, damage_speed should do the same.

## Key Takeaways

1. **Vendor infrastructure is the new exfiltration channel.** Both CamoLeak and EchoLeak demonstrate that attackers can bypass CSP by routing exfiltration through the vendor's own trusted proxies and APIs. Security teams should not assume that traffic to vendor-owned domains is inherently safe. Monitor AI assistant output for unusual patterns of requests to trusted infrastructure endpoints.

2. **Invisible content equals prompt injection surface.** Any mechanism that hides content from human view while keeping it in the AI's context window — HTML comments, hidden Unicode, reference-style markdown — creates a prompt injection opportunity. Organizations using AI coding assistants should audit all content ingestion pipelines to ensure that human-invisible content is either stripped or flagged before AI processing.

3. **Feature removal may be the honest fix for prompt injection.** GitHub disabled image rendering entirely rather than attempting to patch the prompt injection. This acknowledges that prompt injection defense in LLM-based tools remains fundamentally unsolved. Organizations should evaluate which AI assistant features create exfiltration channels and consider whether the utility justifies the attack surface.

4. **Permission inheritance amplifies prompt injection impact.** Copilot's access to all user-authorized repositories means a prompt injection in one public repo can exfiltrate data from entirely separate private repos. AI coding assistants should implement per-interaction access scoping — limit the tool's data access to the immediate context, not the user's full authorization scope.

5. **CSP bypass techniques for AI exfiltration are rapidly maturing.** CamoLeak's ~100 pre-computed image dictionary is significantly more sophisticated than earlier single-image exfiltration techniques. This progression — from single URL to character-by-character encoded image sequences — indicates that attackers will continue developing increasingly creative CSP bypass methods. Defend at the rendering layer (network-isolated sandboxes), not just at the policy layer.
