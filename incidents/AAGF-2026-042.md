# Claude Code Command Injection via Piped sed/echo Bypasses File Write Restrictions (CVE-2026-25723)

## Executive Summary

A command injection vulnerability in Anthropic's Claude Code coding assistant (npm: `@anthropic-ai/claude-code`, versions < 2.0.55) allowed an attacker to bypass the tool's blocklist-based file write restrictions by crafting piped `sed`/`echo` command combinations that appeared benign to the validator but executed in ways that wrote files to unauthorized paths — including `.claude/settings.json`, which stores hooks and MCP server configurations sourced on every session startup. CVE-2026-25723 is the second bypass of the same validation layer in 78 days: a predecessor technique (CVE-2025-64755, sed `w` command variant, November 2025) had been patched in v2.0.31, but the fix hardened only the specific variant without eliminating the underlying blocklist approach. No confirmed in-the-wild exploitation was documented. Anthropic published a patch (v2.0.55) and advisory simultaneously on February 6, 2026, delivering the fix silently via auto-update to users on default settings.

## What Happened

Claude Code is a developer productivity tool that operates with deep integration into a developer's local environment: it reads and writes files, executes shell commands, and manages project configuration. For users who enable "accept edits" mode — a productivity feature that allows Claude Code to apply suggested changes without per-change confirmation — Claude Code validates each proposed command against a blocklist of patterns before executing it. This validation layer is the primary guardrail preventing Claude Code from writing to sensitive paths it should not touch: the `.claude/` configuration directory and paths outside the project working directory.

On November 20, 2025, Anthropic patched the first publicly documented bypass of this validation layer. Security researcher Adam Chester (SpecterOps) demonstrated that sed's `w` (write) command, when invoked with variant delimiter syntax, slipped through the regex patterns used to detect write operations. Claude Code v2.0.31 updated the regex to catch that specific variant.

The fix was incomplete. The validation approach — blocklist-based inspection of command strings — remained architecturally intact. The patch closed a specific hole without changing the fundamental model: Claude Code was attempting to determine whether a shell command would write to a sensitive path by pattern-matching against a list of known-bad constructs. Shell pipelines, composed of multiple tools with overlapping and often surprising capabilities, are not amenable to safe blocklist classification.

Seventy-eight days after the predecessor patch, nil221 reported to Anthropic via HackerOne that a different shell pipeline composition bypassed the same layer. Piped combinations of `sed` and `echo` — tools so common they appear in routine shell one-liners — could be constructed to produce file writes that the validator did not recognize as such. Related research by Flatt Security's "8 ways to pwn Claude Code" documents the sed `e` modifier specifically: `echo test | sed 's/test/touch \/tmp\/pwned/e'` causes `sed` to execute the substitution result as a shell command rather than emitting text output. Whether CVE-2026-25723 uses this exact variant or a different sed/echo pipeline composition that redirects output to a restricted path is not confirmed in public sources — nil221 has not published a proof-of-concept.

The attack chain, under conditions of "accept edits" enabled, is:

1. An attacker influences the commands Claude Code will suggest. This influence can arrive via indirect prompt injection — a malicious repository's `CLAUDE.md` file, a crafted webpage fetched during research, or a response from a compromised MCP server.
2. The crafted command appears plausible: a routine-looking sed one-liner for text processing, piped from echo, perhaps part of a build or initialization step.
3. Claude Code's validator inspects the command, fails to recognize the pipeline as a write operation, and allows it through.
4. The user accepts the command (or auto-accept is enabled).
5. The command executes. The write target determines the severity of the outcome.

The highest-value write target is `.claude/settings.json`. This file is not merely a configuration store — it is the authority document that Claude Code sources on every session startup to determine which tools are permitted, which hooks to execute, and which MCP servers to load. Writing to it transforms a one-time exploit into a persistent backdoor: every future Claude Code session on that machine will execute attacker-controlled hooks or load attacker-specified MCP servers before the user's first keypress. Shell initialization files (`.zshenv`, `.bashrc`) represent a parallel persistence path, surviving Claude Code reinstalls and applying to all shell activity on the machine.

Anthropic's response was technically sound: the advisory and the patch were published simultaneously on February 6, 2026, meaning the fix was ready before public disclosure. The auto-update mechanism delivered the patch silently to the majority of users. The advisory itself is sparse on technical detail — standard practice for responsible disclosure to avoid providing a functional recipe before widespread patching — but the lack of a detailed post-patch technical breakdown means the full scope of the vulnerability class was not communicated to operators who might need to audit their environments.

## Technical Analysis

### Architecture: How Claude Code's File Write Restrictions Work

Claude Code enforces file write permissions through a command validation layer in its "accept edits" pipeline. When the tool is about to execute a command that originated from model suggestions, this layer inspects the command string against a set of blocklist patterns designed to detect writes to the `.claude/` directory and paths outside the project working directory.

The validation logic is implemented in `bashPermissions.ts`. The AAGF-2026-036 analysis (the 50-subcommand deny-rule bypass) previously documented that this file contains a `MAX_SUBCOMMANDS_FOR_SECURITY_CHECK` constant that causes the validator to stop evaluating after 50 subcommands. CVE-2026-25723 is a different failure in the same component: the regex-based pattern matching does not correctly parse shell pipeline composition.

### The Predecessor Bypass (CVE-2025-64755)

Adam Chester's November 2025 research demonstrated that sed's `w` command writes to files natively — it is not just a text transformation tool. The specific bypass exploited sed syntax variants that the validator's regex patterns failed to match:

```bash
echo 'runme' | sed 'w /Users/xpn/.zshenv'
echo 'echo 123' | sed -n '1,1w/Users/xpn/.zshenv'
```

The fix in v2.0.31 updated the regex patterns to catch these variants. But the approach remained: blocklist regex patterns applied to command strings.

### CVE-2026-25723: The Pipeline Composition Bypass

The CVE-2026-25723 bypass uses piped `sed` and `echo` combinations. The validator does not correctly evaluate command pipelines involving shell redirection combined with text processing utilities — the pipeline appears benign in isolation but executes file writes when the full command composition is evaluated by the shell.

The most technically precise parallel, documented by Flatt Security's research on the same validation layer, is the sed `e` modifier:

```bash
echo test | sed 's/test/touch \/tmp\/pwned/e'
```

(Note: the sed `e` modifier is the best-documented adjacent technique from Flatt Security research; the exact technique used by nil221 in CVE-2026-25723 has not been publicly disclosed.)

The `e` modifier in sed substitution (`s/pattern/replacement/e`) executes the substituted result as a shell command. In this example, after substitution, `touch /tmp/pwned` is passed to the shell for execution. The file write does not occur through sed's own file-writing mechanism — it occurs through an arbitrary shell command that `sed` launches as a subprocess. This is why blocklist patterns targeting sed file-write syntax (the `w` command family) do not catch it: the `e` modifier is a code execution primitive, not a write primitive.

Applied to Claude Code's restricted paths:

```bash
echo '#!/bin/bash\n[malicious hook content]' | sed 's/.*/echo & >> .claude\/settings.json/e'
```

From the validator's perspective, this is a benign `echo` piped to `sed` with a substitution expression. No `w` command pattern. No obvious file redirection (`>`, `>>`). The shell, however, executes the substitution result as a command — which performs the write.

### Why Writing to `.claude/settings.json` Is Particularly Dangerous

The `settings.json` file is Claude Code's trust anchor. It contains:

- **Hooks**: Commands executed automatically on session events (startup, file change, tool invocation). A hook entry here runs on every future Claude Code session, independent of any subsequent security fix — it survives version upgrades unless the file is audited and cleaned.
- **MCP server configurations**: Which external tool servers Claude Code will connect to and trust. Adding a malicious MCP server here means every future Claude Code session will offer the attacker's server as a tool source, enabling ongoing indirect prompt injection with zero further user interaction.
- **`bypassPermissions` mode**: The `settings.json` can set permission modes that override Claude Code's normal safety gates. Setting `bypassPermissions: true` eliminates the file write restriction layer entirely — the very layer CVE-2026-25723 bypasses is no longer evaluated for future sessions.

This connects directly to the TrustFall pattern (CVE-2026-33068): writing to `settings.json` to enable `bypassPermissions` is the natural second-stage payload after an initial file-write bypass. The first exploit establishes settings.json write capability; the settings file sets `bypassPermissions: true`; subsequent Claude Code sessions operate without the restriction layer. (This represents a threat model inference from capability analysis, not a documented exploitation path — the chain is technically plausible but has not been observed in the wild.)

The `.claude/settings.json` file is also designed to be committed to source control for team sharing — a poisoned settings file in a shared repository cascades to every developer who clones it, multiplying the blast radius from a single developer machine to an entire engineering team.

### Shell Init File Persistence

The companion persistence path — writing to `.zshenv`, `.bashrc`, or equivalent — does not require Claude Code to be reinstalled and survives tool upgrades. Shell init files execute on every new shell session for the user. Code written here has access to all environment variables visible to the shell: `ANTHROPIC_API_KEY`, AWS credentials (`~/.aws/credentials` path or `AWS_ACCESS_KEY_ID` env vars), SSH agent socket, and any cloud provider CLI tokens in scope.

### CVSS and Attack Conditions

The CVSS v4.0 vector (7.7 HIGH) appropriately reflects the attack conditions:
- **Attack Vector: Network (AV:N)** — the exploit payload can arrive via a malicious remote repository, webpage fetched by the agent, or MCP server response
- **Attack Complexity: Low (AC:L)** — once a prompt injection vector exists, crafting the bypass command requires no specialized access
- **Privileges Required: None (PR:N)** — the attacker does not need any account or credential on the victim machine
- **User Interaction: Passive (UI:P)** — the user must have "accept edits" enabled and must accept the command; this is passive interaction, not an explicit extraordinary action
- **Attack Requirements: Present (AT:P)** — "accept edits" must be enabled; this is the primary friction point, but it is a common productivity configuration and auto-accept environments require no interaction at all

## Root Cause Analysis

**Proximate cause:** Claude Code's blocklist-based command validation layer in `bashPermissions.ts` did not correctly parse shell pipeline composition involving `sed` and `echo`, allowing a piped command that performed a file write to restricted paths to pass validation as benign.

**Why 1: Why did the validator fail to detect the pipeline as a write operation?**
The validator relied on regex pattern matching against command strings — a fundamentally string-centric approach to a problem that requires semantic understanding of shell execution semantics. Shell pipelines can compose many tools, each with capabilities that can be combined to achieve effects not present in any single tool's surface area. A string-based blocklist that enumerates known-bad patterns cannot enumerate all compositions of tools that produce equivalent effects.

**Why 2: Why did Anthropic use a blocklist approach when the predecessor bypass had already demonstrated its fragility?**
The CVE-2025-64755 patch (v2.0.31) updated the blocklist regex to catch the sed `w` command variants — it did not change the validation architecture. This choice had a specific historical context documented in the research: an earlier Claude Code version (v1.0.93) had switched to an allowlist approach in response to Flatt Security's research documenting the broader bypass class. That allowlist was not applied to the "accept edits" file-write validation pathway — that pathway retained blocklist logic that was then separately bypassed twice.

**Why 3: Why was the allowlist approach not applied to the file-write validation pathway?**
The evidence suggests a validation architecture that was siloed by feature path: the allowlist applied to one part of the permission system while the "accept edits" path continued using a separately maintained blocklist. This is consistent with Claude Code's rapid development velocity — features are added and security controls are applied at the feature level rather than at a unified permission enforcement layer. The `MAX_SUBCOMMANDS_FOR_SECURITY_CHECK = 50` constant documented in AAGF-2026-036 is evidence of the same pattern: a hard-coded safety threshold applied at the feature level without architectural safety-by-default.

**Why 4: Why did the development process not catch that the v2.0.31 fix left the bypass class open?**
The fix was validated against the specific sed `w` command variants that Chester demonstrated — it was not validated against the broader class of shell pipeline compositions capable of writing to restricted paths. Closing one hole in a blocklist is not equivalent to eliminating the bypass class. No evidence of security review that evaluated whether the blocklist approach itself was sufficient has appeared in public sources. The 78-day gap between the predecessor fix and this disclosure suggests the broader bypass class was not the subject of internal enumeration after v2.0.31.

**Why 5 / Root cause:** Anthropic's security review process for Claude Code's permission enforcement layer was not designed to evaluate bypass completeness — it evaluated specific reported bypass variants without assessing whether the underlying validation architecture (blocklist-based string matching) was fundamentally capable of providing the intended security property (preventing file writes to sensitive paths via arbitrary shell pipeline composition). The result is a predictable pattern: each patched bypass fixes the specific reported technique while leaving the class open for the next discoverer.

**Root cause summary:** Blocklist-based regex validation applied to shell command strings is architecturally insufficient for the security property Claude Code requires — it can be bypassed by any novel shell pipeline composition that achieves write effects through non-enumerated mechanisms — and Anthropic's post-CVE-2025-64755 patch process did not include evaluation of whether the bypass class had been closed.

## Impact Assessment

**Severity:** Medium — CVSS v4.0 7.7 HIGH, but no confirmed exploitation, and the "accept edits" requirement provides meaningful friction for most users. The `severity` field tracks realized impact; the CVSS 7.7 HIGH score reflects technical attack potential, while the Medium severity rating reflects the near-miss classification (no confirmed exploitation) and the `accept edits` precondition. The near-miss potential is orders of magnitude larger than documented actual harm.

**Who was affected:**
- All `@anthropic-ai/claude-code` users on versions 2.0.31–2.0.54 who had "accept edits" enabled and who interacted with attacker-influenced content (malicious repositories, web content, compromised MCP servers) during the ~78-day window
- Users on auto-accept configurations (CI/CD pipelines using Claude Code, unattended agentic workflows) had no interaction friction at all
- Users who had disabled auto-update or pinned versions below 2.0.55 remained exposed after February 6, 2026

**What was at risk:**
- `.claude/settings.json`: persistent hook implantation, malicious MCP server registration, `bypassPermissions` escalation
- Shell initialization files: persistent arbitrary code execution on every future shell session
- Project files outside scope: CI/CD configuration poisoning, supply chain attack via committed `.claude/settings.json`
- Developer machine credentials: `ANTHROPIC_API_KEY`, cloud provider CLI credentials, SSH agent access — all in scope for any code executing in the developer's shell context

**Quantified impact (where known):**
- Users affected: Zero confirmed. Estimated exposure: hundreds of thousands of developer machines over 78 days.
- Data exposed: None confirmed.
- Financial impact: None confirmed. Potential class: Anthropic API key theft → unauthorized API billing; cloud credential theft → unauthorized infrastructure usage.
- Recovery time: Not required for patched users; auto-update delivered fix without user action.

**Near-miss potential:**
The VentureBeat "Six Exploits Broke AI Coding Agents" framing identifies the core risk precisely: AI coding agents hold developer credentials as a matter of course. The attacker's goal is not to compromise the AI model — it is to steal the credentials the agent holds and use them to access production systems. The 78-day window, applied to a user base of hundreds of thousands, represents a substantial period during which any attacker with the capability to influence Claude Code's command execution (via prompt injection through any of the attack surfaces available to a coding agent) could have established persistent access to developer machines with no post-exploit footprint beyond a modified settings file.

The `.claude/settings.json` supply chain vector is the most concerning dimension: a poisoned settings file committed to a shared repository would distribute the payload to every developer who clones it, transforming a single successful exploit into a team-wide backdoor.

**Containment:** The primary containment mechanism was Anthropic's auto-update, which silently patched the majority of users. Users on pinned versions or with auto-update disabled had no in-product detection or alerting mechanism.

## How It Could Have Been Prevented

1. **Allowlist-based command validation instead of blocklist:** The validation architecture should enumerate permitted command patterns — the known-safe set — rather than attempting to enumerate the unsafe set. Shell commands are Turing-complete compositions; the unsafe set is unbounded. This architectural direction was taken for part of the Claude Code permission system (in v1.0.93, in response to Flatt Security's research) but was not applied to the "accept edits" file-write validation path. Applying allowlist validation uniformly would have prevented both CVE-2025-64755 and CVE-2026-25723.

2. **Semantic command analysis rather than string matching:** The validation layer attempts to determine the effect of a command by matching its string representation. A semantically correct approach would parse the command into its AST (abstract syntax tree), trace tool invocations, and evaluate the capability set of the full pipeline — including subprocess execution via flags like sed's `e` modifier. Tree-sitter, the parser toolchain already present in Claude Code's codebase, is capable of this analysis. (AAGF-2026-036 documented that a working tree-sitter fix for a related bypass class existed in Anthropic's codebase before that vulnerability's external disclosure but was not shipped.)

3. **Write-destination audit logging and alerting:** If Claude Code had logged all filesystem write operations with their actual destinations (resolved after execution), security teams and individual developers would have a tamper-evident audit trail to detect unauthorized writes. For `.claude/settings.json` specifically, a read-before-write hash comparison on session startup would detect unauthorized modifications between sessions.

4. **Immutable or append-only `.claude/settings.json` by default:** The settings file could be protected by a system-level integrity mechanism (macOS SIP-style protection for the `.claude/` directory, or a per-session signature verification) that prevents modification through Claude Code's own execution context. This would not prevent all attack vectors but would block the specific high-value write target that converts a transient exploit into persistent access.

5. **Broader bypass-class analysis after CVE-2025-64755:** After patching the sed `w` command bypass, Anthropic should have commissioned or conducted a complete enumeration of shell pipeline compositions capable of writing to restricted paths — sed `e` modifier, process substitution, tee, awk output redirection, Python/perl one-liners, heredoc redirections, and others. This type of bypass-class completeness analysis would likely have caught CVE-2026-25723 before nil221's report. The 78-day gap between patches suggests this analysis was not done.

## How It Was / Could Be Fixed

**Actual remediation taken:**
Anthropic released Claude Code v2.0.55 on February 6, 2026, simultaneously with the advisory. The patch updated the command validation logic in `bashPermissions.ts` to detect and block the piped sed/echo bypass technique. The auto-update mechanism delivered the fix silently to users on default settings. No PoC was published, consistent with responsible disclosure practice.

**What the fix does not address:**
- The blocklist architecture: v2.0.55 closes the specific reported bypass technique. If the fix follows the same pattern as v2.0.31 (patching the specific variant without architectural change), the bypass class remains open for a third discoverer. The research available does not confirm whether v2.0.55 switched to an allowlist approach or again updated the blocklist.
- The three simultaneous bypass paths documented around the same period: CVE-2026-25724 (symlink bypass, patched separately in v2.1.7) and the `settings.local.json` reliability failure (GitHub #8961, unacknowledged as of AAGF-2026-036's curation date) suggest the permission enforcement layer has multiple concurrent failure modes.
- Post-exploitation detection: No mechanism to detect unauthorized `.claude/settings.json` modifications was introduced.

**Additional recommended fixes:**
- Migrate the "accept edits" validation path from blocklist to allowlist, matching the approach taken in v1.0.93 for other permission pathways
- Introduce session-startup integrity check on `.claude/settings.json` with user-visible alerting on unexpected changes
- Add explicit documentation guidance for teams sharing `.claude/settings.json` via source control: treat it as a security-sensitive file, enforce code review on changes, consider excluding from version control in high-trust-environment deployments
- Conduct and publish a bypass-class completeness analysis after each blocklist-based fix, or replace the architecture entirely

## Solutions Analysis

### 1. Allowlist-Based Command Validation (Architectural Redesign)

- **Type:** Architectural Redesign
- **Plausibility:** 5/5 — An allowlist that enumerates permitted command patterns is definitively more restrictive than a blocklist. The approach was already implemented for part of Claude Code's permission system (v1.0.93), confirming feasibility.
- **Practicality:** 3/5 — Requires careful enumeration of the permitted command set to avoid blocking legitimate functionality. User friction increases if the allowlist is too restrictive. The fact that Anthropic already deployed this for one subsystem suggests it is feasible, but extending it to all validation paths requires design work and likely an extended compatibility period.
- **How it applies:** The file-write restriction validator would enumerate the commands permitted to execute in "accept edits" mode. Any command not on the allowlist — regardless of whether it appears on the blocklist — would be denied by default. A piped sed/echo combination would be explicitly permitted or denied based on whether the specific pipeline composition appears in the approved list, not whether it matches a known-bad pattern.
- **Limitations:** Allowlists require ongoing maintenance as new tools and command patterns are used legitimately. Over-restrictive allowlists degrade the product's utility, creating pressure to expand them in ways that could reintroduce unsafe compositions. Allowlist-based validation still requires semantic analysis to determine what a permitted command actually does — permitting `sed` in general still allows `sed 's/./x/e'` variants unless the specific flag is also enumerated.

---

### 2. Semantic AST-Level Command Analysis (Guardrails / Output Filters)

- **Type:** Guardrails / Output Filters
- **Plausibility:** 4/5 — Parsing shell commands into their AST and evaluating the capability set of the full pipeline — including subprocess execution flags like sed `e` modifier, tee, process substitution, awk output redirection — would catch the bypass class rather than individual variants. Tree-sitter (used in Claude Code's codebase) is capable of this analysis.
- **Practicality:** 2/5 — Shell AST analysis is significantly more complex than regex matching. Shell grammar edge cases (here-strings, variable expansion, process substitution, eval, command substitution) are numerous and frequently parser-breaking. Achieving high confidence that AST analysis captures all write-capable pipeline compositions would require substantial engineering investment and ongoing maintenance as new shell constructs emerge. False positives (blocking legitimate commands) would create developer friction.
- **How it applies:** The validation layer would parse each proposed command using a shell grammar parser, trace the execution flow through the pipeline, and identify any subcommand invocations — including those triggered by tool flags like sed `e`. A pipeline containing a subprocess-execution flag with unresolved content would be flagged for human review or denied.
- **Limitations:** Shell is notoriously difficult to parse statically. Obfuscation via variable expansion (`A='e'; sed "s/x/y/$A"`) may still defeat AST-level analysis without runtime evaluation. This approach raises the bar significantly but does not provide a complete guarantee.

---

### 3. Filesystem-Level Write Interception via Sandbox (Sandboxing and Isolation)

- **Type:** Sandboxing and Isolation
- **Plausibility:** 5/5 — Operating-system-level filesystem sandboxing (e.g., macOS app sandbox, Linux seccomp/namespaces, or Docker-equivalent isolation) enforces write restrictions at a layer beneath the application's own validation logic. A sandbox that denies writes to `.claude/settings.json` and paths outside the project directory cannot be bypassed by any shell pipeline composition — the OS denies the syscall.
- **Practicality:** 2/5 — Sandboxing a developer tool with legitimate needs for broad filesystem access (reading project files, writing build artifacts, executing arbitrary shell commands the developer requested) is architecturally difficult. macOS app sandbox restrictions would break much of Claude Code's core functionality. A project-scoped sandbox that allows writes to the current working directory but denies writes to `.claude/` and home directory dotfiles would require either OS-level container support or a custom filesystem interposition layer (e.g., via macOS KEXT or Linux LD_PRELOAD). Deployment friction for a developer tool distributed via npm is substantial.
- **How it applies:** Claude Code would execute in an environment where filesystem writes outside the project scope are denied at the syscall level. The `sed 's/test/touch /tmp/pwned/e'` variant would fail at the `touch` syscall, not at Claude Code's validator.
- **Limitations:** Sandboxing breaks legitimate operations that Claude Code's users may depend on (global npm installs, configuration file management, system-wide tool setup). The product would need to explicitly enumerate which paths outside the project directory are writable, reintroducing a variant of the allowlist problem at a lower level. Auto-escalation to ask the user for write permissions to specific paths would require a UX redesign.

---

### 4. Human-in-the-Loop for Sensitive Path Writes (Human-in-the-Loop)

- **Type:** Human-in-the-Loop (HITL)
- **Plausibility:** 4/5 — Requiring explicit user confirmation before any write to `.claude/settings.json`, shell initialization files, or any path outside the project working directory would prevent automated exploitation. The user confirmation step is currently absent for these paths in "accept edits" mode.
- **Practicality:** 4/5 — This is a targeted HITL gate: not requiring confirmation for all file writes, only for writes to a small set of high-sensitivity paths. The friction cost is low — developers rarely legitimately write to `.claude/settings.json` or `.zshenv` via an AI agent's suggested command. The confirmation UX (similar to `sudo` prompting on Unix systems) is well-understood. This would not break the "accept edits" workflow for routine code changes.
- **How it applies:** When Claude Code's execution engine resolves a write destination that falls within a defined set of sensitive paths, it bypasses the "accept edits" auto-approval and presents an explicit confirmation dialog with the full command, the resolved destination path, and a warning about the security implications of modifying this file.
- **Limitations:** This is a defense-in-depth measure, not a prevention measure. An attacker with the ability to influence Claude Code's suggested commands can craft a plausible explanation for why the settings file modification is legitimate. Social engineering against the confirmation prompt is possible — developers who trust their AI agent's suggestions may accept without scrutiny. Auto-accept configurations in CI/CD pipelines would bypass this gate unless the sensitive-path check is enforced at a layer that cannot be disabled by configuration.

---

### 5. Settings File Integrity Monitoring (Monitoring and Detection)

- **Type:** Monitoring and Detection
- **Plausibility:** 4/5 — Computing a cryptographic hash of `.claude/settings.json` at session end and verifying it at session start would detect unauthorized modifications between sessions. This would not prevent the initial exploit but would detect the persistence mechanism — allowing the user to identify and remove the malicious hook or MCP server entry before it causes downstream harm.
- **Practicality:** 5/5 — Hash computation on a small JSON file at session boundaries is trivial to implement. The UX (a session-startup warning if the settings hash has changed unexpectedly) is familiar from other security tools. This does not require architectural changes to the validation layer or OS-level sandboxing.
- **How it applies:** Claude Code maintains a signed baseline hash of `.claude/settings.json` stored separately from the settings file itself (e.g., in a system keychain or user-specific credentials store). On each session startup, Claude Code computes the current hash and compares it to the baseline. If the hash has changed since the last legitimate modification through Claude Code's own settings interface, it presents a visible warning with a diff of the changes and prompts the user to review and approve or revert.
- **Limitations:** Does not prevent the initial exploit — only detects post-exploitation persistence. An attacker with write access to both `.claude/settings.json` and the hash storage location could forge a matching hash. The baseline hash storage location needs to be distinct from the Claude Code-accessible filesystem to avoid this attack. Does not address shell initialization file modifications, which would require a separate monitoring mechanism.

## Strategic Council Review

### Phase 1 — Challenger

1. **The "Medium" severity label is internally inconsistent with the CVSS score and the escalation chain.** The frontmatter carries `severity: "Medium"` while the technical analysis correctly cites CVSS v4.0 7.7 HIGH. The CVSS score governs — NIST's v3.1 6.5 MEDIUM was explicitly called out as less accurate. More critically, the draft documents a direct escalation path: write to `.claude/settings.json` → set `bypassPermissions: true` → eliminate the restriction layer entirely → connects to TrustFall (CVE-2026-33068). If the severity field is meant to reflect the highest credible impact under reasonable threat assumptions, "Medium" cannot be reconciled with an escalation chain to full permission bypass. The label appears to be weighted by "no confirmed exploitation" rather than by the impact of successful exploitation, which is a different analytical question.

2. **The sed `e` modifier connection is an inference presented with more confidence than the evidence supports.** The draft states: "The most technically precise parallel, documented by Flatt Security's research on the same validation layer, is the sed `e` modifier" — then uses this to construct the technical mechanism section. But Flatt Security documents the `e` modifier as a *surviving bypass class* in related research, not as the confirmed CVE-2026-25723 technique. nil221's actual technique is unconfirmed in public sources. The technical mechanism section contains example code that is extrapolated from adjacent research, not from the reported vulnerability. This blurs the line between documented fact and analytical inference in a section that looks like primary technical documentation.

3. **The "pattern" claim with two data points is analytically premature.** The draft frames CVE-2026-25723 as evidence of a *pattern* — "blocklist repeatedly defeated." Two data points (CVE-2025-64755 and CVE-2026-25723) in the same component do not constitute a statistically defensible pattern. The 50-subcommand bypass (AAGF-2026-036) is in the same layer but is a different failure mode (threshold escape, not regex bypass) and was noted as having a working fix that was not shipped — a governance issue, not a blocklist architecture issue. Using AAGF-2026-036 to strengthen the pattern claim conflates two distinct failure categories.

4. **The settings.json → bypassPermissions → TrustFall escalation chain is analyst-constructed, not documented in the CVE advisory.** The draft presents this chain in the "Why Writing to `.claude/settings.json` Is Particularly Dangerous" section as if it is a known exploitation path. The GHSA-mhg7-666j-cqg4 advisory does not describe this chain. CVE-2026-33068 (TrustFall) is a separate CVE with its own advisory. The connection between writing settings.json via this bypass and enabling bypassPermissions as a second-stage payload is a credible threat model, but it is presented as documented fact rather than analyst hypothesis. This is the most significant evidentiary gap in the report.

5. **The "accept edits" precondition is understated as a friction point.** The draft describes "accept edits" as "a common productivity configuration" and notes that auto-accept environments require no interaction. But the CVSS v4.0 vector already accounts for this with `AT:P` (Attack Requirements: Present). The analysis does not quantify what fraction of the Claude Code install base actually runs with "accept edits" enabled. If the majority of users operate in interactive confirmation mode, the "widespread" scale rating in the `affected_parties` section is misleading — the at-risk population may be substantially smaller than the total install base.

6. **The 78-day gap framing attributes intent where incompleteness is more likely.** Characterizing the 78-day gap as evidence that "bypass-class completeness analysis was not done" assumes a single, coordinated post-patch review process. An equally plausible explanation is that CVE-2026-25723 was independently discovered by nil221 while Anthropic was working on other priorities, and the gap reflects independent research timelines rather than an internal review failure. The draft does not distinguish between "Anthropic reviewed and missed it" versus "Anthropic did not review" versus "Anthropic reviewed and judged the remaining risk acceptable." These are meaningfully different process conclusions.

### Phase 2 — Steelman

1. **The severity label inconsistency is a presentation issue, not an analytical error — and the underlying severity assessment is defensible.** The `severity: "Medium"` field in AgentFail taxonomy appears to track *realized* impact (no confirmed exploitation, auto-update delivered the patch silently) rather than *potential* impact. The CVSS 7.7 HIGH score captures potential. Both numbers are reported accurately; the apparent inconsistency reflects the taxonomy's design for tracking what actually happened versus what could have happened. The near-miss potential section exists precisely to communicate the gap. The escalation chain to TrustFall is documented in the text with the appropriate qualifier ("connects directly to") — it is not presented as confirmed exploitation, only as the logical second-stage payload.

2. **Inferring the sed `e` modifier as the mechanism is the intellectually honest approach given available sources, and the draft signals its inferential nature.** The draft explicitly states the connection is "the most technically precise parallel" and that "whether CVE-2026-25723 uses this exact variant or a different sed/echo pipeline composition... is not confirmed." This is accurate disclosure of uncertainty. The alternative — refusing to document any technical mechanism because nil221 did not publish a PoC — would make the report less useful for operators trying to understand what class of commands to audit for. Documenting the most evidentially proximate known technique as "the likely mechanism, unconfirmed" is the correct research standard.

3. **The pattern claim is defensible as a pattern of validation architecture failure, not a statistical claim about bypass frequency.** The Challenger's objection is correct that two data points do not make a statistical pattern. But the draft's pattern claim is architectural, not statistical: it argues that blocklist-based regex validation applied to shell command strings *cannot* provide the intended security property, and that this was predictable from the first bypass. This is a qualitative analytical claim about the approach, not a frequency claim. The AAGF-2026-036 evidence (three simultaneous bypass paths, a working fix not shipped) strengthens the architectural argument even if it represents a different failure mode: it demonstrates that the security review process around the same layer is systematically insufficient.

4. **The settings.json supply chain vector is the report's strongest novel contribution.** Neither the VentureBeat nor the Dark Reading coverage identifies the team-wide blast radius from a poisoned settings.json committed to source control. This is genuinely underappreciated in public coverage and is operator-actionable in a way that per-machine CVSS scoring is not. A team that learns from this report and adds code review gates on `.claude/settings.json` commits has concretely reduced their attack surface in a way that patching the CVE alone does not address.

5. **The "accept edits" precondition, while meaningful friction, does not invalidate the widespread scale rating for the exposure window.** The precondition is a prerequisite for the exploit, but it is not a small minority configuration: developer productivity optimization drives adoption of accept-edits and auto-accept modes, particularly in CI/CD pipelines and high-output development environments. More importantly, the `settings.json` supply chain vector does not require "accept edits" on the receiving developers' machines — only on the initial victim's machine. Once a poisoned settings.json is committed to a shared repository, every developer who pulls it is affected regardless of their personal configuration.

### Phase 3 — Synthesis

CVE-2026-25723 is accurately analyzed as a second-order process failure — the proximate vulnerability (blocklist regex miss) matters less than the systemic finding (two consecutive bypasses of the same layer, 78 days apart, with no evidence of bypass-class completeness analysis between them). The Challenger's most substantive objections — the severity label inconsistency, the inferential quality of the technical mechanism, and the constructed nature of the escalation chain — are real but addressable without invalidating the core analysis. The severity inconsistency between the frontmatter `severity: "Medium"` and the CVSS 7.7 HIGH body text should be reconciled with an explicit note in the Impact Assessment section explaining that the field tracks realized impact. The technical mechanism section should be prefaced with a clearer signal that the sed `e` modifier example is the best-documented adjacent technique, not the confirmed exploit technique. The settings.json → bypassPermissions → TrustFall chain should be labeled explicitly as a threat model inference, not a documented exploitation path.

The Challenger's point about the 78-day gap framing attributing intent is valid but does not materially change the operational lesson: regardless of whether Anthropic reviewed and missed it, reviewed and deprioritized it, or did not review it, the consequence for operators is the same — two bypasses of the same layer, same class, 78 days apart. The governance recommendation (bypass-class completeness analysis after any blocklist patch) stands regardless of which internal explanation is correct. The pattern claim holds as an architectural argument even if it is weaker as a statistical one — the draft should clarify this distinction.

The most significant analytical gap is the unresolved question of what CVE-2026-25723's v2.0.55 patch actually changed: if Anthropic switched the "accept edits" path to allowlist validation, the bypass-class concern is addressed and a third bypass is unlikely; if they again updated the blocklist, a third bypass is a matter of time. This is not resolvable from public sources, and the report correctly flags it as an open uncertainty. It is the key question operators should be asking Anthropic directly.

**Confidence level:** Medium — the technical mechanism is well-sourced through the vendor advisory, NVD record, and independent Flatt Security research; the specific exploit technique attributed to nil221 is inferential from adjacent research rather than a published PoC; the state of the v2.0.55 fix and the escalation chain to TrustFall are analyst-constructed and unconfirmed from primary sources. The governance finding (bypass class not closed by the predecessor patch) is the most strongly evidenced claim in the report.

**Unresolved uncertainties:**
- What is the actual CVE-2026-25723 exploit technique — sed `e` modifier, or a different piped sed/echo pipeline composition? Resolved by: nil221 publishing a technical writeup, or Anthropic publishing a post-patch technical disclosure.
- Did v2.0.55 switch the "accept edits" validation path to allowlist-based validation, or update the blocklist again? Resolved by: inspection of the v2.0.55 source diff in `bashPermissions.ts` (open source repository).
- Is the settings.json → `bypassPermissions: true` → TrustFall escalation chain a realistic second-stage attack or a theoretical construct? Resolved by: confirming whether `bypassPermissions` can be set via `settings.json` without Claude Code's own UI intervention, and whether that flag is respected across version updates.
- What fraction of the Claude Code install base ran with "accept edits" or auto-accept enabled during the 78-day window? Resolved by: Anthropic publishing telemetry on feature adoption, or independent survey data on Claude Code configuration patterns.
- Did any in-the-wild exploitation occur during the 78-day exposure window? Likely unresolvable: hook installation in settings.json leaves no forensic footprint distinguishable from legitimate configuration unless the file was actively monitored before and after.

## Key Takeaways

1. **Patching one bypass of a blocklist does not close the bypass class — demand bypass-class analysis.** When a security fix closes a specific variant of an input validation bypass (sed `w` command → sed `e` modifier), the correct response is enumeration of the bypass class, not validation of the specific variant. Operators deploying AI coding tools should ask vendors not just "was this CVE patched?" but "was the bypass class evaluated for completeness, and what architectural change prevents the next variant?" The 78-day gap between CVE-2025-64755 and CVE-2026-25723 is the cost of not asking this question.

2. **`.claude/settings.json` is a high-value attack target that teams should actively monitor.** The settings file stores hooks executed on every session startup and MCP server configurations loaded before user interaction begins. Unauthorized modification converts a transient exploit into persistent access that survives version upgrades. Teams sharing `.claude/settings.json` via source control should treat changes to this file as security-sensitive — require code review on all modifications, verify the file's integrity on each developer machine, and consider whether committing it to shared repositories is appropriate for their threat model.

3. **AI coding agents hold live developer credentials by design — the goal of exploiting them is usually the credential, not the agent.** Claude Code sessions routinely include `ANTHROPIC_API_KEY`, cloud provider CLI credentials, and SSH agent access in their execution environment. An attacker who achieves code execution in a Claude Code session (via this vulnerability or any other) does not need to compromise Anthropic's model to cause material harm — they need only to harvest the credentials in scope. Operators should apply the same credential hygiene to Claude Code's execution environment that they apply to CI/CD systems: scoped, short-lived credentials; environment variable isolation; no long-lived API keys with broad permissions in the agent's shell environment.

4. **Auto-update is a containment mechanism, not a prevention mechanism — pinned versions create silent exposure gaps.** Anthropic's auto-update delivered the v2.0.55 patch silently to most users, making it the most effective containment measure available. But teams running Claude Code in CI/CD pipelines, Docker containers, or reproducible build environments frequently pin npm package versions to ensure deterministic builds. Pinned versions below 2.0.55 remained exposed after February 6, 2026 with no in-product alert. Vulnerability management programs for AI developer tools should track exposure windows for CVEs in pinned dependencies the same way they track CVEs in traditional software dependencies.

5. **Three simultaneous bypass paths in the same layer is a pattern signal, not a coincidence.** CVE-2026-25723 (piped sed/echo), CVE-2026-25724 (symlink following), and the AAGF-2026-036 50-subcommand threshold bypass were all active concurrently in Claude Code's permission enforcement layer. Multiple simultaneous bypass paths in the same component indicate insufficient adversarial testing coverage, not isolated implementation mistakes. Teams evaluating AI coding tools for high-sensitivity deployments should treat concurrent CVEs in a single component as a signal to audit the component's entire validation architecture, not just the reported vulnerabilities.
