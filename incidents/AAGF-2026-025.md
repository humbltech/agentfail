# Three Chained CVEs in Anthropic's Own git MCP Server Enable Full RCE via a Poisoned README

## Executive Summary

Anthropic's own reference implementation of a git Model Context Protocol server — the canonical example developers follow when building MCP git integrations — contained three input validation vulnerabilities that chain into a complete remote code execution path. The attack requires no user clicks after initial configuration: an attacker embeds malicious instructions in a README, GitHub issue, or any document the AI agent will read, and the LLM autonomously executes a six-step sequence combining path traversal, argument injection, and git filter abuse to run arbitrary code with the developer's OS privileges. The vulnerable configuration is not a misconfiguration — it is the exact setup shown in Anthropic's official quickstart documentation, which deploys both the Git MCP server and Filesystem MCP server together, the precise combination that makes the full RCE chain possible.

## What Happened

### What mcp-server-git Is — and Why Reference Status Matters

`mcp-server-git` is Anthropic's official reference implementation of a Model Context Protocol server for git operations. It provides AI agents with tools to perform git actions — reading diffs, checking out branches, initializing repositories, examining history — through MCP's structured tool-calling interface.

Reference implementations carry disproportionate security risk. When a company publishes a reference implementation, developers treat it as the canonical pattern. They study its structure, copy its architecture, and build derivatives. Security flaws in reference implementations do not stay contained within the reference — they propagate to every downstream project built from it, creating a supply chain effect without a traditional supply chain attack. Anthropic's own documentation and tutorials pointed to this code as the example to follow.

The server is used by Claude Desktop, Cursor, and any other MCP-compatible AI coding assistant when configured to provide git capabilities to the agent. As of June 2025, the official quickstart documentation showed a configuration deploying both the Git MCP server and the Filesystem MCP server simultaneously — the exact combination that enables the full RCE chain.

### The Three Vulnerabilities Discovered

Yarden Porat of Cyata Security identified three discrete input validation failures in the codebase and reported all three to Anthropic via HackerOne in June 2025:

**CVE-2025-68143** allowed the `git_init` tool to accept any filesystem path without validation, invoking `git init` on arbitrary directories. There was no boundary check, no allowlist of permitted paths, no restriction to a working directory. An AI agent could be instructed to call `git_init` on `~/.ssh`, `/etc`, or any application data directory — converting it into a git repository and enabling git hook mechanisms.

**CVE-2025-68145** exposed a fundamental security boundary bypass. The `--repository` flag was the server's primary administrative control: operators could configure the server to restrict operations to a single designated directory. However, the flag was validated only at server startup — not at each individual tool invocation. Once the server was running, every subsequent tool call accepted an arbitrary `repo_path` argument without checking it against the configured boundary. This is a classic "validate at registration, not at use" TOCTOU pattern. The security boundary existed on paper but enforced nothing in practice.

**CVE-2025-68144** allowed argument injection through the `git_diff` and `git_checkout` tools. Both passed user-controlled arguments directly to the git CLI without sanitization. The critical payload: passing `--output=/path/to/file` as the `target` argument to `git_diff` caused git to write diff output to an arbitrary filesystem path — an arbitrary file write primitive. Similar injection surfaces existed in `git_checkout`'s branch and ref arguments.

### How the Attack Works

An attacker plants malicious instructions in content the developer's AI agent will read: a README in a repository the agent is working with, a GitHub issue, a project wiki page, a compromised webpage fetched during research. The developer never manually executes anything. The LLM reads the content, interprets the embedded instructions as legitimate directives, and autonomously calls MCP tools with attacker-controlled arguments.

The six-step chain that follows requires both the Git MCP server and the Filesystem MCP server to be active — which is the configuration Anthropic's quickstart documentation showed as standard:

1. The agent is instructed to call git tools with a `repo_path` that falls outside the configured `--repository` scope (CVE-2025-68145 boundary bypass).
2. The agent calls `git_init` on a chosen writable directory — `/tmp/exploit` or `~/.ssh` — making it a git repository (CVE-2025-68143).
3. The agent uses the Filesystem MCP server to write a malicious shell script into the new repository directory.
4. The agent uses the Filesystem MCP server and CVE-2025-68144 to write a malicious `.git/config` that configures a git smudge/clean filter pointing to the payload script, plus a `.gitattributes` file applying the filter to tracked files.
5. The agent uses the Filesystem MCP server to create a trigger file matching the `.gitattributes` filter pattern.
6. The agent calls `git_add` on the repository (via CVE-2025-68144). Git processes the trigger file through the configured clean filter. The filter executes the payload script. RCE achieved.

The payload runs with the OS privileges of the MCP server process — typically the developer's own user account when running locally with Claude Desktop or Cursor. SSH keys, `.env` files, AWS credentials, browser profiles, source code, and any other resource accessible to the user account are within reach.

Yarden Porat summarized the core insight: "Each MCP server might look safe in isolation, but combine two of them — Git and Filesystem in this case — and you get a toxic combination."

### Vendor Response and Patching

Anthropic patched in two stages. The September 2025 partial patch (v2025.9.25) addressed only CVE-2025-68143. The full December 2025 patch (v2025.12.18) resolved all three CVEs. The fix for CVE-2025-68143 was architectural: the `git_init` tool was removed entirely, reflecting a judgment that repository initialization is out of scope for a git operations server. The fixes for CVE-2025-68144 and CVE-2025-68145 added input validation — rejecting arguments that begin with hyphens and adding path resolution at tool invocation time.

Anthropic published no proactive security bulletin, wrote no blog post, and did not respond to The Register's media inquiry. The CVEs became public through GitHub Security Advisories on the `modelcontextprotocol/servers` repository — a standard mechanism but not a proactive communication channel. The key failure was not media relations but operational reach: the affected developers were precisely those who had followed Anthropic's quickstart documentation, and CVE publication through GitHub Advisories does not reach that population reliably.

## Technical Analysis

### CVE-2025-68143: Unrestricted git_init (Path Traversal)

- **GitHub Advisory:** GHSA-5cgr-j3jf-jw3v
- **CWE:** CWE-22 — Improper Limitation of Pathname to Restricted Directory
- **CVSS v3.1:** 8.8 (High); CVSS 4.0: 6.5 (Medium)

The `git_init` tool accepted any filesystem path and passed it directly to `git init` without validation. No boundary was enforced: no check against a working directory, no allowlist of permitted paths, no restriction to non-sensitive locations.

The attack surface is the entire filesystem writable by the MCP server process. Converting `~/.ssh` into a git repository, for example, enables subsequent manipulation of git hooks and config files within that directory — mechanisms that execute arbitrary code when triggered by git operations. Converting `/etc` or application configuration directories similarly weaponizes git's hook system against directories that were never intended as git repositories.

The fix was architectural rather than defensive: Anthropic removed the `git_init` tool entirely in v2025.12.18, acknowledging that repository initialization is outside the appropriate scope of a git operations server. The September 2025 partial patch began this removal.

### CVE-2025-68145: Repository Scope Bypass (--repository Flag Bypass)

- **GitHub Advisory:** GHSA-j22h-9j4x-23w5
- **CWE:** CWE-22 — Path Traversal
- **CVSS v3.1:** 7.1 (High); CVSS 4.0: 6.3 (Medium)

The `--repository` flag was designed as the primary administrative security boundary for the server. Operators could configure the server to restrict all git operations to a single designated directory, preventing the agent from reaching outside a defined workspace.

The flaw: the flag was validated at server startup but not at each tool call. Once the server initialized successfully with a valid repository path, subsequent tool invocations accepted arbitrary `repo_path` arguments. The boundary was never checked again at point-of-use.

This is a well-understood class of vulnerability: time-of-check vs. time-of-use (TOCTOU) validation. The security boundary was real enough to mislead operators into believing they had constrained the server's reach — but the constraint was applied once and never enforced at the moment it mattered.

The fix adds path resolution at tool invocation time, verifying that each requested path is a legitimate subpath of the configured repository before proceeding.

### CVE-2025-68144: Argument Injection in git_diff / git_checkout

- **GitHub Advisory:** GHSA-9xwc-hfwc-8w59
- **CWE:** CWE-88 — Improper Neutralization of Argument Delimiters in a Command
- **CVSS v3.1:** 8.1 (High); CVSS 4.0: 6.3–6.4 (Medium)

The `git_diff` and `git_checkout` tools passed user-controlled arguments directly to the git CLI without sanitization. Git's own argument parsing supports flags that modify output behavior — flags that begin with a hyphen and have no inherent meaning to the MCP tool layer, but carry significant semantics at the git CLI layer.

The critical exploit: the `target` argument to `git_diff` accepts `--output=/arbitrary/path`. Git interprets this as a directive to write diff output to that path rather than stdout. The MCP tool has now become an arbitrary file write primitive — one that can create or overwrite any file accessible to the server process.

Combined with the Filesystem MCP server's legitimate write capabilities, this provides redundant paths to write attacker-controlled content to arbitrary filesystem locations.

The fix validates that arguments do not begin with a hyphen character (rejecting flag-like values) and validates through `rev_parse` to ensure arguments are legitimate git references.

### The 6-Step RCE Chain: Why Combining Servers Creates the "Toxic Combination"

The chain's novelty is its cross-server nature. Neither the Git MCP server nor the Filesystem MCP server alone provides a complete RCE path under normal conditions. The Filesystem MCP server has legitimate write capabilities but no git execution mechanism. The Git MCP server has git execution capabilities but (under the unpatched code) does not have its own arbitrary write path without the argument injection.

Together, they provide complementary attack capabilities that fill each other's gaps:

| Step | Server Used | CVE Exploited | What Happens |
|------|------------|---------------|-------------|
| 1 | Git MCP | CVE-2025-68145 | Scope boundary bypassed; attacker controls target directory |
| 2 | Git MCP | CVE-2025-68143 | `git_init` on attacker-chosen directory (e.g., `/tmp/exploit`) |
| 3 | Filesystem MCP | — | Payload script written to new repo directory |
| 4 | Filesystem MCP + Git MCP | CVE-2025-68144 | Malicious `.git/config` and `.gitattributes` written; clean filter configured to point to payload |
| 5 | Filesystem MCP | — | Trigger file created matching filter pattern |
| 6 | Git MCP | CVE-2025-68144 | `git_add` triggers filter; filter executes payload; RCE |

The git clean/smudge filter mechanism is a legitimate git feature for transforming file content during add/checkout operations. It is not a vulnerability in git itself — it is a designed execution mechanism that the attack chain repurposes. Once the filter is configured and a matching file is added via `git add`, git executes the configured filter binary as a standard part of its operation.

### Indirect Prompt Injection as Delivery Mechanism

The attack does not require the developer to manually call any vulnerable tool. The entire six-step sequence is triggered by the LLM reading attacker-controlled content and interpreting embedded instructions as legitimate directives.

This is indirect prompt injection — the attacker's payload arrives through content the agent processes (a README, an issue, a webpage), not through direct interaction with the user or the agent. The developer's interaction is limited to having configured their AI coding assistant with the Git and Filesystem MCP servers, which is the standard quickstart configuration.

Once the poisoned content is read, the attack is self-executing from the agent's perspective. No user clicks are required. No suspicious prompts appear. The agent executes what it interprets as reasonable repository operations in service of the developer's apparent intent.

## Root Cause Analysis

**5 Whys:**

**Why 1:** Why could the 6-step RCE chain be executed?
Because three input validation failures in `mcp-server-git` could be combined with the Filesystem MCP server's legitimate capabilities to construct a complete attack chain — and indirect prompt injection provided an automated delivery mechanism requiring no user interaction.

**Why 2:** Why did the input validation failures exist?
`mcp-server-git` treated its tools as data processors — functions that accept arguments, invoke git operations, and return results — without applying a security boundary model to its inputs. No argument validation framework was enforced, despite MCP tool inputs originating from LLM output that can be influenced by untrusted content.

**Why 3:** Why was there no security boundary model applied to tool inputs?
The reference implementation was designed and documented without an explicit threat model for indirect prompt injection as a delivery mechanism. The assumption embedded in the design was that tool arguments would come from trusted sources (the user, the LLM following user intent) — not from adversarially crafted content the LLM had processed.

**Why 4:** Why was the threat model for prompt-injection-driven tool abuse absent from a first-party Anthropic reference implementation?
At the time of `mcp-server-git`'s creation, the MCP ecosystem was new and moving rapidly. Secure-by-default patterns for MCP tool implementations — input validation standards, multi-server trust isolation models, argument sanitization requirements — had not yet been established, documented, or enforced even by Anthropic's own teams. The reference implementation was built with functional correctness as the primary goal.

**Why 5 / Root cause:** Why were secure-by-default standards not established before publishing a reference implementation that would propagate to the broader developer ecosystem?
The fundamental gap: MCP tool inputs were not treated as an untrusted attack surface. MCP servers were architected as if their inputs would always reflect legitimate user intent, rather than as if their inputs might be the product of an LLM that has read adversarially crafted content. When an AI agent operates on real-world content — repositories, documents, webpages — any of that content is a potential injection vector. A reference implementation must model this threat, because every derivative project inherits the threat model from the reference. The institutional failure — no mandatory pre-publication security review gate, no formal threat model requirement — sits one level above this and is the systemic root cause.

**Root cause summary:** MCP server tools were treated as data processors rather than security boundaries, with no input validation framework enforced in the reference implementation, despite the fundamental property that MCP tool inputs originate from LLM output that can be influenced by untrusted content. The institutional root cause: no pre-publication security review gate existed to catch these failures before the reference code propagated to the ecosystem.

## Impact Assessment

**Severity:** High

**Actual vs. potential:** Near-miss. No confirmed in-the-wild exploitation was reported prior to coordinated disclosure, and no exploitation was confirmed afterward. However, the full attack chain was demonstrated, the vulnerability window spanned approximately 7 months (June to December 2025), and the default quickstart configuration created the vulnerable setup for any developer following official documentation. The 7-month window on developer machines — high-value targets for quiet credential theft — means "no confirmed exploitation" reflects public reporting gaps rather than verified forensic safety.

**Who was at risk:**
- All users of `mcp-server-git` who followed Anthropic's official quickstart documentation (which deployed both Git MCP and Filesystem MCP servers)
- All developers who built derivative projects from the reference implementation before December 2025
- Claude Desktop, Cursor, and other MCP-compatible AI coding assistant users with the standard configuration

**What was at risk:**
- SSH private keys (`~/.ssh/`) — immediate credential for all git hosts and SSH-accessible servers
- AWS credentials and other cloud provider credential files
- `.env` files across projects — application secrets, database credentials, API keys
- Full source code of projects on the developer's machine
- Persistent backdoor installation via shell profile modification, git hook injection, or crontab addition

**Reference implementation amplification:** The blast radius of a security flaw in a reference implementation is not confined to users of that specific artifact. It extends to every project, team, and organization that built from it as a template. Developers who copied `mcp-server-git`'s patterns for their own MCP servers may have reproduced the same validation failures in code that was never patched by Anthropic's December 2025 update. The actual exposure surface is unknowable from the outside — it depends on how many derivative projects exist, whether they copied validation logic verbatim, and whether they received the December patch through version updates rather than manual porting.

**Scope boundary note:** The attack requires both Git MCP and Filesystem MCP servers running simultaneously. Deployments using only the Git MCP server without the Filesystem MCP server could not execute the full 6-step chain, though the scope bypass (CVE-2025-68145) and argument injection file write (CVE-2025-68144) remained exploitable independently.

## How It Could Have Been Prevented

**1. Tool argument validation as baseline.**
Every MCP tool that accepts user-supplied arguments should validate those arguments before passing them to underlying system calls. The minimum: reject arguments that begin with hyphens (as the final patch did for CVE-2025-68144), verify that path arguments fall within expected boundaries (as the final patch did for CVE-2025-68145), and validate arguments against an allowlist of expected values where the value space is bounded. These are not novel controls — they are standard input validation practices that apply to every CLI-wrapping tool.

**2. Path traversal prevention as standard.**
Any tool that accepts a filesystem path as an argument must validate that path against an expected boundary before use. The `--repository` flag represented the right design intent but the wrong implementation: the path should have been stored, and every tool invocation should have verified the provided path against the stored boundary. A utility function applied consistently across all path-accepting tools would have prevented both CVE-2025-68143 and CVE-2025-68145.

**3. Multi-server trust isolation modeled at design time.**
The "toxic combination" emerges because two servers with individually bounded capabilities can be composed into a more powerful attack surface. MCP server design should explicitly consider what additional capabilities become available when the server is deployed alongside other common servers. Reference implementations should document known multi-server interaction risks and recommend isolation patterns (separate process boundaries, independent permission scopes, minimal cross-server trust).

**4. Threat model for indirect prompt injection in reference implementations.**
Anthropic's reference implementation should have been designed with an explicit threat model: any content the AI agent reads is a potential injection vector. The design question is not "what does a legitimate user ask this tool to do?" but "what can an adversary cause the LLM to ask this tool to do after reading a poisoned document?" Applying this threat model consistently would have surfaced the path traversal and argument injection risks before publication.

**5. Architectural review before publishing as a reference.**
Reference implementations require a higher security bar than internal tools because they propagate to the ecosystem. A security review specifically targeting injection surfaces, path traversal, and multi-server interaction risks — conducted before the repository was published as a reference — would have caught all three CVEs. This review should be mandatory for any first-party Anthropic code designated as reference material.

## How It Was / Could Be Fixed

**Actual remediation taken:**

*CVE-2025-68143 (git_init path traversal):*
- v2025.9.25 (September 2025): Partial fix — initial restriction on git_init
- v2025.12.18 (December 2025): Full fix — git_init tool removed entirely. Anthropic's architectural judgment: repository initialization is out of scope for a git operations server and should not be a tool available to the agent. This is the correct decision — the tool's existence created a risk surface with no commensurate value for a server designed to operate on existing repositories.

*CVE-2025-68145 (repository scope bypass):*
- v2025.12.18: Adds path resolution at each tool invocation, not just at startup. The requested path is verified to be a legitimate subpath of the configured repository boundary before the operation proceeds. Every tool call now re-validates the path.

*CVE-2025-68144 (argument injection in git_diff / git_checkout):*
- v2025.12.18: Validates that arguments do not begin with a hyphen character, rejecting flag-like values that could alter git's behavior. Validates arguments via `rev_parse` to confirm they are legitimate git references before use.

**What was not done:**
- Anthropic issued no proactive security bulletin or blog post
- Anthropic did not respond to The Register's media inquiry
- No communication was directed to developers who had built derivative projects from the reference implementation
- The quickstart documentation was not immediately updated with a security advisory — the specific failure here is that the documentation created the vulnerable configuration, and the patch cannot reach users who are not monitoring GitHub Security Advisories

**Recommended additional measures:**
- Establish an MCP server input validation standard and publish it as a companion to MCP server development documentation, so derivative projects implement validation from the start
- Update quickstart documentation to warn about multi-server trust surfaces and document the security implications of deploying Git MCP and Filesystem MCP together
- Conduct a security review of all other MCP reference implementations for analogous validation failures
- Implement a coordinated outreach mechanism to notify developers who forked or cloned the reference implementation

## Solutions Analysis

### 1. Tool Argument Validation and Sanitization

- **Type:** Input Validation / Secure Defaults
- **Plausibility:** 5/5 — Technically straightforward. The final patch demonstrates the approach: check that path arguments fall within expected boundaries; reject arguments that begin with hyphens; validate git references via `rev_parse`. These are one-to-a-few line additions per tool function.
- **Practicality:** 5/5 — No architectural changes required. Backward compatible for legitimate use cases. Can be retrofitted into existing tools with minimal surface area risk.
- **How it applies:** Directly prevents CVE-2025-68144 (argument injection) and CVE-2025-68145 (scope bypass). Does not address the multi-server composition risk, but eliminates the injection primitives needed for the chain.
- **Limitations:** Requires consistent application across every tool that accepts external arguments — a validation framework that is applied once and forgotten will create gaps as new tools are added. The real protection comes from making validation a required step in the tool implementation pattern, not an optional addition.

### 2. Permission Scoping (Validate-at-Use, Not Validate-at-Registration)

- **Type:** Permission Scoping / Least Privilege
- **Plausibility:** 5/5 — The correct pattern is known and the fix implements it: store the configured boundary at startup, re-validate the provided path against that boundary at every tool invocation.
- **Practicality:** 5/5 — Minimal implementation complexity. The pattern is a guard clause at the top of each path-accepting tool function.
- **How it applies:** Directly prevents CVE-2025-68145. Ensures that the `--repository` flag actually enforces the intended boundary rather than providing a false sense of security.
- **Limitations:** Scope validation only constrains where the tool operates — it does not prevent argument injection within the permitted scope. Must be combined with argument sanitization for complete protection.

### 3. Architectural Redesign: Scope and Tool Selection Review

- **Type:** Architectural Redesign
- **Plausibility:** 4/5 — Removing tools that fall outside the server's appropriate scope is architecturally correct. The `git_init` fix demonstrates this: rather than adding validation to an out-of-scope tool, Anthropic removed it entirely. The right question at design time is "what should this server be able to do?" rather than "how do we validate every possible input to this tool?"
- **Practicality:** 4/5 — Requires upfront judgment about server scope and willingness to remove capabilities that create risk without commensurate value. Breaking change for any derivative project that relied on the removed tool. Requires documentation of the rationale so developers building from the reference understand the scope decision.
- **How it applies:** Eliminates CVE-2025-68143 by removing the tool that caused it. More broadly: a server with a minimal, well-scoped tool set has a smaller attack surface than a server with broad capabilities.
- **Limitations:** Does not prevent injection attacks against remaining tools. Scope definition is a judgment call that can be contested. Removing capabilities that some legitimate users relied on creates friction.

### 4. Multi-Server Isolation and Trust Modeling

- **Type:** Architectural Redesign / Defense in Depth
- **Plausibility:** 4/5 — The "toxic combination" insight is well-defined. The countermeasure — explicit isolation between MCP servers so that one server cannot be used to amplify the attack surface of another — is achievable. Implementation approaches include separate permission contexts per server, cross-server operation logging and anomaly detection, and documentation of known dangerous server combinations.
- **Practicality:** 3/5 — Requires MCP platform-level support for server isolation primitives, which do not currently exist in the MCP specification. At minimum, requires documentation and developer guidance. Full isolation would require a protocol change. The protocol governance dependency means this has no clear owner and no committed timeline.
- **How it applies:** Addresses the root cause of why the six-step chain is possible: two individually-bounded servers compose into a more powerful attack surface. If Filesystem MCP operations in service of a Git MCP chain were detectable or preventable, the chain would break.
- **Limitations:** Defining "dangerous combinations" is difficult without enumerating all possible server pairs. Legitimate workflows often require cross-server operations. Isolation mechanisms could break useful agent behaviors if applied too broadly.

### 5. Monitoring and Anomaly Detection for MCP Tool Invocations

- **Type:** Monitoring / Detection
- **Plausibility:** 3/5 — Logging all MCP tool invocations with their arguments and flagging anomalous patterns (paths outside expected boundaries, hyphen-prefixed arguments, git operations on non-repository directories) is technically feasible with existing tooling.
- **Practicality:** 3/5 — Requires instrumentation of the MCP server layer, which most deployments do not currently have. Establishes a detection capability but not a prevention capability — anomalies are detected after the fact.
- **How it applies:** Would have surfaced exploitation attempts during the vulnerability window if deployed. Provides early warning for novel attack chains not yet covered by validation rules.
- **Limitations:** Detection alone does not prevent the attack. Log analysis and alerting infrastructure must be in place and actively monitored. An attacker who understands the detection rules can attempt to craft chains that evade anomaly thresholds. Best deployed as a complement to prevention controls, not a substitute.

## Key Takeaways

**1. Individual server safety does not guarantee combined safety.**
Each MCP server can appear safe in isolation while creating a dangerous attack surface in combination with other servers. The six-step chain in this incident requires capabilities from two servers; neither server alone provides a complete RCE path. This "toxic combination" pattern is a new class of agentic security risk. Developers and security teams must analyze MCP server deployments as systems — asking what becomes possible when server A and server B are deployed together — not just as individual tools.

**2. Reference implementations propagate security flaws into the ecosystem.**
A security flaw in Anthropic's own example code is not confined to Anthropic's own deployments. Every developer who built from `mcp-server-git` as a template may have reproduced the same validation failures in code that was never updated by Anthropic's patch. The effective blast radius of a reference implementation vulnerability is unknowable and potentially large. Reference implementations must be held to a higher security standard than internal tools, because they set the pattern the ecosystem follows.

**3. Following official documentation can create the vulnerable configuration.**
The six-step chain requires both Git MCP and Filesystem MCP servers deployed together. That is the configuration shown in Anthropic's own quickstart documentation. Security vulnerabilities in official quickstart configurations are particularly damaging: they affect the most trusting and least cautious users — developers who are doing exactly what they were told to do.

**4. Indirect prompt injection makes all agent-readable content a potential attack surface.**
The developer never executes any malicious code. They configure their AI agent with standard MCP servers and go about their work. The attack is triggered when the agent reads a poisoned README, issue, or document. Every piece of content an AI agent can read — every repository, every linked page, every document in the agent's working context — is now a potential initial access vector if the agent has access to vulnerable tools. Security reviews of agentic systems must model this threat explicitly.

**5. "Remove the tool" is sometimes the correct security fix.**
Anthropic's decision to remove `git_init` entirely rather than add validation reflects a correct architectural judgment: the tool was out of scope for a git operations server, and its existence created risk without commensurate value. When a capability does not belong in a server's design, the right answer is to remove it — not to add increasingly complex validation that developers must maintain forever. Scope discipline at design time is cheaper than security maintenance over the lifetime of the tool.

## Strategic Council Review

### Phase 1 — Challenger

**1. The "root cause" is misidentified — it is a symptom, not a cause.**
The report's 5 Whys concludes: "MCP server tools were treated as data processors rather than security boundaries." This is a description of the implementation decision, not an explanation of why that decision was made or persisted. The actual root cause candidates — organizational incentives that prioritized developer experience and adoption speed over security rigor, absence of a mandated security review gate before publishing reference code, and the absence of any formal threat model process for the MCP ecosystem — are mentioned in passing ("moving rapidly," "functional correctness as the primary goal") but never elevated as root cause. The 5 Whys chain stops one level too early at the institutional level, where the real leverage for prevention lives. A future MCP server team reading this report learns "validate your inputs" — they do not learn "require a pre-publication threat model review with explicit sign-off."

**2. "Near-miss" framing may understate actual exposure, not overstate it.**
The report calls this a near-miss because there is "no confirmed in-the-wild exploitation." But the evidentiary standard for that claim is weak: absence of confirmed exploitation after a 7-month unpatched window — during which the attack was both technically feasible and triggered by reading a file — is not strong evidence that exploitation did not occur. Sophisticated attackers targeting developer machines do not announce themselves, and credential theft via git hook execution leaves minimal forensic artifacts on a developer's local machine. The report should either acknowledge the limits of "no confirmed exploitation" as an evidentiary claim, or recalibrate the near-miss framing to note that it reflects an absence of public incident reports rather than a verified absence of exploitation.

**3. The "toxic combination" claim is analytically sound but understates a confounding factor: the chain has multiple points of failure.**
The report presents the 6-step chain as near-deterministic, but Step 6 — triggering a git clean filter via `git_add` — requires the agent to call `git_add` on a repository it has no legitimate reason to interact with. This is the most auditable step in the chain, because a real AI coding assistant running in a developer's session would plausibly generate a log entry or visible action when calling `git_add` on `/tmp/exploit`. The analysis does not engage with whether the chain would survive scrutiny by an attentive developer or whether real agents would call `git_add` without presenting confirmation context to the user.

**4. The "reference implementation propagation" claim lacks quantification and may be overstated.**
The report asserts that security flaws propagate to "every developer who built from it" and creates "a supply chain effect without a traditional supply chain attack." This is plausible but presented as fact rather than estimate. The actual propagation depends on: whether developers copied mcp-server-git's validation logic specifically, whether their forks received dependency updates, and whether they built custom tools (which would copy the pattern) vs. using the package directly (which would be patched by Anthropic's update). The "unknowable blast radius" framing is appropriate — but should acknowledge what factors determine whether propagation was narrow or wide.

**5. Anthropic's media silence is framed as a pattern but may reflect standard disclosure norms.**
For a coordinated disclosure involving three CVEs in a June-to-December patch cycle, not issuing a proactive public blog post and routing disclosure through GitHub Security Advisories is within the range of normal vendor behavior. The comparison to AAGF-2026-022's "non-response pattern" requires both incidents to be aberrant; if the behavior is consistent, it may simply be Anthropic's disclosure policy rather than negligence. The report should distinguish between "Anthropic's disclosure policy falls below best practice for quickstart-originated vulnerabilities" and "Anthropic went silent on a crisis."

**6. The solutions are technically correct but one is rated implausibly optimistic.**
Solution 4 (Multi-server isolation) is rated 4/5 plausibility but 3/5 practicality, noting it "requires MCP platform-level support for server isolation primitives, which do not currently exist in the MCP specification." The report then recommends that documentation warn about multi-server trust surfaces, but this recommendation requires developers to read and act on documentation — a mechanism with a poor track record in security. The solution analysis does not account for the fact that the highest-value fix (isolation primitives in the MCP spec) has no clear owner, no committed timeline, and depends on protocol governance decisions outside any single organization's control.

### Phase 2 — Steelman

**1. The CVSS 8.8 severity rating and High classification are fully justified by the attack chain's demonstrated properties.**
CVE-2025-68143 alone carries CVSS 8.8, which reflects the unbounded path traversal to any filesystem location writable by the process. But the combined chain's effective severity is higher than any individual CVE score suggests, because CVSS scores are computed per-vulnerability, not per-chain. The combined chain achieves: arbitrary code execution, at the OS privilege level of a developer account, triggered by reading a file (zero post-configuration user interaction), with persistent effects that survive the agent session. A chain that achieves all four of these properties simultaneously warrants High regardless of whether exploitation was confirmed.

**2. The "toxic combination" insight is the most analytically novel and defensible contribution in the report.**
The core claim — that individual server safety does not guarantee combined safety — is not intuitive and is not yet established doctrine in MCP security guidance. The report correctly identifies that the Filesystem MCP server provides complementary capabilities that fill the gaps in the Git MCP server's attack surface, and vice versa. Neither server alone provides a complete RCE path; together they do. This compositional vulnerability class is distinct from: (a) a single-server vulnerability, (b) a supply chain attack, and (c) a protocol-level vulnerability affecting all MCP servers. It is a new category — multi-server capability composition creating an emergent attack surface — and naming it precisely is the right contribution at this stage of the MCP ecosystem's maturity.

**3. The reference implementation propagation risk, while unquantified, is structurally real.**
What is structurally true, and supported by general software engineering observation, is that reference implementations are copied at higher fidelity than library dependencies: developers who use a package receive automatic patches via version updates, while developers who copied the validation pattern (or its absence) verbatim into their own MCP server must patch manually. The propagation risk is real at the architectural level, and the report's recommendation — proactive outreach to forkers and cloners, documentation of the validation pattern as mandatory — is the correct response to a real risk even under uncertainty about its magnitude.

**4. The 7-month patch timeline is genuinely problematic for a first-party reference implementation.**
A 7-month timeline to full resolution for a CVSS 8.8 vulnerability in Anthropic's own reference implementation — not a dependency, not a third-party library, but code owned and published by Anthropic — is not defensible as normal. The partial September patch (addressing only CVE-2025-68143 while leaving 68144 and 68145 unpatched for an additional 3 months) is particularly notable: the partial patch may have given operators false confidence that the server was secured when the scope bypass and argument injection remained fully exploitable.

**5. Anthropic's media silence is meaningful specifically because the vulnerable configuration is in quickstart documentation.**
The challenger correctly notes that routing disclosure through GitHub Security Advisories is normal vendor behavior. But the specific aggravating factor here is that the vulnerable configuration is in Anthropic's own quickstart documentation. A company that publishes quickstart documentation creating a vulnerable default configuration has an affirmative obligation to update that documentation and proactively notify developers who followed it — an obligation that routing CVEs through GitHub Advisories does not fulfill. The "no proactive security bulletin" criticism is about the failure to reach the affected population through the same channel that created the exposure.

### Phase 3 — Synthesis

The three-phase review reveals a report that is analytically sound in its core technical claims and novel in its "toxic combination" framing, but carries two structural weaknesses that should be acknowledged in the text rather than left for readers to discover. The first is the near-miss evidentiary limit: the report's confidence that no exploitation occurred rests on an absence of public reports, not on a verified forensic absence. This matters because a 7-month unpatched window on developer machines is exactly the scenario where sophisticated, quiet exploitation would be hardest to detect. The report acknowledges this explicitly in the near-miss intervention note and researcher_notes. The second is the 5 Whys chain stopping one institutional level too early — the published root cause tells engineers what the problem was; an institutional layer (no pre-publication security review gate) would tell organizations what process failure allowed it to reach production. This has been added to Why 5 in the root cause section.

The "toxic combination" and reference implementation propagation claims survive the challenge phase substantially intact. The toxic combination insight is analytically correct and currently underdocumented in MCP security guidance. The Anthropic media silence criticism is the only claim requiring modest recalibration: the specific charge is well-supported (no proactive outreach to developers who followed the vulnerable quickstart documentation); the broader implication (general crisis silence) is inferential and is appropriately bounded in the text.

**Council Confidence:** Medium-High

The technical analysis is strong and the core novel claims (toxic combination, reference implementation propagation) are well-reasoned. Confidence is Medium-High rather than High because two material uncertainties remain: the actual exploitation status cannot be verified from public sources alone, and the propagation magnitude of the reference implementation vulnerability is genuinely unknowable without repository fork/dependency data that is not publicly available.

**Unresolved Uncertainties:**

- **Actual exploitation status during the 7-month window** — would be resolved by: forensic telemetry from Anthropic's HackerOne program, any confirmed incident reports from organizations using Claude Desktop or Cursor with default configuration, or a post-disclosure threat intelligence report from a firm with visibility into credential theft campaigns targeting developers.
- **Magnitude of reference implementation propagation** — would be resolved by: a GitHub dependency graph analysis of projects that imported or forked `mcp-server-git` before December 2025, combined with a sampling of whether those forks preserved the unpatched validation logic.
- **Whether the 6-step chain executes autonomously in real agent sessions** — would be resolved by: a controlled reproduction of the chain in Claude Desktop and Cursor with default configurations, documenting whether any agent UI checkpoint or confirmation prompt interrupts the chain before `git_add` is called on the attacker-constructed repository.
- **Scope and completeness of Anthropic's quickstart documentation update post-patch** — would be resolved by: reviewing the commit history of Anthropic's official MCP documentation repository to confirm whether and when the quickstart was updated to remove or warn about the Git + Filesystem dual-server configuration.
