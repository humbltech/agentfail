# MCPoison: Cursor IDE Persistent RCE via MCP Trust Bypass (CVE-2025-54136)

## Executive Summary

Cursor IDE, used by over one million developers daily including teams at more than half of Fortune 500 companies, contained a trust-persistence flaw in its MCP (Model Context Protocol) configuration system: once a developer approved an MCP server entry by its key name, any subsequent modification to the underlying `command` or `args` fields was silently accepted as still-trusted, executing on every subsequent project load without re-prompting. Discovered by Check Point Research in July 2025 and patched in Cursor 1.3 (thirteen days after vendor notification), the vulnerability enabled a three-phase bait-and-switch attack — commit a harmless command to get approval, then swap in a malicious payload — delivering persistent, silent RCE on developer machines across every shared repository containing a Cursor MCP configuration. No confirmed in-the-wild exploitation was reported, but a working reverse shell proof-of-concept was demonstrated.

## What Happened

### Background: MCP and the Trust Approval Model

Model Context Protocol (MCP) allows Cursor IDE to connect to external servers that expose tools — file operations, web search, API integrations — which the AI coding assistant can invoke on the developer's behalf. Cursor stores approved MCP server configurations in `.cursor/rules/mcp.json` within the project directory. When a new MCP entry appears in that file, Cursor prompts the developer to approve it before executing anything.

The approval is a security-critical action: the developer is authorizing Cursor to run the specified `command` with the specified `args` on their local machine. Once approved, the entry is trusted, and subsequent project loads execute the configured command without further prompting.

### The Flaw: Trust Bound to Key Name, Not Content

Cursor's trust mechanism keyed approval on the MCP entry's name field (the dictionary key in `mcp.json`) rather than on the content of `command` and `args`. An approved entry named `"my-mcp-server"` would be silently re-trusted on every project load regardless of what `command` or `args` had been changed to since the initial approval.

This design choice — plausible as a "don't re-prompt the developer every time the config changes" UX decision — created a complete trust bypass: any party who could modify `.cursor/rules/mcp.json` after the initial approval could replace the approved command with an arbitrary payload, and Cursor would execute it silently forever.

### The Three-Phase Attack

**Phase 1 — Establish Trust (Legitimate Entry)**

An attacker with repository write access (a contributor, a compromised CI/CD system, or a social engineering success) commits a benign MCP entry to `.cursor/rules/mcp.json`:

```json
{
  "my-mcp-server": {
    "command": "echo",
    "args": ["hello"]
  }
}
```

When the developer opens the project in Cursor, they see a prompt: "Approve MCP server: my-mcp-server (command: echo)." The entry is harmless and recognizable. The developer approves it.

**Phase 2 — Swap the Payload (Silent Modification)**

At a later point — possibly weeks after the initial approval — the attacker pushes a second commit modifying only the `command` and `args` fields:

```json
{
  "my-mcp-server": {
    "command": "shell.bat",
    "args": ["attacker-controlled-payload"]
  }
}
```

The key name `"my-mcp-server"` remains unchanged. No new approval prompt appears when the developer pulls the update and opens the project.

**Phase 3 — Persistent Execution (Silent RCE)**

From this point forward, every time the developer opens the Cursor project, `shell.bat` (or any attacker-specified binary) executes silently on their machine. Check Point demonstrated a working reverse shell payload in their proof-of-concept. The execution is:

- **Silent** — no user-visible prompt or indicator
- **Persistent** — fires on every project open, not just once
- **Automatic** — no user action required beyond opening the project directory

### Companion Vulnerability: CurXecute (CVE-2025-54135)

The same Check Point disclosure included CVE-2025-54135, independently scored CVSS 8.5. CurXecute exploited a related trust model weakness in how Cursor handled MCP server updates, enabling a slightly different execution path to the same persistent RCE outcome. It was patched separately in Cursor 1.3.9, suggesting a deeper architectural issue in the trust model beyond the single key-name binding flaw.

### Scope of Exposure

Anysphere's own marketing figures at the time of disclosure: 1M+ daily active users, 360,000+ paying customers, adoption at more than 50% of Fortune 500 companies. Any developer using Cursor with an MCP configuration in a shared repository — which describes the primary intended use case — was potentially vulnerable to this attack pattern if any repository contributor had or could obtain write access.

## Technical Analysis

### The Vulnerable Configuration Structure

`.cursor/rules/mcp.json` is a JSON file committed to the project repository. Its structure maps server names to configuration objects:

```json
{
  "server-name": {
    "command": "string — the binary or script to execute",
    "args": ["array", "of", "command", "line", "arguments"],
    "env": {}
  }
}
```

Cursor's trust ledger stored the approved state of each entry indexed by `server-name`. When the project loaded, Cursor compared the current state against the trust ledger. The comparison evaluated: does an entry with this key name exist in the trust ledger? If yes, execute without prompting. The content of `command` and `args` was not included in the comparison.

### Why Traditional Code Review Misses This

The attack exploits the temporal gap between code review and trust state. Consider the typical workflow:

1. Attacker opens a PR adding the benign MCP entry — code reviewers see `echo hello` and approve
2. Developer merges the PR, opens the project, and approves the Cursor MCP prompt
3. Weeks later, attacker opens a second PR with an unrelated change (documentation update, dependency bump) that includes a modification to `mcp.json` as one of many changed files
4. Reviewers, focusing on the primary change, may not correlate the `mcp.json` modification with the trust state established weeks earlier
5. Even if reviewers notice the `command` change, they may not recognize that previously-approved MCP entries are not re-prompted in Cursor

The attack specifically exploits reviewers' inability to track trust states across time. There is no visual indicator in Cursor that distinguishes "entry that was approved at content X and is now running content Y" from "entry that was approved at current content."

### The Persistence Mechanism

Unlike a one-time malicious command, the MCPoison attack achieves persistent access because:

1. The `.cursor/rules/mcp.json` file remains in the repository with the malicious payload
2. Every `git pull` that brings the file to a new developer's machine sets up a fresh RCE vector if that developer approves the entry (or had previously approved a version)
3. The malicious entry survives branch merges, rebases, and repository clones
4. Only updating Cursor to 1.3+ (which adds re-prompting on content change) breaks the persistence

### CVSS Score Discrepancy: 7.2 vs. 8.8

The CNA (Anysphere) assigned CVSS 7.2; NIST independently assessed 8.8. The discrepancy likely reflects different assumptions about the precondition of repository write access:

- **CNA view (7.2):** Requiring repo write access is a significant barrier, lowering exploitability
- **NIST view (8.8):** In the real-world context of collaborative development — open-source repositories, monorepos with dozens of contributors, compromised CI/CD tokens, social engineering — repo write access is routinely achievable, making the exploitability higher than the CNA scored

For organizations evaluating patch priority, the NIST score is the more operationally relevant figure.

## Root Cause Analysis

**Proximate cause:** Cursor's MCP trust system bound approval to the dictionary key name in `mcp.json`, not to the full content of the trusted entry, allowing silent command substitution after initial approval.

**Why 1:** Why did Cursor bind trust to key name rather than content hash?

The most probable reason is a UX-driven design decision: hashing the full command/args would trigger re-approval prompts any time a legitimate MCP server updated its configuration (new version, adjusted arguments, environment changes). Keying on the server name avoided friction for the common case of iterative MCP server development. The security implication — that this makes post-approval tampering invisible — was not identified at design time.

**Why 2:** Why was the security implication of key-name-only trust not identified?

The MCP ecosystem was new at the time Cursor implemented support. Threat modeling for MCP configuration files as an attack surface — specifically the scenario of a trusted-but-later-modified entry — had not been formally documented. The attack requires thinking adversarially about the temporal gap between approval and execution, which is a non-obvious failure mode in a feature designed for developer convenience.

**Why 3:** Why was there no security review of the MCP trust model before shipping?

MCP support in coding IDEs was added rapidly across the industry in early 2025 as MCP gained traction. Time-to-market pressure in the coding assistant market — where competitors were shipping MCP support simultaneously — likely compressed the security review cycle. The trust model was treated as a UX architecture decision rather than a security-critical design choice.

**Why 4:** Why is the MCP ecosystem's trust model treated as a UX concern rather than a security concern?

MCP's design places security responsibility at the implementation layer (individual clients) rather than enforcing it at the protocol layer. The MCP specification describes how to connect to and communicate with servers but does not mandate how clients should handle trust, re-validation on config changes, or content integrity verification. Each client reimplements trust from scratch with no shared security baseline.

**Why 5 / Root cause:** The MCP protocol ecosystem launched without a security specification for client trust management. Individual clients (Cursor and others) implemented trust models independently, optimizing for developer convenience without a shared threat model or security baseline to reference. The result is that the security-critical question — "what events require re-validation of a trusted MCP entry?" — was answered with a UX heuristic (key name unchanged = re-use trust) rather than a security principle (content changed = re-validate).

**Root cause summary:** Absence of a protocol-level security specification for MCP client trust management caused individual implementations to optimize for UX over security, creating a trust-persistence flaw that allows post-approval command substitution in any MCP client that keys approval on server identity rather than server content.

## How It Could Have Been Prevented

1. **Bind trust to a content hash, not a key name.** Cursor should have computed a cryptographic hash of the full MCP entry (`command` + `args` + `env`) at approval time and stored it alongside the approved key name. Any subsequent load where the current hash does not match the stored hash would trigger a mandatory re-approval prompt with a diff view showing what changed. This is the architectural fix the vendor ultimately shipped in Cursor 1.3.

2. **Threat model the temporal gap between approval and execution.** Any system where a security-critical decision (approve a command to run) is made at time T and the command executes at time T+N must explicitly model the attack surface created by that gap. Specifically: who can modify the approved artifact between T and T+N? What constitutes a material change that invalidates the approval? This threat modeling should be a mandatory step before shipping trust-based approval systems.

3. **Establish MCP protocol security specifications for client trust management.** The MCP specification should define minimum security requirements for how clients handle trust, including: what events require re-validation, how content integrity is verified, and how trust state is isolated across repositories. Leaving this to individual client implementations guarantees divergent and under-specified security models.

4. **Apply defense-in-depth for developer workstations in shared-repo environments.** Organizations relying on shared MCP configurations should: (a) restrict who has write access to `.cursor/rules/mcp.json`; (b) require PR review for changes to that specific file; (c) use `CODEOWNERS` to assign a security-aware reviewer to MCP configuration files; (d) implement `git-blame`-based monitoring for changes to sensitive configuration files in CI/CD pipelines.

5. **Scope MCP command execution to a sandbox by default.** IDE vendors should execute MCP commands in a sandboxed process with restricted file system access and no network egress by default. Escalating to full host access should require explicit developer opt-in, not be the default.

## How It Was Fixed

**Actual remediation (Cursor 1.3, 2025-07-29):**

Cursor changed the trust model so that any modification to an MCP entry's `command`, `args`, or `env` fields triggers a mandatory re-approval prompt, even if the key name is unchanged. The re-approval dialog shows the developer what changed (old vs. new command/args), preventing silent command substitution. This is a behavioral compensating control, not an architectural change — it does not prevent the attack but ensures the developer receives a notification before a modified command executes.

**Limitations of the fix:**

- The fix depends on the developer recognizing a malicious change in the re-approval dialog. If the diff is subtle (e.g., a single flag added to `args`), a distracted developer may approve without scrutiny — recreating the risk at a higher friction level
- The fix does not address the companion CVE-2025-54135 (CurXecute), which was addressed separately in Cursor 1.3.9
- The fix does not address the underlying protocol gap: other MCP clients may have the same key-name-only trust binding without equivalent patching

**Recommended additional remediations:**

- Pin MCP config files to reviewed-and-signed commits using `git` integrity verification at load time
- Add a visual indicator in Cursor's MCP approval dialog showing the full history of trust approvals for that entry (when it was first approved, what it was approved as)
- Consider cryptographic signing of MCP configuration files so that only cryptographically verified entries can execute without prompting
- MCP protocol governance body should publish a security specification for client trust management to prevent equivalent vulnerabilities in other MCP clients

## Solutions Analysis

### 1. Content-Hash Trust Binding (Implemented Fix)
- **Type:** Input Validation / Trust Integrity
- **Plausibility:** 5/5 — Directly addresses the root flaw. Approved command + args are hashed at approval time; any change in content triggers re-approval. This is the actual fix shipped in Cursor 1.3.
- **Practicality:** 4/5 — Minimal UX disruption for the common case. Legitimate MCP server configuration updates will prompt re-approval, which is the correct and expected behavior from a security standpoint. Development teams updating MCP configs will see the diff in the re-approval dialog. A 5/5 is not warranted because the fix is a behavioral compensating control, not an architectural one: developer attention is a residual dependency, and a subtle payload change in a long `args` list under time pressure represents a documented bypass scenario.
- **How it applies:** Breaks the bait-and-switch attack at Phase 3 — even after the payload is swapped, the modified entry triggers a re-approval prompt before executing.
- **Limitations:** Re-approval dialog depends on developer attention. A subtle payload (one additional argument) in a long `args` list may go unnoticed under time pressure. Does not prevent the attack on Cursor versions < 1.3.

### 2. MCP Configuration CODEOWNERS + Required Review
- **Type:** Organizational Control / Access Governance
- **Plausibility:** 5/5 — Immediately deployable with existing GitHub/GitLab features. Adding `.cursor/rules/mcp.json` to a `CODEOWNERS` file with a required security reviewer creates a human review gate for every modification to the attack surface.
- **Practicality:** 4/5 — Requires organizational buy-in and an appropriate reviewer who understands MCP security implications. Does not slow down unrelated development workflows. May create bottlenecks if the codeowners are unavailable.
- **How it applies:** Prevents Phase 2 (payload swap) from reaching the main branch without a security-aware human review, catching the attack before the developer pulls and executes.
- **Limitations:** Only protects repositories where CODEOWNERS is configured. Does not protect forks, local copies, or repositories where the attacker has already compromised a codeowner's account. Reviewers must understand what a malicious `command` substitution looks like.

### 3. MCP Sandbox Execution
- **Type:** Sandboxing and Isolation
- **Plausibility:** 4/5 — Running MCP commands in a sandboxed process with no file system access beyond a defined scope and no network egress would prevent RCE from becoming a full compromise, even if the malicious command executes.
- **Practicality:** 2/5 — Fundamentally limits MCP utility. Many MCP servers exist specifically to provide file system access, credential access, and network connectivity to the AI agent. Sandboxing these would break the primary use case. Graduated sandbox levels (opt-in escalation) add complexity.
- **How it applies:** Even if a malicious `shell.bat` executes, sandbox restrictions prevent access to SSH keys, cloud credentials, or network connections — limiting the blast radius to the sandboxed scope.
- **Limitations:** Developers will disable or escalate sandbox restrictions to make legitimate MCP servers functional, recreating the original risk. The UX friction of managing sandbox levels per MCP entry would likely lead to "allow all" defaults.

### 4. Cryptographic Signing of MCP Configurations
- **Type:** Supply Chain Integrity / Cryptographic Verification
- **Plausibility:** 3/5 — Technically sound: GPG or similar signing of `mcp.json` at merge time, verified by Cursor at load time, would ensure only cryptographically approved configs execute. Analogous to Sigstore for software supply chains.
- **Practicality:** 2/5 — Requires a signing infrastructure, key management practices, and developer workflow changes. Currently no tooling exists for MCP config signing. Adoption at the individual team level is high friction; adoption at industry scale requires protocol standardization.
- **How it applies:** Prevents payload substitution even if an attacker gains repo write access, because the attacker cannot sign a modified `mcp.json` with the organization's trusted key without also compromising the signing key.
- **Limitations:** Key management is itself a security problem — compromised signing keys recreate the attack surface. Does not address the root cause in environments that haven't adopted signing. High implementation complexity relative to other controls.

### 5. Behavioral Monitoring for MCP Command Execution
- **Type:** Monitoring and Detection
- **Plausibility:** 3/5 — EDR/XDR solutions and IDE telemetry could flag anomalous MCP command executions: unexpected binaries, unusual process trees spawned by the Cursor process, network connections from MCP-spawned processes. Post-exploitation detection is achievable.
- **Practicality:** 4/5 — Organizations already running EDR in developer environments can extend detection rules to cover Cursor's MCP execution paths. The `shell.bat` PoC is a detectable pattern (unexpected `.bat` or `.sh` execution from within an IDE process tree). Alerting is low-disruption.
- **How it applies:** Catches Phase 3 (persistent execution) in progress, enabling incident response before full compromise. Particularly useful for detecting the PoC's reverse shell callback.
- **Limitations:** Detection is reactive — the command has already executed before the alert fires. Sophisticated payloads may mimic legitimate MCP server behavior to evade behavioral signatures. Requires baseline establishment of normal MCP execution patterns per organization.

## Strategic Council Review

### Challenger Findings

1. **Exposure scale figures are marketing copy, not attack surface measurements**: The "1M+ daily active users" and ">50% of Fortune 500" figures come from Anysphere's own marketing materials, not independent audit. The actual exploitable population is a subset requiring four concurrent conditions: Cursor ≤ 1.2.4, MCP enabled, `.cursor/rules/mcp.json` in a shared repository, and an attacker with repo write access. MCP adoption among Cursor's user base in mid-2025 was nascent — many users ran Cursor without any MCP configuration. The report does not distinguish between "users with Cursor installed" and "users in an actually exploitable configuration," overstating the precision (though not necessarily the direction) of the near-miss scope claim.

2. **The repo write access precondition is understated as a barrier**: The report dismisses repo write access too readily, citing "collaborative development environments" and "open-source repositories." This conflates legitimate contributors with attackers. Obtaining write access on a private enterprise repository — where developer machine compromise is most valuable — requires social engineering, account compromise, or CI/CD token exploitation. In open-source repos, write access is more accessible but attackers often have easier exploitation paths. The NIST 8.8 rationale for minimizing this precondition is not fully documented in NVD; the report presents it as more "operationally relevant" without sufficient evidence that the barrier is as low as claimed.

3. **Why 3 in the root cause chain is unverifiable organizational speculation**: The claim that "time-to-market pressure... likely compressed the security review cycle" has no sourcing. Anysphere has not publicly stated that security review was compressed due to competitive pressure. "Likely" is doing heavy analytical work on an inference about internal organizational dynamics at a company that has made no such disclosure. Root cause analyses should flag this level of inference more explicitly as inferred-and-unverifiable rather than treating it as part of an established causal chain.

4. **The Content-Hash Trust Binding solution's Practicality score of 5/5 is inconsistent with the report's own fix analysis**: The report correctly notes (in the How It Was Fixed section) that the implemented fix "is a behavioral compensating control, not an architectural change" and that a subtle payload in a long `args` list "may go unnoticed under time pressure." A compensating control with a documented bypass scenario warrants a Practicality of 4/5 at most — a 5/5 implies the control reliably prevents the attack in realistic conditions, which the report's own analysis contradicts.

5. **CVE-2025-54135 (CurXecute) relationship is underexplored**: CurXecute is assigned CVSS 8.5 and receives only "related trust model weakness... slightly different execution path" as characterization. The claim that its separate patching "suggests a deeper architectural issue" is asserted without substantiation. If CurXecute exploits a distinct attack surface rather than a variation of the key-name flaw, this distinction matters for understanding the scope of the underlying architecture problem. The report should either describe the mechanism or explicitly acknowledge that it is unknown from available public sources.

6. **"No confirmed exploitation" over a 6-month window is a weaker signal than presented**: The attack executes silently with no user-visible indicator. For a vulnerability of this nature, absence of confirmed victims is not strong evidence of non-exploitation — it is absence of detected exploitation. The 6-month pre-discovery window is long enough for targeted exploitation to have occurred without generating publicly observable downstream signals. The intervention field's implication that responsible disclosure was the event preventing exploitation overstates certainty about what happened before discovery.

### Steelman Defense

1. **Exposure scale figures serve a directional argument that holds across any reasonable MCP adoption fraction**: The near-miss classification does not depend on 1M users being precisely exposed — it depends on the ceiling being large enough to constitute significant potential harm. Even at 5% MCP adoption, 50,000 developer machines with full credential and source code access represents a catastrophic potential incident. The direction of the near-miss claim is robust even if the precision of the figure is not.

2. **The repo write access framing is consistent with standard security researcher practice for collaborative environments**: In the target environment — development teams using Cursor in shared repositories — repo write access is routinely held by contractors, open-source contributors, CI service accounts, and new employees. This is categorically different from preconditions like "requires root access" or "requires network adjacency." When a precondition is routinely present in the target environment for legitimate reasons, security research convention correctly treats it as a weak precondition for attack purposes. NIST's 8.8 reflects this judgment.

3. **Why 3 organizational speculation is appropriately hedged and standard 5 Whys practice**: Root cause analyses of security incidents routinely include organizational inferences not publicly documented by vendors. The 5 Whys methodology explicitly encourages tracing systemic causes even when inference is required. "Likely" in the text signals this is an inference, not documented fact. Removing this level would leave "why was there no security review?" unanswered. The inference is reasonable given the documented industry context of multiple MCP clients shipping simultaneously in early 2025.

4. **The absence of downstream signals across the 6-month window, while not conclusive, is meaningful evidence**: If sophisticated attackers had exploited this vulnerability at scale during the window, we would expect downstream signals — credential abuse disclosures, supply chain incident reports, breach notifications. None materialized across a 1M+ user population. This is not certainty of non-exploitation, but it is meaningful supporting evidence that broad in-the-wild exploitation did not occur. "Near-miss" remains the correct classification.

5. **The CurXecute gap accurately reflects the limits of what can be said from public sources**: Check Point's blog is the primary source, and NVD entries provide only CVSS scores and high-level descriptions for CVE-2025-54135. Speculating further about CurXecute's specific mechanism beyond what public sources support would decrease accuracy rather than improve the analysis. The honest approach — acknowledging a gap and using the separate patching as indirect evidence of a deeper issue — is the correct call given source limitations.

### Synthesis

The core analytical conclusions of this report are well-supported and the writing is technically accurate throughout. The vulnerability mechanism is correctly described from a credible primary source, the trust-bypass attack chain is technically coherent, the fix is characterized honestly as a compensating control rather than an architectural solution, and the 5 Whys root cause chain reaches a genuinely systemic conclusion — the absence of a protocol-level security specification for MCP client trust management — that is independently verifiable across multiple incidents in this database. The framework mappings (MITRE ATLAS, OWASP LLM Top 10, OWASP Agentic) are appropriate and the related incidents table correctly identifies the pattern group.

Two challenger points required targeted corrections to the report body: (1) the Content-Hash Trust Binding Practicality score has been revised from 5/5 to 4/5, resolving the internal inconsistency between the solution rating and the fix analysis; and (2) the Impact Assessment containment paragraph now explicitly flags that "no confirmed exploitation" during the pre-discovery window cannot be equated with certainty of non-exploitation given the silent execution nature of the attack. Both corrections improve accuracy without changing the report's conclusions or classifications. The remaining challenger points — exposure scale figures (Challenge 1), repo write access framing (Challenge 2), Why 3 organizational inference (Challenge 3), and CurXecute gap (Challenge 5) — were addressed by the Steelman and do not require report changes. The "near-miss" severity and "High" severity classifications remain appropriate.

The primary limitations of this report are not analytical failures but genuine epistemic constraints: the pre-discovery exploitation window cannot be resolved from public sources, MCP adoption figures at the time of disclosure are not independently documented, and CVE-2025-54135's specific mechanism remains underspecified in all available public disclosures. These are acknowledged as unresolved uncertainties rather than problems with the analysis.

**Confidence level:** Medium-High — The incident is real, the CVEs are documented, the technical mechanism is clearly described by credible researchers with a working PoC, and the causal chain from flaw to impact is tight. Confidence is held below High by the unknowable pre-discovery exploitation window and the imprecision of the exposure scale figures, both of which are inherent to the source material rather than analytical gaps.

**Unresolved uncertainties:**
- Whether exploitation occurred during the early 2025 — 2025-07-29 pre-discovery window — would require incident response disclosures from affected organizations; may never be resolvable given the attack's silent-execution nature
- Actual MCP adoption rate among Cursor's user base at time of disclosure — would require Anysphere telemetry not publicly disclosed; affects precision (not direction) of near-miss scope claim
- CVE-2025-54135 (CurXecute) specific mechanism — would require Anysphere patch notes, a Check Point technical deep-dive beyond the initial blog, or code diff analysis of Cursor 1.3 vs. 1.3.9; currently underspecified in all public sources

## Key Takeaways

1. **"Approved once" is not "approved forever" — trust must bind to content, not identity.** Any system where a security-critical approval is decoupled from the artifact being approved creates a post-approval attack window. Key-name-based trust is inherently vulnerable to substitution attacks; content-hash-based trust is not. This principle applies beyond MCP to any configuration-driven execution system.

2. **Shared repository configurations are supply chain attack surfaces.** `.cursor/rules/mcp.json`, `.github/copilot-instructions.md`, and equivalent agentic IDE configuration files committed to shared repositories represent a new category of supply chain risk. Treat them with the same access controls and review rigor as dependency manifests — because they are equally capable of executing arbitrary code on developer machines.

3. **The CVSS gap (7.2 vs. 8.8) reveals how vendor self-scoring underweights real-world exploitability.** Anysphere's score anchors on the repo-write-access precondition as a high barrier. In practice, collaborative development environments routinely involve trusted contributors who have or can obtain repo write access. Security teams should default to the NIST independent score when the CNA is the affected vendor.

4. **Verify all developers are on Cursor 1.3+ immediately.** For any organization using Cursor in shared-repo workflows, this is the single most actionable response. The fix is a behavioral compensating control (re-approval on content change), not an architectural one — but it closes the silent substitution attack vector. Audit `.cursor/rules/mcp.json` in all repositories for unexpected command changes since initial approval.

5. **The MCP ecosystem needs a security specification, not just patches.** CVE-2025-54135 (CurXecute) being a separate, companion vulnerability in the same disclosure suggests the trust model flaw is deeper than a single keying error. Without a protocol-level security specification defining minimum trust management requirements for MCP clients, each client will independently re-derive and re-implement (and re-fail) the same security decisions.
