# Ghost Billing on a Disabled API Key: 864,077 Tokens in 5 Minutes at 3 AM, $21.69 Negative Balance

## Executive Summary

Over three days in April 2026 (April 6–8), Derek Robinson of Alpha2Zulu LLC experienced three separate billing anomalies on his Anthropic Claude Code account totaling $21.69 in unexpected charges. The most alarming event — and the one that elevates this from an ordinary billing dispute to a potential platform integrity incident — occurred at 3:10–3:15 AM UTC on April 8: 864,077 tokens were consumed in exactly five minutes while his API key was manually disabled and no user session was active. Simultaneously, Anthropic's platform charts showed the cache rate hitting exactly 100% at the moment of the spike, consistent with a server-side cache redelivery event. Support ticket #215473797766657 received zero human response over 96+ hours. Anthropic's April 23, 2026 postmortem on a thinking cache bug (March 26 – April 10 window) acknowledged the bug but did not address how an API key that was manually disabled could accumulate 864,077 tokens in five minutes with no active session.

## What Happened

Derek Robinson, operating as Alpha2Zulu LLC, used Claude Code as a coding assistant via the Anthropic API with direct key billing (not a Max subscription plan). Over a three-day period beginning April 6, 2026, he experienced a cascade of billing anomalies that collectively drained his account from a positive balance to a negative one.

**Incident 1 — April 6 (74:1 ratio, service outage context):** The first anomaly coincided with a documented Anthropic service outage. Token consumption showed an input-to-output ratio of approximately 74:1. A normal Claude Code session might show ratios in the range of 5:1 to 15:1; a 74:1 ratio suggests either an extreme amount of input context being resent, or input tokens being counted multiple times. The billing impact was severe: the account swung from +$8.46 to -$13.23 — a single-session charge of over $21. This is the largest single financial impact of the three incidents.

**Incident 2 — April 7 (175:1 ratio, active session):** The following day, during an active working session, the ratio climbed to 175:1. The user was present and actively using Claude Code; the session was not unattended. The anomalous ratio strongly suggests either context re-transmission compounding (each API call re-sending the entire accumulated conversation history, as documented in AAGF-2026-001's Mechanism A) or the thinking cache bug Anthropic later acknowledged, which caused thinking tokens to be re-billed on cache reads.

**Incident 3 — April 8 (203:1 ratio, failed boot):** On April 8, during what appears to have been a failed Claude Code boot sequence rather than a productive session, a 203:1 ratio was observed. The ratio climbed with each incident even as the sessions became shorter and less productive.

**The 3:10 AM Event — The Central Anomaly:** At some point on April 8, recognizing the billing anomalies, Robinson manually disabled his API key. This is not a passive expiration or soft suspension — it is an explicit, manual action through the Anthropic Console that should prevent any further API calls from being processed against the key. At 3:10 AM UTC, with the key disabled and no active user session, the platform recorded 864,077 tokens consumed over exactly five minutes. Simultaneously, the Anthropic platform's cache rate chart showed the cache hit rate reaching exactly 100%.

Robinson installed Claude Code v2.1.94 around this time and noticed a previously unseen "API Usage Billing" header in the interface. No changelog entry explains when this header was introduced or what it signifies.

He filed support ticket #215473797766657 and received no human response within 96+ hours. An independent Senior Cybersecurity Lead reviewed the pattern and assessed it as consistent with "regression and cache re-reads with interrupts" behavior — a known failure mode in server-side caching systems where a cached payload is delivered even after the intended recipient has gone away or been invalidated.

Separately, Anthropic published an April 23 engineering postmortem acknowledging a thinking cache bug that affected the March 26 – April 10 window. The postmortem acknowledged that thinking tokens were being re-billed on cache reads. It did not address how an API key that was manually disabled could accumulate 864,077 tokens in five minutes while no session was running.

## Technical Analysis

### Platform and Configuration
- **Tool:** Claude Code CLI (v2.1.94 observed during Incident 3)
- **Billing mode:** Direct API key billing (not Max subscription)
- **Model:** Not specified in available sources; likely Claude 3.x or Claude Sonnet based on the period
- **Monitoring:** No real-time billing dashboard; Anthropic platform charts available after the fact

### Three Compounding Mechanisms

The three incidents appear to involve three distinct but potentially overlapping technical mechanisms. They are ordered from most-established to least-established based on available evidence.

**Mechanism B1 — Sentinel Bug (Context Re-Transmission / Runaway Loop)**

This mechanism is shared with AAGF-2026-001. Community researcher ArkNill, who reverse-engineered eleven confirmed bugs in the March–April 2026 billing crisis (GitHub Issue #41930, 678+ comments), documented a "Sentinel Bug" as one of the primary drivers of anomalous token ratios. The Sentinel Bug causes token counts to be inflated through repeated re-transmission of accumulated context across API calls. The 74:1, 175:1, and 203:1 ratios observed across Robinson's three incidents are consistent with this mechanism: each ratio is higher than the last, suggesting accumulating context that grows with each session. A healthy session ratio of ~10:1 inflated to 74:1 implies the system was re-sending approximately 7.4x more context than the session's outputs warranted.

**Mechanism B2 — Resume Bug (Interrupted Session Re-Ingestion)**

ArkNill's analysis also documented a "Resume Bug": when a Claude Code session is interrupted (by a crash, disconnect, or failed boot), the session resume mechanism re-ingests large portions of prior context without crediting prior cache reads. This is consistent with the April 8 failed boot scenario where Incident 3 (203:1 ratio) occurred. The Resume Bug reportedly causes 10–20x token inflation on interrupted sessions. Combined with an already-inflated context window from Incidents 1 and 2, a failed boot could produce an extremely large re-ingestion event.

**Mechanism B3 — Server-Side Cache Redelivery (The 3:10 AM Ghost Billing Event)**

This is the most technically alarming mechanism and the one unique to this incident. The simultaneous occurrence of:
1. 864,077 tokens consumed in exactly 5 minutes
2. API key manually disabled
3. No active user session
4. Cache rate chart hitting exactly 100%

...is consistent with a server-side cache redelivery scenario. In this model, Anthropic's caching infrastructure had cached a large prompt/context payload associated with Robinson's API key. When the thinking cache bug's regression triggered a re-read — or when a server-side process queued a cache delivery that was not gated by API key validity checks — the cached payload was delivered and billed against the key despite its disabled state.

The 100% cache rate is the critical data point. In a normal session, cache hit rates fluctuate based on context changes. A cache rate hitting exactly 100% simultaneously with a token spike, while no session is active, strongly implies the token consumption was entirely composed of cache reads — not new inference work. This is precisely what a cache redelivery event would look like: the server is re-delivering a stored cache payload (100% cache hits, by definition) to a billing endpoint.

**Why This Matters: The API Key Validity Check**

A correctly implemented API key disable should propagate to all infrastructure layers simultaneously. The key should be checked before any token consumption is recorded, not after. If cache redelivery occurred post-disable, it implies either: (a) the cache redelivery process does not check API key validity before billing, or (b) the disable propagation was delayed, and a redelivery event queued before the disable completed executed after it. Both interpretations represent a billing infrastructure integrity failure.

### The "API Usage Billing" Header

The appearance of a previously unseen "API Usage Billing" header in Claude Code v2.1.94 without a changelog entry is a secondary data point. It suggests a server-side change to billing classification logic was deployed around the time of these incidents, without notification to users. This is consistent with the broader pattern documented in AAGF-2026-001 (silent cache TTL reduction without changelog), but the specific significance of this header cannot be determined from available evidence alone.

### Anthropic's Thinking Cache Bug (March 26 – April 10, 2026)

Anthropic's April 23 postmortem acknowledged a thinking cache regression affecting this exact window. The bug caused thinking tokens — the internal reasoning tokens used by extended thinking models — to be re-billed on cache reads, rather than billing only the cache read fee. This directly explains the anomalous input-to-output ratios: thinking tokens are billed as output tokens (at 5x the input rate), so if they were being double-billed on cache reads, the effective output token count would be inflated, distorting the ratio.

However, the thinking cache bug does NOT explain the 3:10 AM event: thinking token re-billing still requires an active session to generate a cache read request. A disabled API key with no active session cannot generate a cache read request through normal API usage. The Anthropic postmortem's silence on this specific event is therefore itself significant.

## Root Cause Analysis

### Chain 1: Anomalous Token Ratios (Incidents 1–3)

**Proximate cause:** Three Claude Code sessions across April 6–8 produced input-to-output token ratios of 74:1, 175:1, and 203:1 — far above normal — resulting in unexpected charges.

**Why 1:** The ratios were elevated because context was being re-transmitted at scale on each API call (Mechanism B1 / Sentinel Bug) and potentially because thinking tokens were being double-billed on cache reads (Anthropic's thinking cache bug).

**Why 2:** Context re-transmission grew across sessions because Claude Code's architecture re-sends accumulated conversation history on each API call, and Robinson had multiple sessions contributing to context accumulation; the thinking cache bug existed because Anthropic's March caching infrastructure changes introduced a regression in thinking token billing logic.

**Why 3:** Neither the Sentinel Bug nor the thinking cache regression was disclosed to users during the affected window (March 26 – April 10), so Robinson had no way to recognize the anomalous billing as platform-caused rather than usage-caused.

**Why 4:** Anthropic did not publish a changelog or proactive notification for either the cache TTL changes or the thinking cache regression, consistent with the broader pattern of silent platform changes documented across the March–April 2026 incident cluster.

**Why 5 / Root cause:** Anthropic's platform architecture treats cost transparency as an optional feature rather than a safety requirement, and its deployment practices allow pricing-affecting regressions to run unacknowledged for weeks while users absorb the cost.

### Chain 2: Ghost Billing on a Disabled API Key (3:10 AM Event)

**Proximate cause:** 864,077 tokens were billed to a manually disabled API key at 3:10 AM UTC with no active user session.

**Why 1:** Token consumption was recorded despite key disable because either (a) the cache redelivery process did not check API key validity before billing, or (b) a redelivery event initiated before the disable propagated executed after it.

**Why 2:** The cache redelivery process did not check API key validity because billing checks may not be applied to server-initiated cache operations — only to client-initiated API calls. This would be an architectural assumption that "all token-consuming events are client-initiated," which a server-side cache redelivery violates.

**Why 3:** This architectural gap exists because Anthropic's billing infrastructure was designed around the expectation that all token consumption is demand-driven (user makes request, server responds). The introduction of server-side caching with active redelivery behavior introduces a supply-driven token consumption path that the billing validity checks may not cover.

**Why 4:** The architectural gap was not caught because Anthropic has no documented process for auditing whether new caching behaviors interact correctly with key-level billing controls.

**Why 5 / Root cause:** Anthropic's billing infrastructure contains an unverified assumption that all billable token events are client-initiated and therefore subject to API key validity checks. Server-side cache operations may bypass these checks, creating a path by which a disabled key can be billed — a fundamental integrity violation.

**Root cause summary (Chain 2):** If the ghost billing interpretation is correct, Anthropic's server-side caching infrastructure contains a billing validity enforcement gap that allows token charges to be posted against disabled API keys, representing a billing integrity failure with no inherent upper bound on impact.

## How It Could Have Been Prevented

1. **API key validity check applied before any token consumption event, including server-initiated cache operations.** The correct architectural control is to gate all token billing — regardless of whether the event is client-initiated or server-initiated — against a real-time check of the associated API key's enabled status. This is a fundamental billing integrity requirement that appears to be missing for cache redelivery operations.

2. **Real-time spending cap or hard stop enforced when key is disabled.** When a user manually disables an API key, the platform should immediately terminate any in-flight or queued processing associated with that key and post no further charges. This is the expected user mental model and appears not to be the actual system behavior.

3. **Proactive user notification for the thinking cache bug window.** Anthropic's April 23 postmortem confirmed the thinking cache regression affected March 26 – April 10. Proactive notification to affected users during this window — even a simple dashboard alert — would have allowed Robinson to recognize the anomalous ratios as platform-caused and avoid accumulating charges.

4. **Real-time billing dashboard with session-level cost visibility.** A real-time or near-real-time billing view would have surfaced the anomalous ratios on April 6 (74:1) before Incidents 2 and 3 amplified the damage. The existing multi-day reporting lag meant Robinson had no visibility into whether Incident 1's charges were a platform anomaly or his own usage until significant further charges had accumulated.

5. **Mandatory spending cap at API key creation.** A default per-key spending cap (configurable, but required to explicitly raise) would have limited the blast radius of any single billing anomaly, including those caused by platform bugs rather than user configuration choices.

6. **Public changelog for all pricing-affecting platform changes.** The silent deployment of the v2.1.94 "API Usage Billing" header, like the earlier cache TTL reduction, represents a pricing-affecting change with no user notification. A public changelog — standard practice for any API-first business — would have alerted users to inspect their billing behavior when changes were deployed.

## Solutions Analysis

### Solution 1: Real-Time API Key Validity Enforcement on All Billing Paths

- **Type:** Architectural Redesign
- **Plausibility:** 5/5 — This is a fundamental correctness requirement, not a new feature. Any billing system should check the validity of the billing instrument before posting a charge. The fact that this appears not to happen for server-initiated cache operations suggests a gap in the billing architecture, not a novel engineering challenge. The fix is to add the validity check to all billing paths, not just client-initiated ones.
- **Practicality:** 4/5 — Requires an audit of all token consumption code paths to identify which ones do and do not check key validity before billing. Server-side cache infrastructure may be a separate codebase from the API gateway billing layer. Cross-team coordination is required. The 4 (not 5) reflects organizational complexity rather than technical difficulty.
- **How it applies:** Had API key validity been checked synchronously before billing the 864,077 tokens at 3:10 AM, the disabled key would have blocked the charge. The mechanism — whatever triggered the cache redelivery — would have fired but produced no billable event.
- **Limitations:** Does not prevent the underlying cache redelivery from firing; only prevents it from generating a charge. If the redelivery triggers downstream side effects (model inference, API calls to third parties), those could still occur. Requires Anthropic to audit and fix; not something a user can implement.

### Solution 2: Anomaly Detection on Token Consumption Ratios

- **Type:** Monitoring and Detection
- **Plausibility:** 4/5 — The 74:1, 175:1, and 203:1 ratios are far outside the normal range for Claude Code sessions. An anomaly detection system with a simple threshold (e.g., flag any session where the input:output ratio exceeds 30:1) would have flagged all three incidents. The 3:10 AM event — 864,077 tokens with no active session — would be flagged by any activity-based anomaly detector.
- **Practicality:** 3/5 — Requires instrumented session-level metrics and a monitoring pipeline. For Anthropic, this is straightforward; for individual users, it requires deploying third-party tooling (CCusage, etc.) since the official dashboard does not provide session-level ratio metrics. The 3 reflects the gap between what is technically possible for Anthropic and what is practically available to users today.
- **How it applies:** A session-level ratio alert would have fired on April 6 (74:1 ratio) and allowed Robinson to stop using the API key before Incidents 2 and 3 accumulated further charges. An activity-with-no-session alert would have fired on April 8 at 3:10 AM, enabling immediate investigation.
- **Limitations:** Detection is not prevention — by the time an alert fires, the charge has been posted. The monitoring pipeline needs to be deployed before the incident to be useful. For the 3:10 AM event specifically, the user was asleep and any alert would have been reviewed after the fact.

### Solution 3: Mandatory Spending Cap with Hard-Stop Enforcement

- **Type:** Permission Scoping / Least Privilege
- **Plausibility:** 5/5 — A hard spending cap enforced at the billing layer would have terminated all further charges once the account balance reached a configured threshold, regardless of the mechanism driving charges. This is a standard feature on comparable API platforms and would have limited the blast radius of all three incidents.
- **Practicality:** 4/5 — Technically straightforward; requires a one-time UX change to the API key creation flow and backend enforcement. The 4 (not 5) reflects that enterprise invoiced customers may require separate treatment, and that a spending cap configured by the user can be raised or removed. Effectiveness depends on the cap being set to a meaningful value before any incident occurs.
- **How it applies:** A default cap of, for example, $25 would have terminated billing after Incident 1 (which alone accounted for more than $21 in charges), preventing Incidents 2 and 3 and — potentially — the 3:10 AM event from incurring further charges. The ghost billing event would still have occurred, but its financial impact would have been bounded.
- **Limitations:** Does not address the underlying ghost billing mechanism. A disabled key with a zero remaining balance may still have the cache redelivery event fire; the cap prevents financial impact but not the technical event. Caps set too high by users who expect large legitimate workloads provide little protection.

### Solution 4: Cryptographic Binding of Cache Payloads to Active Key State

- **Type:** Cryptographic / Integrity Controls
- **Plausibility:** 3/5 — In principle, cached payloads could be encrypted or signed with a key-state-dependent material such that they become undecipherable or invalid the moment the API key is disabled. This would prevent server-side redelivery of a cached payload to a disabled key even if the validity check is not implemented at the billing layer.
- **Practicality:** 2/5 — Requires significant changes to the caching infrastructure: cryptographic binding of cache entries to key state, key-state-dependent decryption at redelivery time, and coordination between the key management system and the caching layer. This is a meaningful engineering investment. Current adoption of cryptographic controls in LLM caching infrastructure is near zero, making this a novel implementation rather than an industry-standard pattern.
- **How it applies:** Had the cache payload been bound to Robinson's API key's active state, the server-side redelivery at 3:10 AM would have failed cryptographically when the key was disabled — producing no tokens and no charge, regardless of whether the billing validity check was present.
- **Limitations:** High engineering cost relative to the simpler billing-path validity check solution (Solution 1). Adds latency to cache read/write operations. Does not address the thinking cache bug or context accumulation mechanisms. Useful as a defense-in-depth layer after Solution 1 is implemented, not as a primary fix.

## Strategic Council Review

### Challenger Findings

**1. Is "disabled key" actually proven? The alternative explanation cannot be dismissed.**

The user reports that the API key was manually disabled before the 3:10 AM event. However, the available evidence is the user's account in a GitHub issue, not independently verifiable billing records, Anthropic's internal logs, or a timestamped screenshot of the key's disabled state at 3:10 AM. An alternative explanation — that the disable completed after the token consumption began, not before — cannot be excluded. API key disable operations may be eventually consistent across Anthropic's infrastructure: a key might be marked disabled in the Console UI but have in-flight or queued operations that complete before the disable propagates to all processing nodes. Under this interpretation, the 864,077 tokens were consumed by a request that was in-flight or queued at the moment the key was disabled, and the billing is technically correct. This is a significantly less alarming explanation than server-side cache redelivery on a hard-disabled key, and it cannot be ruled out without Anthropic's internal logs.

**2. Is $21.69 really an incident worth documenting?**

The direct financial harm is $21.69. By strict taxonomy, this is Medium severity. The case for documenting it rests entirely on the mechanism, not the amount. The counter-argument: the mechanism interpretation (server-side cache redelivery bypassing billing validity checks) is not proven. If the correct explanation is the thinking cache bug plus the eventual-consistency delay on key disable, the incident is a straightforward Medium billing anomaly during a documented platform bug window — significant for the affected user but not a platform integrity finding. The risk of over-elevation is that resources get directed toward investigating an anomaly that has a mundane explanation, at the expense of investigating the thinking cache bug's actual scope and impact, which may be larger.

**3. The 100% cache rate is consistent with but not proof of cache redelivery.**

A 100% cache rate simultaneously with a token spike is consistent with two explanations: (a) server-side cache redelivery (every token came from cache because no new inference was happening), or (b) an in-flight session that happened to have 100% cache hit rate at the moment of high consumption. During a heavily cached session — which a Claude Code session with large context accumulation could be — hitting 100% cache briefly is possible without a rogue redelivery event. The evidence is suggestive, not conclusive.

**4. The Anthropic postmortem's silence on the disabled-key event is interpretable multiple ways.**

The report treats the postmortem's silence on the ghost billing event as "itself significant evidence that the anomaly was not explained by the thinking cache bug alone." This is a reasonable inference but it is also possible that: (a) the postmortem authors were not aware of the specific disabled-key scenario from the GitHub issue, (b) the disabled-key scenario was considered and judged to be explained by eventual consistency (see Point 1), or (c) the postmortem was scoped to the thinking token re-billing issue only and the disabled-key event was out of scope. Silence is weak evidence; the absence of an explanation is not the same as the absence of an explanation.

**5. Support ticket non-response and GitHub silence are documented patterns, not unique to this incident.**

The report cites 96 hours of no human response on ticket #215473797766657 and zero Anthropic staff responses on Issue #44726 as significant findings. These are valid observations, but they are consistent with the documented pattern across the entire March–April 2026 incident cluster. Issue #41930 (678+ comments) also received minimal Anthropic engagement. The support posture is a systemic Anthropic behavior, not a signal specific to this incident that there is something unusual about this case that Anthropic is avoiding.

### Steelman Defense

**1. The ghost billing event is the most technically significant finding in the AgentFail database, regardless of dollar amount.**

The $21.69 financial figure is the weakest aspect of this incident report. The mechanism is the strongest. If server-side cache redelivery can post charges to a disabled API key, then the user's fundamental assumption about what "disabling a key" means — that it stops all further billing — is false. This is not a billing dispute; it is a contract violation. Every API platform that offers key disable functionality implicitly promises that disabling a key stops further charges. If Anthropic's billing infrastructure violates this implicit contract for any class of operations, the integrity of all key-level billing controls is undermined. The $21.69 figure is the blast radius for this user's cached payload size; the mechanism has no inherent upper bound.

**2. Issue #29108 (closed NOT_PLANNED) provides direct evidence of Anthropic's awareness and non-response.**

The strongest piece of evidence in this report is not the 3:10 AM event itself — it is that GitHub Issue #29108, which documented credits draining after session close and API key deletion, was closed by Anthropic as NOT_PLANNED. This is not silence or inaction; this is an affirmative decision by Anthropic not to fix a related behavior. Combined with the ghost billing event, this creates a coherent narrative: Anthropic was aware that post-disable billing events were possible, was informed by users that this was unacceptable, and chose not to address it. The ghost billing incident then occurred within the billing crisis period. The NOT_PLANNED closure is not circumstantial — it is documented institutional knowledge of the failure mode.

**3. The eventual-consistency explanation does not fully exonerate the platform.**

Even if the most charitable interpretation is correct — that the 864,077 tokens were an in-flight request that completed after the key was disabled due to propagation delay — this is still a platform integrity problem. Users who disable an API key expect billing to stop. If key disable operations have propagation delays during which charges can still accumulate, this should be documented, and the propagation delay should be measured and bounded. A user who disables a key to stop runaway billing and then observes charges continuing for an unknown duration has been failed by the platform, even under the benign interpretation.

**4. The thinking cache bug window directly overlaps all three incidents.**

All three of Robinson's incidents (April 6–8) fall within Anthropic's documented thinking cache bug window (March 26 – April 10). The anomalous token ratios (74:1, 175:1, 203:1) are exactly consistent with thinking tokens being double-billed on cache reads: thinking tokens are billed as output tokens at 5x input price, so double-billing them would dramatically inflate the effective output token count relative to input. The thinking cache bug alone provides a coherent explanation for Incidents 1–3 that does not require the Sentinel Bug or Resume Bug as additional factors. This is actually the steelman for the financial harm: Anthropic's own postmortem confirms the platform was broken during these exact dates.

**5. The documentation of this incident serves the community even if the ghost billing mechanism remains unresolved.**

The value of this incident report is not contingent on proving the ghost billing mechanism. Even if the eventual-consistency explanation is correct, this report documents: (a) three billing anomalies during Anthropic's acknowledged thinking cache bug window; (b) Anthropic's prior awareness of post-disable billing via Issue #29108; (c) the 96-hour support non-response pattern; and (d) the need for cryptographic or architectural controls to make key disable synchronous and complete. These findings are valuable for practitioners regardless of which ghost billing interpretation proves correct.

### Synthesis

**Final assessment:**

The three Incidents 1–3 (anomalous token ratios on April 6–8) are high-confidence findings. They fall within Anthropic's documented thinking cache bug window, the ratios are exactly consistent with thinking token double-billing, and the NOT_PLANNED closure of Issue #29108 demonstrates prior institutional knowledge of the failure class. The financial harm of $21.69, while modest, was real and produced no remediation.

The 3:10 AM ghost billing event is the analytically uncertain but strategically critical finding. The Challenger review correctly identifies that the "disabled key" interpretation relies on the user's account rather than independently verifiable logs. The alternative interpretation (eventual-consistency delay during key disable propagation) cannot be excluded. However, the Steelman is equally correct that even the alternative interpretation represents a platform integrity problem: if key disable is not synchronous and complete, users relying on it as a billing stop mechanism are being misled. Both interpretations require Anthropic to investigate and document the actual behavior of key disable operations relative to in-flight and queued billing events.

This incident should be classified as **Medium-High** — above a routine Medium billing dispute because the mechanism (whichever interpretation is correct) represents a gap in billing integrity controls, but below a confirmed High/Critical because the $21.69 figure is within Medium range and the ghost billing mechanism interpretation is not yet independently confirmed.

The most important unresolved question is the one Anthropic has not answered publicly: can a manually disabled API key be billed for token consumption originating from server-side operations? The answer to this question determines whether this incident is a Medium billing anomaly during a known bug window, or the first documented instance of a Critical-class infrastructure integrity failure.

**Confidence level:** Medium — High confidence on the three billing anomalies during the thinking cache bug window; medium confidence on the ghost billing mechanism interpretation; low confidence on the exact causal attribution across Mechanisms B1, B2, and B3 without Anthropic's internal investigation.

User account in a GitHub issue is insufficient to establish this definitively. Materiality: high — this is the pivotal question for the ghost billing interpretation.

## Key Takeaways

1. **API key disable may not be a synchronous, complete billing stop — deploy a spending cap as your primary circuit breaker.** The behavior of Anthropic's key disable operation relative to in-flight billing events is undocumented and appears to be at least eventually consistent. Do not rely on key disable as your first line of defense against runaway billing; configure a workspace spending cap in the Anthropic Console before any unsupervised or autonomous workflow begins. Key disable is a last resort with uncertain timing; a spending cap is a hard, platform-enforced ceiling.

2. **Anomalous token ratios (>30:1 input:output) are a diagnostic signal, not just a billing concern.** Robinson's three incidents showed escalating ratios: 74:1, 175:1, 203:1. These ratios are not achievable through normal Claude Code usage. Any session that shows a ratio above 30:1 should be investigated before continuing. Third-party tools (CCusage, shepard-obs-stack) provide session-level ratio visibility that the official Anthropic dashboard does not. Deploy these before any extended Claude Code session and set anomaly alerts.

3. **Anthropic's thinking cache bug window (March 26 – April 10, 2026) likely affected more users than those who filed tickets.** The thinking cache regression caused thinking tokens to be double-billed on cache reads — a silent, automatic inflation of charges for any user of extended thinking models during this window. If your Claude Code usage during this period shows anomalous charges and you have not filed a support ticket, do so now with token ratio data as evidence. Anthropic's own postmortem confirms the platform was broken; the burden of proof is lower than for a general billing dispute.

4. **The NOT_PLANNED closure of Issue #29108 (post-deletion billing) establishes that Anthropic was aware of post-disable billing events and chose not to fix the mechanism.** This is not a new platform problem and it is not an isolated report. Before relying on key deletion or disable to stop billing in a production environment, verify the behavior in a test environment with small stakes and document the propagation delay you observe. Design your billing safety architecture around the assumption that key-level controls have unknown latency, not instant effect.

5. **Support ticket silence is not resolution — escalate and document publicly.** Ticket #215473797766657 received no human response in 96+ hours. GitHub Issue #44726 has zero Anthropic staff responses. The documented pattern across the March–April 2026 incident cluster is that individual support tickets are not escalated to engineering review; only public community pressure on GitHub (Issue #41930, 678+ comments) has produced partial responses. If you experience a billing anomaly that you believe is platform-caused, file the support ticket and simultaneously open a public GitHub issue with full technical details. The community's documented experience is that public visibility accelerates platform response, however partially.
