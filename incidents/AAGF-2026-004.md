# Claude Code Subagent API Key Inheritance: $152 in Unauthorized API Charges with Rate Limit Starvation Side-Effect

## Executive Summary

A developer on Anthropic's Max Plan ($200/month) was charged $152.04 in direct API fees across five development sessions in March 2026 because Claude Code's Agent tool spawned subagent child processes that silently inherited `ANTHROPIC_API_KEY` from the parent shell environment. The parent session ran correctly under Max Plan's OAuth subscription, but each subagent — unaware of the OAuth context — picked up the ambient environment variable and routed its requests through pay-per-use API billing. A secondary, underappreciated harm compounded the direct financial loss: the 800+ API requests generated by subagents consumed the shared account-level rate limit (800K tokens/minute for Sonnet), throttling the developer's legitimate Max Plan CLI work at the same time they were being charged for it. The issue remained open with zero Anthropic staff responses at time of curation, and the partial fix shipped in v2.1.98 does not cover the Agent tool subagent code path.

## What Happened

tciovacco was a software developer on Anthropic's Max Plan — the $200/month subscription tier that grants unlimited (fair-use) Claude Code CLI access without per-token API charges. He was using Claude Code's Agent tool, which enables a parent Claude Code session to spawn subagent child processes for parallelized or delegated workloads.

Separately, he had set `ANTHROPIC_API_KEY` in his shell environment for a different desktop application — a common configuration pattern for developers who use multiple Anthropic-powered tools. He had no reason to believe this environment variable would affect his Claude Code sessions, which he understood to be running under his Max Plan OAuth subscription.

During five development sessions across March 25–26, 2026, each time the parent Claude Code session used the Agent tool to spawn a subagent, the subagent process inherited the full parent shell environment including `ANTHROPIC_API_KEY`. Claude Code's authentication priority logic gives environment variable API keys precedence over OAuth subscription tokens. The subagents therefore authenticated as API key users — pay-per-use — not as Max Plan subscribers.

The parent session continued to use Max Plan correctly. The subagents did not. Over the five sessions, subagents generated over 800 API requests with individual requests ranging from 50,000 to 166,000 input tokens, firing every 3–10 seconds with 2–3 concurrent subagents active simultaneously. The developer's billing dashboard accumulated $152.04 in API charges he did not intend, did not know were occurring, and had no mechanism to detect in real time.

At the same time, the rate of API requests generated by the subagents consumed the shared account-level rate limit — 800,000 tokens per minute for Claude Sonnet across the entire account. This throttled the developer's legitimate Max Plan CLI work running concurrently: the same sessions he was paying $200/month to use were being slowed by the unauthorized API traffic from his own subagents. He was simultaneously paying twice and getting degraded service.

The developer filed GitHub issue #39903 on March 27. At time of curation, the issue had zero Anthropic staff responses and was marked open.

## Technical Analysis

### Platform and Configuration

- **Tool:** Claude Code CLI, Agent tool for subagent spawning
- **Operating system:** Windows 11
- **Billing mode:** Max Plan OAuth (parent session) + API key billing (subagents, unintended)
- **Environment variable present:** `ANTHROPIC_API_KEY` (set for separate desktop application, not for Claude Code)
- **Request volume:** 800+ API requests, 50K–166K input tokens each, every 3–10 seconds, 2–3 concurrent subagents
- **Financial impact:** $152.04 in direct API charges

### The Authentication Inheritance Chain

The failure has three distinct layers, each of which would need to be addressed independently for a complete fix.

**Layer 1 — OS Child Process Environment Inheritance**

POSIX-compliant operating systems (and Windows, following analogous semantics) propagate the parent process's environment block to all child processes by default. When Claude Code spawns a subprocess — whether via the Bash tool, the Agent tool, or any other mechanism — that subprocess receives a complete copy of the parent's environment at time of fork, including all environment variables.

This is not a bug; it is documented OS behavior that most inter-process workflows rely on for configuration propagation. The correct fix is not to change the OS but to selectively scrub the child environment of credentials before propagation.

**Layer 2 — Claude Code Authentication Priority Logic**

Claude Code resolves authentication credentials via a priority chain. Based on Anthropic's published support documentation (confirmed post-incident), the priority order gives `ANTHROPIC_API_KEY` (environment variable) precedence over the authenticated OAuth subscription token. The rationale is developer ergonomics: when a developer explicitly sets an API key, they likely want it used.

This priority logic was designed for the common case of a developer setting a key for explicit API usage, not for the case where the key is ambient from an unrelated application. In the parent session, the OAuth token was correctly present and used. In the subagent child process, the OAuth session context was not propagated — only the flat environment variables were — and the priority logic resolved to the API key.

**Layer 3 — Agent Tool Subagent Architecture: No Subscription Context Propagation**

The Agent tool, when spawning a subagent, passes environment variables from the parent process but does not propagate the parent's OAuth session context or subscription-tier metadata. The parent session has a running OAuth authentication state (tokens, session scope, subscription entitlements). This state is held in the process's memory and in the Claude Code daemon's session management layer — not in a form that child processes can inherit through the standard environment mechanism.

A child process spawned by the Agent tool starts a new Claude Code authentication resolution cycle from scratch. It reads environment variables (finds `ANTHROPIC_API_KEY`), does not find an OAuth session token in the environment, and authenticates as an API key user. The subscription entitlement of the parent session is invisible to the child.

**What "Correct" Behavior Would Look Like Architecturally**

There are two architecturally sound approaches to fixing this:

*Option A — Environment Scrubbing Before Subagent Spawn:* Before spawning any subagent, the Agent tool should remove or mask `ANTHROPIC_API_KEY` from the child process environment. This is what `CLAUDE_CODE_SUBPROCESS_ENV_SCRUB` attempts for Bash tool subprocesses, but it is not applied to the Agent tool code path. The limitation is that this requires the developer to opt in and is a Linux-only implementation.

*Option B — Explicit Subscription-Bound Token Propagation:* Anthropic's `claude setup-token` command generates a subscription-bound OAuth token suitable for headless/subprocess use, and `CLAUDE_CODE_OAUTH_TOKEN` is a documented subscription-safe environment variable. If the Agent tool propagated the parent's OAuth token via `CLAUDE_CODE_OAUTH_TOKEN` rather than relying on the general environment, subagents would authenticate under the same subscription tier as the parent. This is a positive-security design — subagents are explicitly granted subscription context — rather than a negative-security design (scrubbing API keys reactively).

The correct architectural model is that subagents spawned by a parent subscription session should inherit the subscription context, not independently re-resolve authentication from an environment that may contain unrelated credentials. The parent session acts as the authoritative billing principal; child processes should delegate to it, not re-authenticate independently.

### The Rate Limit Starvation Second-Order Effect

This incident introduces a harm vector not present in the financial-loss incidents that preceded it in the AgentFail database. When subagents routed to API billing, their requests consumed the shared account-level rate limit — 800,000 tokens per minute for Claude Sonnet across the entire Anthropic account.

API key requests and Max Plan OAuth requests share the same account-level rate limit. There is no separate rate limit pool for "paid subscription traffic" versus "API traffic." This means the 800+ subagent requests firing every 3–10 seconds at 50K–166K tokens each were not merely generating unexpected bills — they were filling the shared token bucket, triggering rate limiting on the developer's legitimate Max Plan CLI work running concurrently.

The developer was simultaneously:
1. Being charged $152.04 for API calls he did not intend
2. Having his $200/month Max Plan CLI work throttled by those same unintended calls

This second-order effect transforms a pure financial harm into also an operational harm. The developer did not receive the full value of their Max Plan subscription during the affected sessions. This is a novel harm vector: unauthorized API consumption degrading authorized subscription value on the same account. No other incident in the AgentFail database to date documents this mechanism.

The quantified impact of the rate limit starvation is difficult to establish from available evidence (300K–600K token/minute spikes on API side are estimated from the described request volumes and token ranges, not from verified rate limit logs). However, the mechanism is structurally sound: at 2–3 concurrent subagents each making requests at 50K–166K tokens, 800+ requests over two days, the token/minute consumption during active bursts would materially intersect with the 800K/min ceiling.

## Root Cause Analysis

**Proximate cause:** A developer with `ANTHROPIC_API_KEY` set in their shell environment for an unrelated application launched Claude Code with the Agent tool. Subagent child processes inherited the environment variable, resolved authentication against it at higher priority than the parent's OAuth subscription, and routed 800+ requests to pay-per-use API billing, generating $152.04 in unintended charges.

**Why 1:** The subagents resolved to API key billing because Claude Code's authentication priority logic treats `ANTHROPIC_API_KEY` as higher precedence than an OAuth subscription token, even when the parent session is running under subscription.

**Why 2:** The authentication priority logic prioritizes API keys over OAuth because the system was designed for a single-process model where a developer explicitly setting `ANTHROPIC_API_KEY` signals intent to use API billing. The design did not anticipate the multi-agent case where a parent's subscription session spawns children that may encounter ambient API keys from unrelated applications.

**Why 3:** The multi-agent case was not handled because when the Agent tool was designed to spawn subagent processes, the architecture did not include a mechanism to propagate subscription context (OAuth session, billing tier) from parent to child. Child processes start a fresh auth resolution cycle, which is the simple, stateless implementation — but not the correct one for subscription-scoped billing.

**Why 4:** Subscription context was not propagated because auth state is session-local (in-memory OAuth tokens and daemon session data) and the OS environment inheritance mechanism — which is how child processes receive configuration — carries only flat key-value strings, not live session objects. Exporting subscription context would require either converting it to an environment-safe form (e.g., `CLAUDE_CODE_OAUTH_TOKEN`) or creating a daemon-mediated auth delegation mechanism. Neither was implemented.

**Why 5 / Root cause:** The Agent tool was architected to spawn child processes using OS environment inheritance for configuration, without recognizing that the environment is a shared, ambient space that developers commonly populate with credentials for multiple tools simultaneously. There was no design requirement to ensure that subagents spawned from a subscription session inherit the subscription billing context rather than independently re-resolve authentication from an environment that may contain unrelated, higher-priority credentials.

**Root cause summary:** The Agent tool's subagent spawning mechanism relies on OS environment inheritance for configuration but does not include a subscription context propagation step, creating a class of billing mismatch failures whenever an API key is present in the developer's environment for any reason, regardless of whether that key was intended for Claude Code.

## How It Could Have Been Prevented

1. **Remove or unset `ANTHROPIC_API_KEY` from the shell environment when using Claude Code on Max Plan.** The most direct prevention: if the variable is not present, the auth priority logic cannot route to API billing. However, this requires the developer to know about the auth priority ordering, which is not prominently documented at the point of Max Plan subscription setup. Placing this warning in the Claude Code onboarding flow or subscription dashboard would address the "developer shouldn't need to know this" concern.

2. **Use `CLAUDE_CODE_OAUTH_TOKEN` instead of `ANTHROPIC_API_KEY` for non-Claude-Code applications, or use application-scoped env management.** Anthropic documents `CLAUDE_CODE_OAUTH_TOKEN` as a subscription-safe environment variable for headless Claude Code use. If applications that need a generic Anthropic API key used a different env var name (e.g., `ANTHROPIC_API_KEY_MYAPP`), ambient exposure to Claude Code's auth resolution would be eliminated. Application-scoped env management tools (direnv, per-project `.env` files) can also scope which applications see which credentials.

3. **Anthropic: propagate subscription OAuth token to Agent tool subagents explicitly.** The `claude setup-token` command generates a subscription-bound OAuth token. If the Agent tool automatically exported `CLAUDE_CODE_OAUTH_TOKEN` from the parent session's OAuth context into the child process environment, subagents would authenticate under the parent's subscription tier regardless of what other API keys are present. This is the correct architectural fix for the multi-agent case.

4. **Anthropic: extend `CLAUDE_CODE_SUBPROCESS_ENV_SCRUB` to cover the Agent tool code path and all platforms.** The v2.1.98 opt-in env scrubbing was scoped to Bash tool subprocess sandboxing on Linux. Applying the same scrubbing to Agent tool spawned subagents on all platforms (including Windows 11, where this incident occurred) would address the immediate billing routing failure. This is a defensive fix rather than the correct architectural approach, but it provides protection while a proper solution is designed.

5. **Anthropic: warn during session start when `ANTHROPIC_API_KEY` is present but billing mode is Max Plan OAuth.** A one-time warning at Claude Code session startup — "ANTHROPIC_API_KEY is set in your environment. Subagents spawned by the Agent tool may use this key for API billing rather than your Max Plan subscription. [Dismiss / Learn more / Unset key]" — would have prevented this incident at near-zero platform engineering cost.

**Actual remediation taken:**
The developer filed GitHub issue #39903 and received no Anthropic staff response. The $152.04 in charges has no documented refund. The developer's immediate workaround was presumably to unset `ANTHROPIC_API_KEY` from their shell environment for Claude Code sessions. **Additional recommended fixes:**

For developers affected by this pattern now:
- Run `unset ANTHROPIC_API_KEY` before starting Claude Code sessions on Max Plan, or add this to your shell profile's Claude Code startup alias
- Alternatively, run `claude setup-token` to generate a `CLAUDE_CODE_OAUTH_TOKEN` for subscription-bound headless use, and ensure non-Claude-Code apps use a differently named env var

For Anthropic (systemic platform fixes):
- Implement positive-security subscription context propagation in the Agent tool: export parent OAuth token as `CLAUDE_CODE_OAUTH_TOKEN` to child processes, ensuring subagents inherit subscription billing tier
- Extend env scrubbing to the Agent tool code path and to Windows (the affected platform in this incident)
- Add a startup warning when both `ANTHROPIC_API_KEY` and Max Plan OAuth credentials are detected in the same session
- Add billing path transparency to the Agent tool: each subagent should log its authenticated billing mode (Subscription / API Key / [key suffix]) at spawn time, visible in verbose output
- Consider inverting the auth priority for the Agent tool specifically: if a parent session is subscription-authenticated, child agents should inherit subscription context unless the developer explicitly opts into API key override

## Solutions Analysis

### Solution 1: Subscription Context Propagation in Agent Tool (Architectural Redesign)

- **Type:** Architectural redesign
- **Plausibility:** 5/5 — The mechanism is clear: export the parent session's OAuth token as `CLAUDE_CODE_OAUTH_TOKEN` into the child environment before Agent tool subagent spawn. `CLAUDE_CODE_OAUTH_TOKEN` is already a documented subscription-safe env var. The child process's auth resolution would then find a subscription-bound token and use it, regardless of any ambient `ANTHROPIC_API_KEY`. This would definitively prevent the billing inheritance failure for any developer on Max Plan using the Agent tool.
- **Practicality:** 4/5 — Requires a targeted change in the Agent tool spawning code to extract and serialize the parent's OAuth token into the child environment. No novel infrastructure; the token is already accessible to the parent process. The 4 (not 5) reflects that OAuth tokens have expiry and refresh cycles — the spawning logic would need to ensure the exported token is fresh and handle token refresh across long-running subagent sessions. This is a known problem in OAuth systems with well-established patterns.
- **How it applies:** In this incident, the subagent's auth resolution found `ANTHROPIC_API_KEY` in the environment and used it. If `CLAUDE_CODE_OAUTH_TOKEN` had been present with higher priority (or equal priority, since `ANTHROPIC_API_KEY` specifically is documented as having higher priority — requiring an explicit auth priority adjustment for the multi-agent case), the subagent would have authenticated as Max Plan, and no API charges would have been generated.
- **Limitations:** Requires an auth priority rule change for the Agent tool code path specifically: `CLAUDE_CODE_OAUTH_TOKEN` must take precedence over `ANTHROPIC_API_KEY` when the former is present and parent-session-derived. This must not regress the case where a developer intentionally wants a subagent to use a different API key. A distinction between "parent-propagated subscription token" and "developer-set override" may need to be explicit in the environment variable naming or via a separate flag.

---

### Solution 2: Environment Scrubbing Extended to Agent Tool and All Platforms (Permission Scoping / Least Privilege)

- **Type:** Permission scoping / least privilege
- **Plausibility:** 4/5 — Removing `ANTHROPIC_API_KEY` from the child process environment before spawn would prevent the subagent from finding and using it. This is what `CLAUDE_CODE_SUBPROCESS_ENV_SCRUB` does for Bash tool subprocesses on Linux. Extending this to the Agent tool code path on all platforms (including Windows 11, where this incident occurred) would prevent the billing inheritance failure. The residual risk is that a developer who intentionally wants subagents to use an API key would lose that capability without an explicit opt-in mechanism.
- **Practicality:** 3/5 — The existing Linux-only Bash tool implementation demonstrates feasibility. The 3 (not 4) reflects two gaps: (1) Windows env manipulation in process spawn code has platform-specific considerations, and the existing implementation is Linux-only; (2) the opt-in design (`CLAUDE_CODE_SUBPROCESS_ENV_SCRUB=1`) means developers must know to enable it — a default-on design would be more protective but risks breaking workflows where intentional key inheritance is wanted. Also, this is a negative-security approach (scrubbing what should not be there) rather than positive security (ensuring correct credentials are propagated).
- **How it applies:** With env scrubbing active on all platforms for the Agent tool code path, this incident's subagents would have started with no `ANTHROPIC_API_KEY` and would have needed to authenticate via another mechanism. If no subscription token was available, they would have failed to start — a better outcome than silently billing the API account.
- **Limitations:** Negative-security design — scrubs known bad credentials rather than ensuring correct ones are present. Does not solve the case where a developer has no OAuth token available for the subagent and also has the API key scrubbed. In that scenario, the subagent fails entirely rather than billing incorrectly — arguably still better, but could break legitimate headless workflows if misconfigured.

### Solution 3: Startup Warning for Auth Mode Conflict (Guardrails / Output Filters)

- **Type:** Guardrails / output filters
- **Plausibility:** 5/5 — Detecting the conflict (Max Plan OAuth active + `ANTHROPIC_API_KEY` present in environment) is trivially implementable at session startup. Displaying a one-line warning or a dismissible notice requires no architectural changes to the auth system, no subprocess behavior changes, and no platform-specific code.
- **Practicality:** 5/5 — Extremely low implementation cost. The condition is deterministic and detectable. A single flag in the session startup code to check for both credential types and emit a warning would address the knowledge gap that makes this incident possible. Given that the information asymmetry — developer does not know Claude Code's auth priority ordering affects subagents — is the core enabling condition, a targeted warning at session start is the highest-ROI intervention available.
- **How it applies:** tciovacco would have seen at Claude Code startup: "Note: `ANTHROPIC_API_KEY` is detected in your environment. Agent tool subagents will use this API key for billing rather than your Max Plan subscription. To prevent unexpected API charges, unset this variable or use `claude setup-token` for subscription-safe headless authentication." This warning, before any subagent work, would have given the developer the information needed to prevent the incident.
- **Limitations:** Does not prevent the failure if the developer dismisses or ignores the warning. Addresses only the information gap, not the architectural vulnerability. Does not help developers who encounter the issue because `ANTHROPIC_API_KEY` is set by a script or CI environment they do not directly control.

### Solution 4: Billing Path Transparency in Agent Tool Verbose Output (Monitoring and Detection)

- **Type:** Monitoring and detection
- **Plausibility:** 4/5 — Each time the Agent tool spawns a subagent, that subagent resolves its auth credentials at spawn time. The billing mode used (Subscription / API Key / last four chars of key) is a deterministic output of this resolution. Logging this to verbose output per subagent spawn would make billing path visible to any developer monitoring their session.
- **Practicality:** 4/5 — Implementation requires adding a log line at the subagent spawn point that reports resolved auth mode. No architectural changes; this is an instrumentation addition. The 4 (not 5) reflects that this is a detection-during-incident control, not a prevention control. It requires the developer to be watching verbose output during the session and to understand that "API Key: sk-ant-****abcd" in a Max Plan session represents a billing problem.
- **How it applies:** In this incident, five development sessions generated 800+ API requests across two days. If the Agent tool had logged "Subagent spawned — Auth mode: API Key (sk-ant-****1234), Billing: pay-per-use" on the first subagent spawn, the developer would have had an opportunity to identify and terminate the sessions before $152.04 accumulated. This would not have prevented the first few requests but would have materially limited the blast radius.
- **Limitations:** Reactive, not preventive. Requires developer presence and active monitoring. Verbose output is often not watched in active development sessions. Does not help in scripted or CI contexts where output may not be reviewed in real time.

### Solution 5: Auth Mode Confirmation Step for High-Volume Subagent Workflows (Human-in-the-Loop)

- **Type:** Human-in-the-loop (HITL)
- **Plausibility:** 3/5 — A confirmation prompt when the Agent tool first spawns a subagent in API key billing mode — "Subagent will use API key billing, not your Max Plan subscription. Estimated cost for this workflow: [estimate]. Proceed? [Yes / Switch to subscription / Cancel]" — would provide a decision point before any charges are incurred. The plausibility is limited because cost estimation for multi-agent workflows is imprecise, and the prompt would need to be triggered correctly (first subagent spawn per session under conflicting auth conditions) to avoid becoming noise.
- **Practicality:** 3/5 — Moderate implementation complexity. Requires: (1) detection of auth mode mismatch at subagent spawn; (2) interrupting the Agent tool workflow for user input, which conflicts with the autonomous/headless use cases the Agent tool is designed for; (3) cost estimation for multi-agent subagent workflows, which is imprecise. In scripted or CI contexts, a blocking interactive prompt would break automation. An opt-out flag (`CLAUDE_CODE_SKIP_BILLING_PROMPT=1`) would be needed, which re-introduces the risk for developers who set it without understanding the implications.
- **How it applies:** A single confirmation prompt on the first subagent spawn of the first affected session would have surfaced the billing mode mismatch before any API charges were generated. The developer could have chosen to stop, investigate, and unset `ANTHROPIC_API_KEY` before proceeding.
- **Limitations:** Does not work for headless/automated workflows. Cost estimates for multi-agent workflows are highly imprecise. Adds friction to the "happy path" of developers who intentionally use API key billing for subagents. Best implemented as an optional, opt-in friction layer rather than a mandatory gate, which reduces its protective value for developers who have not opted in.

## Strategic Council Review

### Challenger Findings

**1. Is this a "bug" or documented behavior?**
Anthropic published a support article (post-incident) explicitly stating: "Claude Code prioritizes environment variable API keys over authenticated subscriptions." This is now documented behavior, not an undisclosed bug. The auth priority ordering — `ANTHROPIC_API_KEY` > OAuth token — was a design decision, not an implementation accident. The counter-argument that documentation published after incidents have occurred is not a defense holds weight, but practitioners evaluating the current state of the platform should note that the priority ordering is now officially stated. This shifts some responsibility to the developer: setting `ANTHROPIC_API_KEY` in a shared shell environment when using Max Plan is documented as having consequences. The incident happened before this documentation existed, but that fact needs to be in the analysis, not assumed away.

**2. Is the $152.04 figure verified?**
The amount is stated by the reporting user in GitHub issue #39903. It is not verified against Anthropic billing records, API invoices, or independent account audits. It is consistent with the described mechanics (800+ requests, 50K–166K tokens each, at Sonnet API pricing) but has not been independently validated. The analysis should treat this as "reported by primary claimant, plausible given described request volume, unverified against billing records."

**3. Does the partial fix (v2.1.98 SUBPROCESS_ENV_SCRUB) substantially address the risk?**
No. `CLAUDE_CODE_SUBPROCESS_ENV_SCRUB` is (a) opt-in, not default-on; (b) scoped to Bash tool subprocess sandboxing, not the Agent tool code path; (c) Linux-only, while this incident occurred on Windows 11. The partial fix addresses a different code path from the one that failed here. The claim that the Agent tool subagent inheritance path "is not covered" is accurate and material. Framing v2.1.98 as a fix for this incident would be misleading.

**4. Is Medium severity too conservative given nine related issues?**
The taxonomy rates severity on actual impact of the individual incident, not on pattern potential. $152.04 is unambiguously below the $10,000 High threshold. However, nine related GitHub issues spanning months, with the same root mechanism, and with the only partial fix applying to a different code path, indicates that the per-incident Medium rating does not capture the aggregate systemic risk. The report should make this tension explicit: the incident is Medium by taxonomy, but the pattern is High-severity systemic risk that practitioners should weight accordingly.

**5. Is the rate limit starvation mechanism verified or speculative?**
The mechanism is architecturally sound — API key requests and Max Plan OAuth requests share the same account-level rate limit in Anthropic's documented architecture. The claim that "300K–600K token/minute spikes on the API side crowded out Max Plan CLI work" is a derived estimate from described request volumes, not a verified metric from Anthropic's rate limit logs or the developer's direct observation. The report should be explicit that the rate limit starvation is a structurally plausible second-order effect, not a directly confirmed measurement.

**6. User error is underweighted.**
The developer set `ANTHROPIC_API_KEY` in their shell environment globally — available to all processes in all shell sessions — while also using Max Plan. This is a configuration practice that, even without Anthropic's documented auth priority ordering, carries inherent credential-scope risk. A developer practicing strict credential hygiene (application-scoped env vars, per-project `.env` files, shell profile conditions) would not have been affected. The incident's root cause is architectural, but it required a specific user configuration practice to manifest. Both the platform failure and the user configuration practice should be in the causal chain.

### Steelman Defense

**1. The auth priority design is not disclosed at the point of subscription setup — post-hoc documentation is not a defense.**
The Anthropic support article documenting auth priority was published after incidents had been reported, not before Max Plan subscribers began using the Agent tool. A developer purchasing a $200/month Max Plan subscription and enabling the Agent tool for subagent workflows has no reasonable path to discover that ambient env vars from unrelated applications will override their subscription billing without actively searching for a support article that did not exist at time of use. The principle of least surprise demands that a subscription-tier product protect the subscriber's billing mode from ambient environment contamination, especially in multi-agent architectures where child processes are spawned without explicit developer intervention at spawn time.

**2. Nine related issues over months is a diagnostic pattern, not coincidental noise.**
Issues #8327, #12352, #16489, #27468, #34609, #37686, #39903, #44669, and #45572 represent independent reporters filing the same root mechanism across an extended period. For a developer tool with a relatively small user base compared to consumer products, nine independently filed issues on a single failure mode is a statistically significant signal. The absence of Anthropic staff responses on #39903 and the partial fix that explicitly excluded the affected code path (Agent tool) constitute a documented organizational non-response to a known, multi-user failure pattern.

**3. The rate limit starvation effect is a novel and important harm vector even if its quantification is imprecise.**
The mechanism — unauthorized API consumption by one billing path degrading legitimate subscription performance on the same account — has not been previously documented in the AgentFail database. Whether the developer experienced 50ms additional latency or multi-second throttling during the affected sessions is unknown; the key contribution is establishing the category. A developer using Multi-agent workflows who also has an ambient API key has a non-zero probability of their subscription CLI work being degraded by their own subagents. This is worth documenting even with imprecise quantification, precisely because it is a second-order effect that would not be obvious from reviewing billing records alone.

**4. The correct architectural fix (subscription context propagation) exists and is well-scoped.**
`CLAUDE_CODE_OAUTH_TOKEN` as a documented, subscription-safe environment variable demonstrates that Anthropic has the primitives needed to fix this correctly. The Agent tool propagating the parent's OAuth token as `CLAUDE_CODE_OAUTH_TOKEN` into child process environments is a targeted, backward-compatible change that would eliminate the billing inheritance failure without disrupting intentional API key usage patterns (with a clear opt-override mechanism). The existence of a clear, low-disruption fix makes the ongoing non-response harder to justify.

**5. The Medium severity rating, while taxonomically correct, appropriately flags systemic risk.**
Rating this incident Medium per taxonomy while explicitly documenting that nine related issues across months create a High-severity systemic pattern is the correct approach for an incident database that serves both immediate harm assessment and systemic risk discovery. Inflating individual severity to capture systemic risk would distort the taxonomy; the correct practice (used here) is to note both the per-incident rating and the pattern-level risk. Practitioners reading this report have both data points.

## Key Takeaways

1. **Unset `ANTHROPIC_API_KEY` from your shell profile before using Claude Code on Max Plan, especially with Agent tool workflows.** Anthropic's documented auth priority gives `ANTHROPIC_API_KEY` (env var) precedence over OAuth subscription tokens. If the variable exists in your environment for any reason — a different desktop app, a leftover config, a CI script — every Agent tool subagent will bill to your API account, not your subscription. Until Anthropic ships a fix for the Agent tool code path, this is the only reliable prevention. Add `unset ANTHROPIC_API_KEY` to your Claude Code startup alias or use application-scoped env management (direnv, per-project `.env` files) to scope credentials to specific tools.

2. **A partial fix shipped for one code path does not mean the vulnerability class is resolved.** `CLAUDE_CODE_SUBPROCESS_ENV_SCRUB` in v2.1.98 and the fix for issue #43333 addressed Bash tool subprocess sandboxing and the `-p` flag routing bug respectively. Neither covers the Agent tool subagent spawning path. When a vendor ships a partial fix, verify that the specific code path affecting your workflow is covered before assuming you are protected. Check the fix scope notes, not just the changelog headline.

3. **Billing-path pollution creates a second-order operational harm that billing records alone will not reveal.** When unauthorized API calls share your account's rate limit, they throttle legitimate subscription work in the same sessions — you pay twice and receive degraded service. This harm is invisible in billing dashboards (which show only charges, not throttled requests) and requires either rate limit telemetry or active latency monitoring to detect. If your Claude Code sessions feel slower than expected while the Agent tool is active, check for auth mode conflicts before blaming network conditions or model load.

4. **Zero vendor responses on a GitHub issue does not mean zero risk.** Issue #39903 had zero Anthropic staff responses through curation. The absence of an official response does not mean the issue is not real, not affecting others, or not worth mitigating. Nine independently filed issues on the same root mechanism confirm the failure mode is not rare. If your tooling vendor is non-responsive to a documented, reproducible failure pattern, apply mitigations based on community-documented workarounds and treat the unresolved code path as a known risk in your operational model.

5. **Multi-agent architectures introduce credential scope risks that single-agent workflows do not.** When you spawn subagents, you are creating new processes that re-resolve authentication from scratch — including from ambient environment state you may not have consciously set for that tool. Before deploying multi-agent workflows in production or in sessions where billing is sensitive, audit your shell environment for credential variables that could affect child process auth resolution. This is a general principle that applies to any multi-agent framework, not just Claude Code: credential hygiene in parent processes directly affects the billing and permission model of spawned children.
